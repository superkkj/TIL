# CS 개념 완전분해 템플릿

> 목적: 정의를 외우는 공부가 아니라, 면접 꼬리질문을 버틸 수 있게 개념의 뿌리, 내부 동작, 실패 상황, 선택 기준까지 분해한다.
>
> 100% 방어 템플릿은 없다. 면접관은 구현체 버전, 수학적 증명, 운영 경험, 직접 구현까지 계속 깊게 들어갈 수 있다. 이 템플릿의 목적은 대부분의 CS 질문에서 “외워온 답변”처럼 보이지 않고, 원리 기반으로 대답할 수 있게 만드는 것이다.

## 산출물 모드

템플릿을 특정 개념에 적용할 때 기본 산출물은 **교재형 완성본**이다.

```text
기본값: 교재형 완성본
  모든 섹션을 설명, 예시, 근거, 내부 동작, 반례, 꼬리질문과 답변까지 채운다.
  빈칸, 체크박스, "직접 써보세요", "스스로 도출" 같은 문제풀이 형식은 넣지 않는다.
  독자가 문서만 읽고 해당 개념을 강의할 수 있어야 한다.
  모르는 사실은 답처럼 채우지 말고 U(미확인)로 표시한다.

워크북형
  사용자가 명시적으로 요청한 경우에만 만든다.
  빈칸, 체크박스, 직접 작성 과제를 둘 수 있다.

면접카드형
  사용자가 명시적으로 요청한 경우에만 만든다.
  20초/1분 답변, 핵심 반례, 꼬리질문 답변 위주로 압축한다.
```

교재형 완성본으로 작성할 때도 19번(인출·복습)과 20번(최종 점검)은 남긴다. 다만 빈칸 체크리스트가 아니라, 해당 개념을 공부한 뒤 수행할 복습 절차와 점검 기준으로 작성한다.

## 설명 레이어 규칙

모든 핵심 개념은 **정석 설명 + 쉬운 설명**을 한 세트로 쓴다. 정석 설명만 있으면 외운 말처럼 되고, 쉬운 설명만 있으면 면접 답변의 정확도가 떨어진다.

```text
1. 정석 설명
   공식 문서, 구현체, 교과서 용어 기준으로 정확하게 설명한다.
   면접에서 그대로 말해도 틀리지 않을 문장이어야 한다.

2. 쉬운 설명
   사람 말, 사물 비유, 숫자 예시, 짧은 코드로 다시 설명한다.
   "처음 보는 사람이 어디서 막힐지"를 먼저 풀어준다.

3. 다시 정석 용어로 환원
   쉬운 설명을 다시 공식 용어와 연결한다.
   예: "사물함이 bucket이고, 사물함 전체가 table이며, 몇 % 차면 이사할지 정하는 값이 load factor다."
```

작성 형식:

```text
정석 설명:
  ...

쉬운 설명:
  ...

다시 정리:
  쉬운 설명의 __는 정석 용어로 __다.
```

적용 대상:

```text
용어 분해: 모든 어려운 단어에 적용한다.
내부 구성 요소: 표 + 단순화 코드 + 쉬운 비유를 같이 둔다.
내부 동작 순서: 정석 단계 후 "사람 말로 다시 말하면"을 붙인다.
성능 분석: Big-O만 쓰지 말고 숫자 예시를 붙인다.
대안 비교: 이론 비교 후 "언제 뭘 고를지"를 실무 문장으로 바꾼다.
답변 버전: 쉬운 설명으로 이해한 뒤, 면접용 정석 문장으로 압축한다.
```

금지:

```text
쉬운 설명이 정석 설명을 대체하면 안 된다.
비유는 이해 보조일 뿐이고, 마지막에는 반드시 실제 용어로 돌아온다.
"대충 이런 느낌"으로 끝내지 않는다.
```

예:

```text
load factor

정석 설명:
  HashMap의 load factor는 capacity가 자동 증가하기 전에 해시 테이블이 얼마나 가득 차도록 허용할지를 나타내는 값이다.

쉬운 설명:
  사물함이 16칸 있는데 16칸이 전부 찰 때까지 기다리면 너무 붐빈다.
  그래서 기본값 0.75는 "12칸 정도 차면 더 큰 사물함으로 이사하자"는 기준이다.

다시 정리:
  사물함 전체가 table, 사물함 칸 수가 capacity, 실제 들어간 물건 수가 size,
  이사 기준선이 threshold, 그 기준선을 계산하는 비율이 load factor다.
```

## 도메인 중립성 규칙

이 템플릿은 HashMap이나 자료구조 전용이 아니다. HashMap 예시는 샘플일 뿐이고, 실제 문서를 만들 때는 해당 개념의 도메인 언어로 바꿔야 한다.

```text
문서 작성 전 먼저 고정할 것:
  이 개념의 주 입력은 무엇인가?
  이 개념의 주 출력은 무엇인가?
  빠르게 만들고 싶은 핵심 연산은 무엇인가?
  그 대가로 포기하는 것은 무엇인가?
  비교 대상은 같은 문제를 푸는가, 아니면 다른 축의 문제를 푸는가?
```

도메인별로 바뀌는 말:

```text
자료구조:
  입력 = key / index / element
  출력 = value / node / ordering
  핵심 연산 = get / put / range / traversal

네트워크:
  입력 = packet / segment / request
  출력 = response / ACK / retry / connection state
  핵심 연산 = send / receive / retransmit / route

DB:
  입력 = row / query / transaction
  출력 = result set / commit result / lock state
  핵심 연산 = read / write / index scan / commit / rollback

OS/메모리:
  입력 = process / thread / address / page
  출력 = scheduling decision / mapped frame / protection result
  핵심 연산 = context switch / page fault / allocation / synchronization

동시성:
  입력 = shared state / critical section / task
  출력 = visibility / atomicity / ordering guarantee
  핵심 연산 = lock / CAS / wait / notify / dispatch
```

예시를 그대로 베끼면 안 되는 곳:

```text
HashMap 문서에서 "key", "bucket", "load factor"가 중요하다고 해서
TCP 문서에도 억지로 key/bucket 식으로 설명하지 않는다.

TreeMap 비교가 HashMap에는 맞지만,
스레드/프로세스 비교에서는 비교 축이 "주소 공간, 자원 공유, 스케줄링 단위"로 바뀐다.

"범위 검색"은 자료구조에서는 key range일 수 있지만,
DB에서는 인덱스 스캔, 네트워크에서는 지연/대역폭/순서 보장, OS에서는 주소 범위/권한 문제가 될 수 있다.
```

조건 질문이 나오면 **조건이 어느 축을 대상으로 하는지** 먼저 구분한다. 이 규칙은 범용이고, 구체 예시는 각 개념 문서의 꼬리질문에 둔다.

```text
주 식별자/주소 축:
  이 개념이 빠르게 찾도록 설계된 기준인가?

정렬/범위 축:
  순서, 범위, 인접 값, 최소/최대가 필요한가?

부가 속성/상태 축:
  원래 빠르게 찾도록 설계된 기준이 아니라면 전체 탐색이나 보조 구조가 필요한가?

주의:
  면접에서 나온 특정 질문 예시는 템플릿에 고정하지 않는다.
  템플릿에는 판단 기준만 두고, 실제 사례는 해당 개념 문서에 넣는다.
```

## 사용 규칙 (템플릿보다 먼저 읽는다)

```text
티어를 나눈다:
  T1 (빈출 상위 20~30개): 전체 섹션 적용
  T2: 축약판만 적용 — 0 / 2 / 4 / 8 / 10 / 16 / 17
  T3: 한 줄 카드 (계약 한 줄 + 대가 한 줄)

타임박스: T1 개념당 90~120분. 넘기면 그 개념은 "미완"으로 표시하고 다음으로 넘어간다.
  미완은 실패가 아니다. 완벽하게 채운 3개보다 80%로 채운 10개가 면접에서 더 강하다.

해당 없는 섹션은 "해당 없음" 한 줄 쓰고 건너뛴다.
  예: REST vs RPC에서 15번(직접 구현), CAP에서 7번(내부 동작 순서).
  억지로 채운 섹션은 공부가 아니라 문서 작성 노동이다.

작성 순서는 자유지만, 마지막은 반드시 19번(인출·복습) → 20번(최종 점검) 순서로 끝낸다.
  채우거나 읽기만 하고 덮고 말하기를 안 하면 이 템플릿은 절반만 작동한 것이다.
  교재형으로 공부할 때도 같다: 읽기 = 입력, 19번 = 출력. 출력 없이는 면접에서 안 나온다.
```

## 0. 기준 선언

먼저 기준을 정한다.

```text
어떤 언어 기준인가?
어떤 라이브러리 기준인가?
어떤 버전 기준인가?
공식 문서가 보장하는 내용인가?
구현체 내부 세부사항인가?
```

예:

```text
Java HashMap 기준입니다.
공식 API 보장과 OpenJDK 구현 세부는 구분해서 말하겠습니다.
```

### 0.1 증거 수집

본문 작성 전에 근거 소스를 먼저 박아둔다. 이 칸이 비면 이후 섹션 전체가 "그럴듯한 설명"으로 흐른다.

```text
공식 문서: 어디를 볼 것인가 (Javadoc / RFC / 공식 레퍼런스)
구현 소스: 어디를 볼 것인가 (OpenJDK 파일명 등)
실험: 직접 돌려서 확인할 것 1개
미확인 목록: 조사했지만 확인 못 한 것
이후 섹션에서 출처 없이 쓴 문장은 U(미확인)로 표시한다.
```

예:

```text
공식 문서: Javadoc(java.util.HashMap)
구현 소스: OpenJDK HashMap.java (putVal, hash, resize, treeifyBin)
실험: 같은 hashCode를 반환하는 key 100만 개 put → 조회 시간 측정 (충돌 최악 확인)
미확인: 해시 테이블 최초 등장 연도 (U)
```

## 2. 태어나기 전 문제

이 개념이 등장하기 전의 원래 문제를 "직접 겪어본다".
말로 아는 것과 naive 코드를 손으로 써보고 아는 것은 면접에서 완전히 다른 답이 나온다.

```text
이 개념 없이 같은 일을 하는 코드를 3~5줄로 직접 쓴다 (naive 구현).
그 방식이 깨지는 지점을 숫자로 말한다: "N이 __쯤 되면 __가 __만큼 나빠진다"
실제 역사 순서를 확인한다 (무엇이 먼저 있었나). 공식 확인이 안 되면 U(모름)로 표시한다.
한 문장으로 압축한다: "전에는 __로 했고, N=__부터 __가 터져서, __가 나왔다"
```

예:

```text
naive 구현: List<Pair>를 앞에서부터 순회하며 key를 equals로 비교
깨지는 지점: N=10만이면 조회 1회당 평균 5만 번 비교, 조회가 초당 수천 번이면 CPU가 비교에 소모된다
역사: 배열/연결리스트 → 해시 테이블 (1950년대, U: 정확한 최초 구현)
압축: "전에는 순회 비교로 찾았고, 데이터와 조회량이 늘면 O(n)이 터져서, 해시로 위치를 계산하는 HashMap이 나왔다"
```

## 3. 문제 유형 분류

이 개념이 푸는 문제를 계약으로 정의하고, 유형 태그를 붙이고, "무엇이 아닌가"까지 말한다.

```text
계약 한 줄: "받는 것(입력/전제) → 내놓는 것(출력/보장)"

도메인별 유형 태그 (해당하는 것만):
  자료구조: 단건 조회 / 범위 검색 / 정렬 순회 / 최소·최대·다음 값
  동시성: 경합 / 가시성 / 실행 순서
  네트워크: 신뢰성 / 순서 / 흐름·혼잡 제어
  DB: 정합성 / 격리 / 복구 / 읽기·쓰기 성능
  기타: 메모리 / I/O / 저장 공간

결정 질문: 이 개념을 빼고 요구사항을 다시 쓰면 무슨 문장이 남는가?
  → 그 문장이 곧 문제 정의다.

무엇이 아닌가(non-goal)를 1개 이상 말한다.
  → 비교 꼬리질문(vs 대안)은 전부 여기서 출발한다.
```

예:

```text
계약: key 하나 → value 하나, 평균 O(1) 조회 (전제: hashCode가 잘 분산)
태그: 자료구조 / 단건 조회
개념을 빼면 남는 문장: "수만 건 중에서 key가 일치하는 값을 즉시 찾아야 한다"
non-goal:
  범위/정렬 검색은 풀지 않는다. hash는 순서를 보존하지 않는다.
  → "그럼 정렬/범위가 필요하면요?" → TreeMap/B+Tree 계열 비교로 이어진다.
```

## 4. 핵심 아이디어

변환 문장과 지불한 대가, 두 칸으로 고정한다. 자유 서술로 쓰면 뭉개진다.

```text
변환 문장 (고정 프레임): "__ 문제를 __ 문제로 바꾼다"

지불한 대가: 공짜는 없다. 무엇을 내주고 그걸 얻었는가?
  → 대가를 말할 수 있어야 10번(성능)과 11번(대안 비교)이 연결된다.
    대안 비교는 결국 "어떤 대가를 낼 것인가" 싸움이다.

패턴 이름 붙이기: 공간↔시간 교환 / 사전 계산 / 간접화 / 복제 / 로그 선행 기입 / …
  → 패턴 어휘가 쌓이면 처음 보는 개념도 "이거 그 패턴이네"로 매핑된다.
```

예:

```text
HashMap
  변환: 검색을 "순회" 문제에서 "위치 계산" 문제로 바꾼다.
  대가: 메모리(빈 버킷)와 순서를 내주고 평균 O(1)을 얻는다.
  패턴: 공간↔시간 교환

TCP
  변환: "신뢰 없는 전달"을 "순번 부여 + 재전송" 문제로 바꾼다.
  대가: 지연과 헤더 오버헤드를 내주고 순서·무손실 보장을 얻는다.
  패턴: 확인응답 + 재시도

DB 인덱스
  변환: 테이블 풀스캔을 트리 탐색 문제로 바꾼다.
  대가: 쓰기 비용과 저장 공간을 내주고 읽기 속도를 얻는다.
  패턴: 사전 계산
```

## 5. 내부 구성 요소

구조 안에 있는 부품을 전부 나열한다.

```text
내부에 어떤 부품이 있는가?
각 부품의 역할은 무엇인가?
어떤 부품이 성능에 영향을 주는가?
어떤 부품이 정합성에 영향을 주는가?
공식 구현체가 있다면, 내부 부품이 연결되는 코드를 10~30줄로 단순화해 보여줄 수 있는가?
```

예:

```text
HashMap 내부 구성 요소
  key
  value
  hashCode
  bucket array
  entry
  equals
  load factor
  resize
```

교재형 완성본에서는 표만 쓰지 말고, 가능하면 바로 아래에 **내부 구조 코드(단순화)**를 붙인다.

```text
코드에 반드시 드러낼 것:
  1. 저장 공간: 배열, 리스트, 트리, 버퍼, 파일, 테이블 등
  2. 저장 단위: Node, Entry, Segment, Page, Frame 등
  3. 핵심 상태값: size, capacity, threshold, offset, sequence number 등
  4. 핵심 연결: 다음 노드, 부모/자식, index 계산, 포인터, 참조 관계
  5. 성능을 바꾸는 상수/조건: load factor, timeout, batch size, page size 등

주의:
  공식 구현을 그대로 복붙하지 않는다.
  이해용으로 단순화했다고 명시한다.
  실제 구현 세부와 다른 부분은 "단순화"라고 표시한다.
  null, 예외, 동시성 같은 중요한 계약을 코드에서 생략했다면 생략했다고 적는다.
```

예:

```java
// HashMap 내부 구조 코드(단순화)
class HashMap<K, V> {
    Node<K, V>[] table;   // bucket 배열
    int size;             // 저장된 엔트리 수
    int threshold;        // resize 발동선
    float loadFactor;     // 얼마나 차면 늘릴지

    static class Node<K, V> {
        final int hash;
        final K key;
        V value;
        Node<K, V> next;  // 충돌 시 다음 노드
    }
}
```

## 6. 핵심 연산·계약 분석

이 개념을 지탱하는 핵심 연산 1~3개를 고르고, 각 연산의 계약까지 내려간다.
함수가 없는 개념이면 상태 전이(TCP 상태도)나 프로토콜 메시지를 연산으로 취급한다.

```text
핵심 연산 1~3개는 무엇인가?
  예: HashMap = hashCode/equals/resize
      TCP = 핸드셰이크/ACK·재전송
      트랜잭션 = commit/rollback/락 획득

각 연산의 계약:
  입력 → 출력 (타입과 범위)
  사전조건: 뭐가 보장돼 있어야 실행 가능한가?
  사후조건: 끝나면 뭐가 보장되는가?
  실패 모드: 실패하면/안 지켜지면 어떻게 되는가?

결정성: 같은 입력이면 같은 출력이어야 하는가?
  아니라면 무엇이 결과를 바꾸는가?

붕괴 시나리오를 구체적으로 1개 쓴다:
  "이 계약이 깨지면 → __가 일어나서 → 상위 구조가 __하게 무너진다"
```

예:

```text
hashCode()
  입력: key 객체 → 출력: int (전 범위)
  사전조건: equals가 true인 두 객체는 같은 값을 반환해야 한다 (계약)
  사후조건: bucket index 계산에 쓸 수 있는 결정적인 값
  결정성: 같은 객체(상태)면 항상 같은 값. 다른 객체는 같은 값이어도 된다(충돌 허용)
  붕괴 시나리오: 호출마다 다른 값을 반환하면
    → put한 키를 get이 다른 bucket에서 찾는다
    → 예외 없이 null이 나오는 조용한 데이터 유실

TCP 재전송
  입력: 전송한 세그먼트 + 타이머 → 출력: 재전송 여부
  사전조건: 수신측이 ACK로 순번을 보고한다
  사후조건: 타임아웃/중복 ACK 시 유실로 간주하고 다시 보낸다
  붕괴 시나리오: ACK 처리가 잘못되면 → 불필요한 재전송 폭증 → 혼잡 악화
```

## 7. 내부 동작 순서

동작을 말로만 설명하지 말고, 순서로 쪼갠다.

```text
생성 시 어떤 일이 일어나는가?
저장 시 어떤 순서로 움직이는가?
조회 시 어떤 순서로 움직이는가?
삭제 시 어떤 순서로 움직이는가?
크기가 커질 때 어떤 일이 일어나는가?
각 단계에서 어떤 비교나 계산을 하는가?

구체 입력 하나를 잡고 실제 값으로 손 계산한다.
  머릿속 시뮬레이션이 아니라 진짜 숫자가 나와야 한다.
```

예:

```text
put
  key의 hashCode 계산
  bucket index 계산
  bucket 확인
  같은 key인지 equals로 비교
  없으면 저장, 있으면 value 교체

get
  key의 hashCode 계산
  bucket index 계산
  bucket으로 이동
  equals로 실제 key 확인
  value 반환

손 계산 (capacity=16 기준):
  h = "abc".hashCode() = 96354
  spread = 96354 ^ (96354 >>> 16) = 96354 ^ 1 = 96355   ← 구현 세부(상위 비트 섞기)
  index = 96355 & (16 - 1) = 3   → bucket[3]
```

## 10. 성능 분석

평균만 말하지 말고 전제와 최악을 같이 말한다.

```text
평균 시간복잡도는?
최악 시간복잡도는?
공간복잡도는?
그 성능이 성립하는 전제는?
전제가 깨지면 어떻게 되는가?

그 복잡도를 직접 유도한다 — 3줄 논증.
  "O(1)입니다"에서 멈추면 면접관의 "왜요?"에서 갈린다.
  유도 과정에서 전제가 저절로 드러난다.
```

예:

```text
HashMap get/put은 평균 O(1)에 가깝다.
단, hash function이 bucket에 잘 분산된다는 전제가 필요하다.
충돌이 많으면 성능이 떨어진다.

유도 (평균 O(1)):
  n개 key가 m개 bucket에 균등 분산되면 bucket당 평균 n/m개.
  load factor 초과 시 resize하므로 n/m은 0.75 이하로 유지된다.
  → bucket당 기대 비교 횟수가 상수 → 기대 O(1). (전제 "균등 분산"이 유도에서 드러남)

유도 (최악 O(n) 또는 O(log n)):
  모든 key가 한 bucket에 몰리면 연결 리스트 순회 O(n).
  Java 8+ 구현 세부: 한 bucket이 8개를 넘으면 트리화되어 O(log n).
```

## 11. 대안 비교

면접 꼬리질문은 대부분 “그럼 왜 이걸 썼나요?”로 온다.

```text

정렬 순회         지원 안 함(별도 정렬)  O(n)
min / max / next  지원 안 함        O(log n)

LinkedHashMap
  삽입 순서 또는 접근 순서를 유지해야 할 때 고려한다.
```
