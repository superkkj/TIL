# 00. CS 개념 공통 코어 템플릿

태그: CS, 면접, 공통템플릿, 꼬리질문

---

## 목적

정의를 외우는 문서가 아니라, 면접 꼬리질문을 버틸 수 있게 개념의 뿌리, 내부 구조, 동작 순서, 실패 상황, 선택 기준까지 분해한다.

이 템플릿은 모든 폴더에 공통으로 적용한다. 단, 구체 질문은 반드시 폴더별 오버레이 템플릿에서 가져온다.

## 산출물 모드

```text
기본값: 교재형 완성본
  모든 섹션을 설명, 예시, 근거, 내부 동작, 반례, 꼬리질문과 답변까지 채운다.
  빈칸, 체크박스, "직접 써보세요" 같은 문제풀이 형식은 넣지 않는다.
  독자가 문서만 읽고 해당 개념을 강의할 수 있어야 한다.
  모르는 사실은 답처럼 채우지 말고 U(미확인)로 표시한다.

워크북형
  사용자가 명시적으로 요청한 경우에만 만든다.

면접카드형
  사용자가 명시적으로 요청한 경우에만 만든다.
```

### 교재형 완료 게이트

아래 항목 중 하나라도 빠지면 `교재형 완성본`으로 표시하지 않는다. 짧은 용어라도
해당 용어 자체의 설명을 억지로 늘리지 말고, 그 용어가 참여하는 상위 구조와 실제
연산까지 연결해서 답한다.

| 검증 항목 | 완료 조건 |
|---|---|
| 정의 | 포함 관계와 제외 대상을 함께 말한다. |
| 탄생 배경 | 이전 방식과 그 방식이 깨지는 조건을 설명한다. |
| 내부 구조 | 저장 단위와 참조 관계를 그림 또는 코드로 재현한다. |
| 정상 동작 | 입력부터 결과까지 상태 변화를 순서대로 추적한다. |
| 불변식 | 동작 전후 반드시 유지되어야 하는 규칙을 말한다. |
| 비용 | 평균·최악·상환 비용과 각각의 전제를 구분한다. |
| 실패 | 예외와 조용한 오동작을 각각 하나 이상 설명한다. |
| 선택 | 대안 두 개 이상과 선택 기준을 비교한다. |
| 구현/추적 | 최소 코드 또는 손 추적 예제를 제공한다. |
| 꼬리질문 | 정의·원리·실패·비교·실무 질문에 답을 제공한다. |
| 근거 | 홈페이지가 아니라 주장을 확인할 수 있는 정확한 문서/소스 위치를 연결한다. |

### 형식만 채운 문서를 막는 검증 기준

아래 항목은 제목만 존재해서는 안 되고, 실제 내용으로 확인되어야 한다.

| 항목 | 최소 검증 증거 |
|---|---|
| 상태 추적 | 구체 입력 하나를 넣고 연산 전·중·후 상태를 끝까지 그린다. |
| 구현 | 핵심 불변식을 드러내는 최소 코드를 제공하고 생략 범위를 밝힌다. |
| 실패 재현 | 예외가 나는 사례와 예외 없이 잘못되는 사례를 구분한다. |
| 비용 분석 | `n`의 의미, 평균·최악·상환 중 어떤 비용인지, 성립 전제를 적는다. |
| 비교 선택 | 대안 이름만 나열하지 않고 동일 연산을 기준으로 선택 이유를 설명한다. |
| 꼬리질문 | 정의·내부·실패·비교·실무 축에서 답이 달린 질문을 제공한다. |
| 상하위 연결 | 원자 용어는 상위 구조의 어느 연산에서 쓰이는지 링크하고 추적한다. |

분량 자체를 완료 조건으로 사용하지는 않는다. 그러나 모든 문서가 똑같이 짧고,
질문 수와 문단 구조가 기계적으로 같다면 개념별 완료 게이트를 실제로 적용했는지
다시 검토한다. 상위 개념 문서는 전체 동작을 설명하고, 하위 용어 문서는 상위 문서와
중복 복사하지 말고 해당 용어의 계산·경계·실패를 더 깊게 설명한다.

용어의 크기에 따라 분량은 달라질 수 있다. `root`, `leaf`, `LIFO` 같은 원자 용어는
상위 개념 문서보다 짧을 수 있지만, 정의만 반복하고 끝내서는 안 된다. 최소한
`상위 구조에서의 역할 → 연산 중 사용 지점 → 경계 사례 → 반대/비교 개념`까지
연결한다.

## 설명 레이어 규칙

모든 핵심 개념은 **정석 설명 + 쉬운 설명 + 다시 정석 용어로 환원** 순서로 쓴다.

```text
정석 설명:
  공식 문서, 구현체, 교과서 용어 기준으로 정확하게 설명한다.

쉬운 설명:
  사람 말, 사물 비유, 숫자 예시, 짧은 코드로 다시 설명한다.

다시 정리:
  쉬운 설명의 어떤 부분이 실제 용어로 무엇인지 연결한다.
```

주의:

```text
쉬운 설명이 정석 설명을 대체하면 안 된다.
비유는 이해 보조일 뿐이고, 마지막에는 반드시 실제 용어로 돌아온다.
특정 주제의 예시를 다른 도메인에 억지로 복붙하지 않는다.
```

### 어려운 용어를 방치하지 않는 규칙

독자가 해당 개념을 처음 공부한다고 가정한다. 정석 설명이 정확하더라도, 아래와 같은
표현을 뜻풀이 없이 사용하면 완료된 문서로 보지 않는다.

```text
영문 용어: runtime, component type, covariance, comparator, fan-out 등
축약어: ADT, BFS, DFS 등
구현 용어: capacity, threshold, resize, probing, rotation 등
분석 용어: 불변식, 전제, 상환 비용, 점근적 상한 등
```

작성할 때는 다음 규칙을 적용한다.

1. 어려운 용어가 처음 나오는 자리에서 괄호 또는 바로 다음 문장으로 뜻을 푼다.
2. 용어가 세 개 이상이면 정의 아래에 `어려운 말 먼저 풀기` 표를 둔다.
3. 쉬운 뜻만 적지 말고 현재 문서의 그림이나 코드에서 무엇을 가리키는지 연결한다.
4. 영어를 한국어로 바꾸는 데서 끝내지 않는다. 예를 들어 `runtime = 실행 시간`이
   아니라 `프로그램이 실제로 실행되고 있는 동안`이라고 설명한다.
5. 한 문장에 새 용어가 둘 이상 나오면 문장을 나누고, 입력부터 결과까지 순서로
   다시 설명한다.
6. 뒤쪽 내부 동작·실패·성능 섹션에서도 새로운 용어가 등장하면 같은 규칙을 다시
   적용한다. 첫 정의의 쉬운 설명만으로 전체 문서의 설명 의무가 끝나지 않는다.
