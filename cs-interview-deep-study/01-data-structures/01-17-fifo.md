## 1. 정의

### 정석 설명

FIFO는 First In First Out의 약자로, 먼저 들어온 항목이 먼저 나가는 순서 정책이다.

### 쉬운 설명

줄 서기다. 먼저 온 사람이 먼저 처리된다.

### 어려운 말 먼저 풀기

| 용어 | 쉬운 뜻 |
|---|---|
| FIFO | 먼저 들어온 항목을 먼저 내보내는 순서 규칙이다. |
| 순서 정책 | 저장 구조와 별개로 어떤 항목부터 처리할지를 정한 규칙이다. |
| enqueue | Queue 뒤에 새 항목을 넣는 동작이다. |
| dequeue | Queue 앞의 가장 오래된 항목을 꺼내는 동작이다. |
| 공정성 | 먼저 기다린 항목을 앞서 처리해 뒤 항목이 계속 추월하지 않게 하는 성질이다. FIFO는 도착 순서에 관한 공정성을 제공한다. |

### 다시 정리

쉬운 설명은 이해를 위한 비유다. 실제 면접 답변에서는 위 정석 설명의 용어와
전제 조건으로 돌아와 설명한다.

---

## 2. 탄생 배경과 필요성

### 왜 필요한가

요청이나 작업을 들어온 순서대로 공정하게 처리해야 할 때 필요하다.

---

## 3. 해결 범위와 비목표

### FIFO가 보장하는 것과 보장하지 않는 것

FIFO는 먼저 enqueue된 element가 먼저 dequeue된다는 상대 순서를 보장한다. 처리
시간, 스레드 공정성, 장애 시 유실 방지까지 자동으로 보장하는 개념은 아니다.

```text
enqueue A -> enqueue B -> enqueue C
dequeue 결과: A -> B -> C
```

---

## 4. 내부 구조와 핵심 원리

### 예

```text
enqueue A
enqueue B
enqueue C
dequeue -> A
dequeue -> B
dequeue -> C
```

### FIFO를 정확한 사건 순서로 나누기

시스템에서는 다음 시점이 서로 다르다.

```text
요청 생성 -> Queue 도착 -> dequeue -> 처리 시작 -> 처리 완료 -> 응답
```

FIFO Queue가 직접 보장하는 것은 보통 enqueue 순서에 따른 dequeue 순서다. 네트워크
도착 순서, 여러 consumer의 처리 완료 순서, 장애 후 재처리 순서까지 자동으로 같아지는
것은 아니다.

#### 단일 consumer

```text
dequeue A -> A 완료 -> dequeue B -> B 완료
```

처리를 직렬로 완료한 뒤 다음 작업을 꺼내면 완료 순서도 FIFO가 된다.

#### 여러 consumer

```text
consumer 1: A 시작 ---------------- A 완료
consumer 2:   B 시작 ---- B 완료

dequeue: A -> B
완료:    B -> A
```

FIFO와 처리량을 위한 병렬성 사이에 순서 trade-off가 생긴다.

### head-of-line blocking

앞 작업이 오래 걸리면 뒤 작업이 준비되어 있어도 기다린다.

```text
Queue: [A 30초] [B 10ms] [C 10ms]
              ^ B와 C는 A 뒤에서 대기
```

작업을 독립 partition으로 나누거나, deadline/priority 기준을 도입하거나, 여러 worker로
병렬화할 수 있다. 하지만 각 방식은 전역 FIFO 보장을 약하게 만들 수 있다.

---

## 5. 핵심 연산

FIFO: First In First Out 자체가 독립 ADT 연산을 정의하지 않는 경우에는 상위 자료구조의 삽입·조회·삭제 과정에서 이 개념이 사용되는 지점을 기준으로 설명한다.

`ADT`는 내부 코드를 정하는 말이 아니라, 사용할 수 있는 연산과 그 연산이 지켜야 할
규칙을 정한 사용 계약이다. 이 용어 자체에 독립 연산이 없다면 실제 자료구조의 어느
동작에 참여하는지를 따라가며 이해한다.

---

## 6. 상태 추적과 구현

위 핵심 연산의 예시를 입력부터 결과까지 따라가며 값·index·Node·Stack·Queue 중 실제로 변하는 상태를 확인한다.

---

## 7. 불변식

정상 상태에서 유지해야 할 구체 조건은 위 내부 구조와 연산 설명을 기준으로 확인한다. 상위 자료구조의 불변식과 충돌하는 변경은 허용하지 않는다.

여기서 `불변식`은 삽입·조회·삭제 전후에도 계속 참이어야 하는 규칙이다. 쉽게 말해
자료구조가 망가지지 않았다고 판단하는 필수 조건이며, 연산이 끝난 뒤에도 다시 맞아야 한다.

---

## 8. 실패·예외·경계 상황

### 적용과 한계

작업 도착 순서가 중요할 때 자연스럽지만 긴 작업 A가 앞에 있으면 짧은 B와 C도
기다리는 head-of-line blocking이 생길 수 있다. 우선순위나 마감 시간이 중요하면
Priority Queue, 공정한 멀티스레드 처리가 필요하면 별도의 스케줄링 정책이 필요하다.

---

## 9. 성능과 비용

FIFO: First In First Out 자체가 독립 연산을 모두 정의하는 것은 아니다. 접근 비용은 배열·Node·Tree 등 실제 저장 표현과 상위 자료구조의 연산 계약을 기준으로 판단한다.

---

## 10. 대안 비교와 선택 기준

이 개념만 떼어 자료구조를 선택하지 않는다. 필요한 조회·삽입·삭제·정렬·범위·순회 연산을 기준으로 인접 개념과 상위 자료구조를 비교한다.

## 14. 꼬리질문 방어

### 질문 지도

| 축 | 질문 | 답변 핵심 |
|---|---|---|
| 내부 | FIFO는 무엇의 순서를 보장하나? | enqueue와 dequeue의 상대 순서이며 처리 완료 시점은 별개다. |
| 실패 | head-of-line blocking이란? | 앞의 긴 작업 때문에 뒤의 짧은 작업도 시작하지 못하는 현상이다. |
| 비교 | 우선 작업을 먼저 처리해야 하면? | FIFO가 아니라 Priority Queue나 scheduler 정책을 사용한다. |

### Q1. FIFO가 항상 공정성을 보장하나?

<details><summary>답 보기</summary>

순서 정책 자체는 먼저 들어온 항목을 먼저 처리하지만, 실제 시스템 공정성은 우선순위, 타임아웃, 재시도, 스케줄러 정책에 따라 달라질 수 있다.

</details>

### Q2. 메시지 큐를 FIFO라고 하면 처리 완료 순서도 같은가?

<details><summary>답 보기</summary>

반드시 그렇지 않다. 여러 consumer가 병렬 처리하면 전달 시작 순서는 FIFO여도 작업
시간 차이로 완료 순서는 바뀔 수 있다. 제품의 partition, consumer, 재시도 계약을
별도로 확인해야 한다.

</details>

### Q3. FIFO는 데이터 저장 위치도 순서대로라는 뜻인가?

<details><summary>답 보기</summary>

아니다. FIFO는 관찰 가능한 제거 순서다. 원형 배열은 물리적으로 뒤 index 다음에 0번
index를 사용해도 dequeue 결과는 FIFO일 수 있다.

</details>

### Q4. 재시도가 있으면 FIFO가 깨질 수 있는가?

<details><summary>답 보기</summary>

가능하다. A 처리 실패 후 뒤로 재삽입하고 B가 성공하면 완료 순서는 B, A가 된다.
정확한 보장은 사용하는 Queue나 메시지 시스템의 재시도 계약을 확인해야 한다.

</details>

### Q5. FIFO와 round-robin은 같은가?

<details><summary>답 보기</summary>

아니다. FIFO는 하나의 대기열에서 도착 순서를 따르는 정책이다. round-robin은 여러
대상에 처리 기회를 순환 배분하는 스케줄링 정책이다.

</details>

### Q6. FIFO가 적합하지 않은 요구사항은?

<details><summary>답 보기</summary>

긴급도, deadline, 작업 비용이 도착 순서보다 중요한 경우다. Priority Queue나 별도
scheduler가 더 직접적일 수 있다.

</details>
