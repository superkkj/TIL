## 1. 정의

### 정석 설명

hash는 입력 값을 해시 테이블의 위치 계산 등에 쓰기 쉬운 값으로 변환한 결과다. Java HashMap에서는 key의 `hashCode()`와 내부 보정 `hash()`가 bucket index 계산에 사용된다.

### 쉬운 설명

긴 이름표를 숫자 지문처럼 바꾼 것이다. 이 숫자 지문으로 사물함 위치 후보를 계산한다.

### 어려운 말 먼저 풀기

| 용어 | 쉬운 뜻 |
|---|---|
| hash | 원래 key를 계산하기 쉬운 정수 값으로 바꾼 결과다. 원본 그 자체나 배열 위치는 아니다. |
| hash function | key를 넣으면 hash를 돌려주는 계산 규칙이다. |
| 결정적(deterministic) | 같은 실행 조건에서 같은 key를 넣으면 같은 결과가 나오는 성질이다. |
| 비트 섞기 | hash의 위쪽 비트 정보도 아래쪽 비트에 반영되도록 섞어 bucket 쏠림을 줄이는 계산이다. |
| bucket index | hash를 현재 table 크기에 맞는 실제 칸 번호로 줄인 최종 위치 후보다. |

### 다시 정리

쉬운 설명은 이해를 위한 비유다. 실제 면접 답변에서는 위 정석 설명의 용어와
전제 조건으로 돌아와 설명한다.

---

## 2. 탄생 배경과 필요성

### 왜 필요한가

문자열, 객체, 이메일 같은 key를 배열 index로 바로 쓸 수 없다. hash는 다양한 key를 배열 위치 계산이 가능한 숫자 세계로 옮긴다.

---

## 3. 해결 범위와 비목표

이 문서가 직접 설명하는 범위는 hash: 해시의 정의와 상위 자료구조에서의 역할이다. 정렬·동시성·영속성·특정 시간 복잡도는 별도 불변식이나 구현 계약이 있을 때만 보장된다.

쉽게 말하면 이 개념의 이름만 보고 다음 성질까지 자동으로 있다고 단정하면 안 된다.
`동시성`은 여러 실행 흐름이 함께 접근해도 안전한지, `영속성`은 프로그램이 끝난 뒤에도
파일이나 DB 등에 데이터가 남는지, `구현 계약`은 실제 클래스나 API가 무엇을 약속하는지를
뜻한다. 이 성질들은 사용하는 구현체의 공식 설명에서 각각 확인해야 한다.

---

## 4. 내부 구조와 핵심 원리

### 입력에서 bucket까지의 세 단계

```text
원본 key -> hash function -> hash value -> compression -> bucket index
```

Java `HashMap`에서는 key의 `hashCode()`가 첫 정수 값을 만들고, OpenJDK 구현의
내부 `hash()`가 비트를 섞은 뒤, table 길이에 맞춰 index를 계산한다. `hashCode`,
내부 hash, bucket index는 같은 값이 아니다.

### 좋은 비암호학적 hash의 성질

- 결정성: 같은 상태의 같은 key는 같은 값을 낸다.
- 분산성: 실제 입력 집합을 bucket 전체에 가능한 한 고르게 퍼뜨린다.
- 계산 비용: hash 계산 자체가 조회보다 지나치게 비싸지 않다.
- 작은 입력 차이가 특정 bucket에 반복적으로 몰리지 않게 한다.

“충돌이 절대 없다”는 조건은 아니다. 유한한 출력 공간에 더 많은 입력을 매핑하면
비둘기집 원리에 의해 충돌은 피할 수 없다.

### 암호학적 hash와 구분

HashMap용 hash는 빠른 bucket 분산이 목적이다. SHA-256 같은 암호학적 hash는
역상·충돌 공격에 대한 보안 성질이 목적이다. 둘 다 입력을 고정 크기 값으로
바꾸지만 요구사항과 비용이 다르다.

### OpenJDK 내부 비트 섞기

```java
static int hash(Object key) {
    int h;
    return key == null ? 0 : (h = key.hashCode()) ^ (h >>> 16);
}
```

최종 index 계산이 table 길이에 따라 hash의 일부 낮은 비트를 사용하므로 높은 비트도
낮은 쪽 계산에 참여시키려는 구현이다. 이 식은 Java API 보장이 아니라 OpenJDK 17u
구현 세부다.

## 5. 핵심 연산

### String hashCode 계산

Java 17 `String.hashCode()` 계약은 문자열 문자값을 31을 사용한 다항식으로 계산한다.
개념적으로 왼쪽부터 다음처럼 누적할 수 있다.

```text
h = 0
h = 31 * h + 현재 문자 값
```

`"ABC"`:

```text
'A'=65: 0 * 31 + 65 = 65
'B'=66: 65 * 31 + 66 = 2081
'C'=67: 2081 * 31 + 67 = 64578
```

서로 다른 문자열도 같은 결과를 만들 수 있다.

```text
"Aa" = 65 * 31 + 97 = 2112
"BB" = 66 * 31 + 66 = 2112

hashCode 같음
equals false
```

## 6. 상태 추적과 구현

### 손 추적

```text
hash = 35
capacity = 16
mask = 15
index = 35 & 15 = 3
```

이 예에서 `35`는 hash이고 `3`은 bucket index다. capacity가 바뀌면 같은 hash도
다른 index로 갈 수 있다.

### Java key가 bucket에 도착하는 세 단계

OpenJDK 17u `HashMap` 기준:

```text
key
 ↓ key.hashCode()
객체가 계산한 int
 ↓ h ^ (h >>> 16)
HashMap 내부에서 비트를 섞은 hash
 ↓ hash & (table.length - 1)
현재 table의 bucket index
```

`hashCode`, 내부 hash, bucket index는 같은 값이 아니다.

---

## 7. 불변식

### 결정성과 불변 key

저장 때와 조회 때 같은 key 상태가 같은 hash를 만들어야 같은 시작 위치로 갈 수 있다.
key의 동등성 관련 필드를 저장 후 변경하면 다른 hash가 나와 기존 entry를 못 찾을 수
있다.

---

## 8. 실패·예외·경계 상황

### 주의

일반적인 hash는 원본의 대소 관계를 보존하도록 설계하지 않는다. 따라서 hash 값의
대소만 보고 원본 key의 대소나 인접 관계를 판단할 수 없다.

---

## 9. 성능과 비용

hash: 해시 자체가 독립 연산을 모두 정의하는 것은 아니다. 접근 비용은 배열·Node·Tree 등 실제 저장 표현과 상위 자료구조의 연산 계약을 기준으로 판단한다.

## 14. 꼬리질문 방어

### 질문 지도

| 축 | 질문 | 답변 핵심 |
|---|---|---|
| 내부 | hash와 bucket index는 왜 다른가? | hash를 현재 capacity 범위로 압축한 결과가 index다. |
| 실패 | 결정성이 깨지면? | put과 get의 시작 위치가 달라져 저장한 key를 못 찾는다. |
| 비교 | HashMap hash와 SHA-256 차이는? | 전자는 빠른 분산, 후자는 보안 성질이 핵심이다. |

### Q1. hash만 같으면 같은 key인가?

<details><summary>답 보기</summary>

아니다. 서로 다른 key도 같은 hash를 가질 수 있다. hash는 위치 후보를 찾는 값이고, 진짜 같은 key인지는 equals로 확인해야 한다.

</details>

### Q2. hash는 원본을 압축한 값인데 복원할 수 있나?

<details><summary>답 보기</summary>

일반적으로 hash만으로 원본을 유일하게 복원할 수 없다. 여러 입력이 같은 출력으로
갈 수 있기 때문이다. HashMap이 실제 entry에 원본 key도 보관하는 이유 중 하나다.

</details>

### Q3. 좋은 hash면 충돌이 전혀 없는가?

<details><summary>답 보기</summary>

아니다. 일반적인 큰 입력 공간을 유한한 출력에 매핑하면 충돌은 가능하다. 좋은 hash는
실제 입력을 충분히 고르게 분산해 특정 bucket 집중을 줄이는 것이 핵심이다.

</details>

### Q4. hash 값의 대소로 원본 key를 정렬할 수 있는가?

<details><summary>답 보기</summary>

일반 hash는 원본 대소 관계를 보존하지 않는다. 범위·정렬이 필요하면 comparator 기반
tree나 별도 정렬 index를 사용한다.

</details>

### Q5. HashMap용 hash와 SHA-256을 같은 목적으로 쓰는가?

<details><summary>답 보기</summary>

아니다. HashMap용 hash는 빠른 bucket 분산이 목적이고 암호학적 hash는 역상·충돌 공격
저항 같은 보안 성질이 중요하다.

</details>
