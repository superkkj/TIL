## 1. 정의

### 정석 설명

O(1)은 입력 크기가 커져도 연산 비용이 입력 크기에 비례해 증가하지 않는다는 뜻이다.

### 쉬운 설명

데이터가 10개든 100만 개든 바로 특정 칸으로 가는 느낌이다.

### 어려운 말 먼저 풀기

| 용어 | 쉬운 뜻 |
|---|---|
| constant time | 입력 크기 `n`이 커져도 연산 단계 수가 `n`에 비례해 늘지 않는다는 뜻이다. |
| direct access | index나 주소 계산으로 원하는 위치에 바로 접근하는 방식이다. |
| worst-case | 가능한 입력 중 작업량이 가장 큰 경우의 비용이다. |
| average-case | 정한 입력 분포나 구현 전제에서 평균적으로 드는 비용이다. 전제 없이 “평균”이라고만 말하면 불완전하다. |
| amortized | 가끔 비싼 연산이 있어도 긴 연산 묶음 전체 비용을 연산 수로 나누어 보는 분석이다. |
| O(1) ≠ 한 번 | 실제 계산이 여러 단계여도 단계 수가 `n`에 따라 커지지 않으면 O(1)일 수 있다. |

### 다시 정리

쉬운 설명은 이해를 위한 비유다. 실제 면접 답변에서는 위 정석 설명의 용어와
전제 조건으로 돌아와 설명한다.

---

## 2. 배우는 이유와 실제 쓰임

| 질문 | 먼저 잡을 답 |
|---|---|
| 왜 배우나 | 입력 `n`이 커져도 연산 단계 수가 `n`에 비례해 늘지 않는다는 뜻과, 평균·기대·상환 O(1)을 구분하기 위해 배운다. |
| 판단 근거 | NIST Big-O 정의에서 O(1)은 입력 크기에 대한 상수 상한이다. Oracle `HashMap`의 상수 시간 설명에는 hash 분산이라는 전제가 붙는다. |
| 실제로 어디에 쓰이나 | 배열 index 접근, 배열 기반 Stack·Queue 구현의 끝 원소 확인, 적절히 분산된 hash table의 기대 조회처럼 입력 전체를 순회하지 않는 연산을 분석할 때 쓴다. 정확한 보장은 구현 계약을 확인한다. |
| 기억할 장면 | 상자 안 물건이 10개든 100만 개든 정해진 칸 하나만 바로 본다. |

---

## 3. 해결 범위와 비목표

이 문서가 직접 설명하는 범위는 O(1): 상수 시간의 정의와 상위 자료구조에서의 역할이다. 정렬·동시성·영속성·특정 시간 복잡도는 별도 불변식이나 구현 계약이 있을 때만 보장된다.

쉽게 말하면 이 개념의 이름만 보고 다음 성질까지 자동으로 있다고 단정하면 안 된다.
`동시성`은 여러 실행 흐름이 함께 접근해도 안전한지, `영속성`은 프로그램이 끝난 뒤에도
파일이나 DB 등에 데이터가 남는지, `구현 계약`은 실제 클래스나 API가 무엇을 약속하는지를
뜻한다. 이 성질들은 사용하는 구현체의 공식 설명에서 각각 확인해야 한다.

## 5. 핵심 연산

O(1): 상수 시간 자체가 독립 ADT 연산을 정의하지 않는 경우에는 상위 자료구조의 삽입·조회·삭제 과정에서 이 개념이 사용되는 지점을 기준으로 설명한다.

`ADT`는 내부 코드를 정하는 말이 아니라, 사용할 수 있는 연산과 그 연산이 지켜야 할
규칙을 정한 사용 계약이다. 이 용어 자체에 독립 연산이 없다면 실제 자료구조의 어느
동작에 참여하는지를 따라가며 이해한다.

---

## 6. 상태 추적과 구현

위 핵심 연산의 예시를 입력부터 결과까지 따라가며 값·index·Node·Stack·Queue 중 실제로 변하는 상태를 확인한다.

---

## 7. 불변식

정상 상태에서 유지해야 할 구체 조건은 위 내부 구조와 연산 설명을 기준으로 확인한다. 상위 자료구조의 불변식과 충돌하는 변경은 허용하지 않는다.

여기서 `불변식`은 삽입·조회·삭제 전후에도 계속 참이어야 하는 규칙이다. 쉽게 말해
자료구조가 망가지지 않았다고 판단하는 필수 조건이며, 연산이 끝난 뒤에도 다시 맞아야 한다.

---

## 8. 실패·예외·경계 상황

### 주의

O(1)이 항상 매우 빠르다는 뜻은 아니다. 상수 비용이 클 수 있고, HashMap처럼 평균 O(1)은 전제가 깨지면 나빠질 수 있다.

---

## 9. 성능과 비용

### O(1)이 의미하는 것

연산 단계 수가 입력 크기 n에 따라 증가하지 않는다는 점근적 상한이다. 실제 한 번의
연산이 CPU 명령 하나라는 뜻은 아니다. 고정된 1,000단계도 O(1)이고 매우 큰 객체를
복사한다면 그 객체 크기를 입력 변수에서 누락했는지 다시 봐야 한다.

### 구현체와 비용 종류를 붙인다

| 연산 | 정확한 표현 |
|---|---|
| 배열 `a[i]` | 유효 index 직접 접근 O(1) |
| 연결 Stack push/pop | O(1) |
| 동적 배열 Stack push | 상환 O(1), 확장 시 O(n) |
| 원형 배열 Queue offer/poll | 일반 연산 O(1), 확장 시 O(n) 가능 |
| Java PriorityQueue offer/poll | O(log n), O(1)이 아님 |
| HashMap get | hash 분산 전제의 기대 O(1) |

ADT 이름만으로 복잡도를 확정하지 않고 구현과 전제를 말한다.

### O(1)을 판단하는 질문

```text
1. 입력 크기 n은 무엇인가?
2. n이 두 배가 될 때 실행 단계 수가 늘어나는가?
3. 내부에서 복사·순회·hash 계산이 숨어 있는가?
4. worst, expected, amortized 중 무엇인가?
```

#### 예 1: 배열 index

```java
int value = values[index];
```

배열 길이 n에 대해 한 index 접근은 O(1)이다. 그러나 index가 유효한지 확인하거나
주소 계산하는 고정 작업이 “0 비용”이라는 뜻은 아니다.

#### 예 2: 첫 원소 찾기

```java
return values[0];
```

배열이 비었을 때 예외 처리가 필요하지만, n에 따른 반복은 없어 O(1)이다.

#### 반례: 첫 번째 0 찾기

```java
for (int value : values) {
    if (value == 0) return value;
}
```

첫 원소가 0이면 best O(1)이지만 부재하면 worst O(n)이다. 조기 종료 가능성과 최악
복잡도를 혼동하지 않는다.

### O(1) 공간의 함정

정수 변수 몇 개만 사용하면 보통 입력 n에 대해 O(1) 추가 공간이다. 하지만 “한 객체”가
n 크기 내부 배열을 가진다면 변수 하나라고 O(1) 공간이 아니다. 참조 개수와 참조가
가리키는 새 저장 공간을 함께 본다.

## 14. 꼬리질문 방어

### 질문 지도

| 축 | 질문 | 답변 핵심 |
|---|---|---|
| 내부 | O(1)은 명령 한 개인가? | 아니며 n과 무관한 상수 단계라는 점근 표현이다. |
| 실패 | 동적 배열 push를 최악 O(1)이라 하면? | 확장 시 O(n) 복사를 놓친 것으로 상환 O(1)이라 해야 한다. |
| 비교 | HashMap O(1)과 배열 O(1) 차이는? | 전자는 분산 가정의 기대 비용, 후자는 index 직접 접근이다. |

### Q1. HashMap get은 항상 O(1)인가?

<details><summary>답 보기</summary>

Javadoc 기준으로 hash가 잘 분산된다는 전제에서 기본 연산이 constant-time 성능을 가진다. 충돌이 심하면 나빠질 수 있다.

</details>

### Q2. 문자열 key HashMap get도 key 길이와 무관한 O(1)인가?

<details><summary>답 보기</summary>

Map entry 수 n에 대해서는 기대 O(1)이라고 표현하지만, hash가 아직 계산되지 않은
길이 m 문자열의 hash 계산과 충돌 시 equals 비교는 O(m)일 수 있다. 어떤 입력 크기를
분석하는지에 따라 `O(m)` 또는 `O(1)` 설명이 달라진다.

</details>

### Q3. 1초 sleep은 O(1)인가?

<details><summary>답 보기</summary>

입력 크기 n과 무관한 고정 대기라면 점근적으로 O(1)이라고 표현할 수 있다. 하지만
실제 지연은 매우 크므로 Big-O가 빠름을 의미하지 않는 좋은 예다.

</details>

### Q4. 배열 마지막 원소 접근도 O(1)인가?

<details><summary>답 보기</summary>

유효 index `length-1`을 계산해 직접 접근하므로 배열 길이에 따른 순회가 없어 O(1)이다.

</details>

### Q5. 동적 배열 append를 왜 평균 O(1)이 아니라 상환 O(1)이라고 하나?

<details><summary>답 보기</summary>

확장 시점의 O(n) 복사가 있지만 여러 append 연산 열에 복사 비용을 분산하면 연산당
O(1)로 제한되기 때문이다. 입력 확률 분포의 평균이 아니다.

</details>
