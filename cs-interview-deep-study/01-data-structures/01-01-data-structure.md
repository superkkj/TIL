## 1. 정의

### 정석 설명

자료구조는 데이터를 저장하고 접근하는 방식을 정한 구조다. 같은 데이터라도 배열, 리스트, 해시 테이블, 트리, 힙 중 어디에 놓느냐에 따라 조회, 삽입, 삭제, 순회, 정렬, 범위 검색 비용이 달라진다.

### 쉬운 설명

책을 아무 상자에나 넣으면 넣기는 쉽지만 찾기가 어렵다. 가나다순 책장에 꽂으면 찾기는 쉽지만 중간에 새 책을 넣을 때 자리를 조정해야 한다. 자료구조는 "어떤 일을 쉽게 만들기 위해 데이터를 어떻게 놓을 것인가"의 선택이다.

### 어려운 말 먼저 풀기

| 용어 | 쉬운 뜻 |
|---|---|
| 원소(element) | 구조 안에 저장하는 데이터 한 개다. 배열의 한 칸에 든 값이 원소다. |
| 연산(operation) | 데이터를 넣기, 찾기, 지우기처럼 자료구조에 시키는 일이다. |
| 불변식(invariant) | 연산 전후에도 반드시 지켜져야 하는 규칙이다. 예를 들어 Stack은 중간이 아니라 맨 위에서 꺼낸다. |
| ADT(추상 자료형) | 내부 코드보다 “무슨 일을 어떤 규칙으로 제공하는가”를 정의한 사용 설명서다. |
| 구현체 | ADT의 규칙을 실제 배열, Node, 코드로 만든 것이다. |

### 다시 정리

쉬운 설명은 이해를 위한 비유다. 실제 면접 답변에서는 위 정석 설명의 용어와
전제 조건으로 돌아와 설명한다.

## 2. 배우는 이유와 실제 쓰임

| 질문 | 먼저 잡을 답 |
|---|---|
| 왜 배우나 | 같은 데이터도 어떻게 놓느냐에 따라 조회·삽입·삭제 비용이 달라진다. 요구사항에 맞는 구조를 고르는 판단력을 기르기 위해 배운다. |
| 판단 근거 | 위 NIST 사전은 자료구조별 정의와 연산을 구분하고, Java Collections Framework도 `List`, `Set`, `Queue`, `Map`을 서로 다른 계약으로 제공한다. |
| 실제로 어디에 쓰이나 | 순서 보존은 `List`, 중복 제거는 `Set`, key 조회는 `Map`, 대기 작업은 `Queue`, 우선순위 처리는 `PriorityQueue`처럼 프로그램의 저장 방식을 정할 때 쓴다. |
| 기억할 장면 | 코드를 쓰기 전에 “이 데이터로 가장 자주 할 일이 무엇인가?”를 먼저 묻는다. |

### 왜 필요한가

데이터가 적을 때는 아무 구조나 써도 차이가 작다. 데이터가 많아지거나 요청이 많아지면 핵심 연산 비용이 서비스 성능을 결정한다.

### 문제의 뿌리

컴퓨터 메모리는 결국 바이트의 연속이지만, 프로그램이 원하는 연산은 서로 다르다.
`i`번째 값을 즉시 읽고 싶은 경우, key로 값을 찾는 경우, 가장 작은 값을 계속
꺼내는 경우를 같은 배치로 모두 최적화할 수는 없다. 자료구조는 데이터의 **논리적
관계**, 메모리의 **물리적 배치**, 외부에 제공하는 **연산 계약**을 묶은 선택이다.

```text
요구사항 -> 빠르게 해야 할 연산 -> 필요한 불변식 -> 자료구조 선택
```

## 3. 해결 범위와 비목표

이 문서가 직접 설명하는 범위는 자료구조의 정의와 상위 자료구조에서의 역할이다. 정렬·동시성·영속성·특정 시간 복잡도는 별도 불변식이나 구현 계약이 있을 때만 보장된다.

쉽게 말하면 이 개념의 이름만 보고 다음 성질까지 자동으로 있다고 단정하면 안 된다.
`동시성`은 여러 실행 흐름이 함께 접근해도 안전한지, `영속성`은 프로그램이 끝난 뒤에도
파일이나 DB 등에 데이터가 남는지, `구현 계약`은 실제 클래스나 API가 무엇을 약속하는지를
뜻한다. 이 성질들은 사용하는 구현체의 공식 설명에서 각각 확인해야 한다.

## 4. 내부 구조와 핵심 원리

### 내부를 보는 네 층

| 층 | 확인할 질문 | 예 |
|---|---|---|
| 논리 모델 | 데이터가 어떤 관계인가? | 순서, 계층, key-value |
| 저장 단위 | 무엇을 한 덩어리로 저장하는가? | element, node, entry |
| 연결 방식 | 다음 데이터를 어떻게 찾는가? | index 계산, 참조, 비교 |
| 연산 계약 | 무엇을 보장하는가? | 중복, null, 순서, 복잡도 |

### ADT, 자료구조, 구현체 구분

```text
ADT/인터페이스
  무엇을 할 수 있는가: List, Queue, Map

자료구조 원리
  어떤 불변식으로 구현하는가: dynamic array, linked list, hash table, tree

구현체
  구체 코드와 계약: ArrayList, LinkedList, HashMap, TreeMap
```

`Queue`라고 해서 모두 FIFO는 아니며 `PriorityQueue`는 우선순위로 head를 정한다.
인터페이스 이름만 보고 내부 비용을 확정하지 않는다.

### 구조 손상을 검증하는 방법

자료구조 구현 테스트는 반환값뿐 아니라 내부 불변식을 매 연산 뒤 확인해야 한다.

```text
Linked List
  forward/backward link가 서로 일치하는가?

BST
  모든 subtree 허용 범위가 유지되는가?

Heap
  shape와 parent-child 우선순위가 유지되는가?

Hash Table
  모든 entry를 자신의 hash 탐색 경로로 다시 찾을 수 있는가?
```

정상 입력, 빈 구조, 원소 하나, 중복, 최대 용량, 삭제 후 재삽입, 최악 모양을 포함한다.

## 5. 핵심 연산

자료구조 자체가 독립 ADT 연산을 정의하지 않는 경우에는 상위 자료구조의 삽입·조회·삭제 과정에서 이 개념이 사용되는 지점을 기준으로 설명한다.

`ADT`는 내부 코드를 정하는 말이 아니라, 사용할 수 있는 연산과 그 연산이 지켜야 할
규칙을 정한 사용 계약이다. 이 용어 자체에 독립 연산이 없다면 실제 자료구조의 어느
동작에 참여하는지를 따라가며 이해한다.

위 핵심 연산의 예시를 입력부터 결과까지 따라가며 값·index·Node·Stack·Queue 중 실제로 변하는 상태를 확인한다.

## 7. 불변식

정상 상태에서 유지해야 할 구체 조건은 위 내부 구조와 연산 설명을 기준으로 확인한다. 상위 자료구조의 불변식과 충돌하는 변경은 허용하지 않는다.

여기서 `불변식`은 삽입·조회·삭제 전후에도 계속 참이어야 하는 규칙이다. 쉽게 말해
자료구조가 망가지지 않았다고 판단하는 필수 조건이며, 연산이 끝난 뒤에도 다시 맞아야 한다.

---

## 8. 실패·예외·경계 상황

### 불변식과 실패

자료구조의 연산 전후에는 구조가 약속한 규칙이 유지되어야 한다. 배열은 유효
index 범위, 연결 리스트는 끊어지지 않은 링크, BST는 정렬 관계, heap은 부모-자식
우선순위가 불변식이다. 연산 결과만 맞아 보여도 불변식이 깨지면 다음 연산에서
누락, 무한 순회, 잘못된 탐색이 발생한다.

---

## 9. 성능과 비용

### 비용을 말하는 법

시간 복잡도만 말하지 않는다. 추가 메모리, 캐시 지역성, 재배치 비용, 동시 접근,
구현 복잡도도 함께 본다. 평균 O(1)과 최악 O(n), O(1)과 상환 O(1)은 서로 다른
주장이므로 전제를 붙여야 한다.

## 10. 대안 비교와 선택 기준

### 자료구조 선택 질문

| 질문 | 후보 |
|---|---|
| index로 바로 접근해야 하는가? | Array, ArrayList |
| key로 value를 찾아야 하는가? | Hash Table, HashMap |
| 정렬/범위가 필요한가? | Tree, B+Tree |
| 우선순위가 필요한가? | Heap, Priority Queue |
| 먼저 들어온 순서가 중요한가? | Queue |
| 나중에 들어온 것을 먼저 꺼내야 하는가? | Stack |

### 선택을 손으로 추적하는 예

`회원 ID로 회원 한 명 조회`는 정확한 key 조회이므로 HashMap이 후보가 된다.
`점수 80~90점 회원 조회`는 범위 조건이므로 정렬 트리나 DB 인덱스가 후보가 된다.
같은 회원 데이터라도 질문이 달라지면 적합한 자료구조도 달라진다.

## 12. 실무 판단 기준

### 선택을 연산 비율로 구체화하기

“조회가 많다”만으로는 부족하다. 어떤 조회인지와 쓰기 비용을 함께 적는다.

```text
요구사항 예
  key 단건 조회: 초당 10,000회
  신규 삽입:     초당 10회
  key 범위 조회: 없음
  정렬 순회:     없음

후보
  HashMap: 단건 key 조회에 적합
  TreeMap: 필요 없는 정렬 유지 비용 추가
  List: 조회마다 전체 순회 가능
```

반대로 범위 조회가 핵심이면 평균 O(1)이라는 이유만으로 HashMap을 선택하면 안 된다.

## 14. 꼬리질문 방어

### 질문 지도

| 축 | 질문 | 답변 핵심 |
|---|---|---|
| 내부 | 자료구조와 알고리즘은 어떻게 다른가? | 자료구조는 데이터 관계와 연산 계약, 알고리즘은 그 구조를 이용하는 절차다. |
| 실패 | Big-O만 보고 고르면 왜 위험한가? | 메모리, cache, 입력 분포, 동시성, 구현 상수가 빠진다. |
| 선택 | 읽기와 쓰기 요구가 모두 강하면? | 한 구조로 강제하지 말고 보조 index와 정합성 비용을 함께 설계한다. |

### Q1. 자료구조를 고르는 첫 기준은?

<details><summary>답 보기</summary>

저장할 데이터 모양보다 먼저 "가장 자주 하는 연산"을 본다. 조회, 삽입, 삭제, 순회, 정렬, 범위 검색 중 무엇이 핵심인지가 선택 기준이다.

</details>

### Q2. 같은 데이터를 두 자료구조에 중복 저장해도 되나?

<details><summary>답 보기</summary>

가능하지만 쓰기 때 두 구조를 원자적으로 갱신해야 한다. 조회 성능을 얻는 대신
메모리와 정합성 책임을 지는 공간-시간 교환이다.

</details>

### Q3. 자료구조가 다르면 같은 알고리즘 복잡도도 달라지는가?

<details><summary>답 보기</summary>

그렇다. binary search는 중간 index O(1) 접근이 가능한 정렬 배열에서는 O(log n)이지만,
LinkedList는 중간 node 이동 비용 때문에 같은 방식의 전체 비용이 O(log n)이 아니다.

</details>

### Q4. 가장 빠른 자료구조 하나로 통일하면 안 되는가?

<details><summary>답 보기</summary>

연산마다 강점이 다르다. HashMap은 단건 key 조회, TreeMap은 정렬·범위, Heap은 head
우선순위에 강하다. 한 연산을 빠르게 만들면 메모리·갱신·순서 비용을 지불할 수 있다.

</details>

### Q5. 보조 index를 추가하면 무엇이 어려워지는가?

<details><summary>답 보기</summary>

쓰기 때 원본과 index를 함께 갱신해야 한다. 중간 실패나 동시 변경에서 둘이 달라지지
않도록 정합성·원자성·복구 전략이 필요하다.

</details>

### Q6. Big-O 외에 실무에서 확인할 것은?

<details><summary>답 보기</summary>

실제 입력 크기와 분포, 메모리, cache locality, 객체 할당, 동시성, I/O, 구현 상수,
운영 측정값을 확인한다.

</details>

## 15. 최종 점검

### 문서를 덮고 답하기

```text
자료구조을 한 문장으로 정의할 수 있는가?
왜 필요한지 이전 방식의 한계와 연결할 수 있는가?
내부 구조와 핵심 연산을 손으로 추적할 수 있는가?
연산 전후에도 반드시 지켜져야 하는 규칙인 불변식과, 그 규칙이 깨지는 실패 상황을 설명할 수 있는가?
보통 입력의 평균 비용, 가장 나쁜 입력의 최악 비용, 가끔 발생하는 큰 비용을 여러 연산에 나누는 상환 비용 중 무엇인지 전제를 붙여 말할 수 있는가?
대안 자료구조와 선택 기준을 비교할 수 있는가?
```
