## 1. 정의

### 정석 설명

leaf는 자식이 없는 노드다. 트리의 끝에 위치한다.

### 쉬운 설명

더 이상 아래로 내려갈 곳이 없는 마지막 폴더나 조직도 맨 아래 팀원이다.

### 어려운 말 먼저 풀기

| 용어 | 쉬운 뜻 |
|---|---|
| leaf | child가 하나도 없는 끝 Node다. |
| internal node | leaf가 아니며 하나 이상의 child를 가진 Node다. |
| 종료 조건(base case) | 재귀 호출을 더 하지 않고 답을 반환하는 조건이다. leaf는 트리 재귀에서 자주 종료 지점이 된다. |
| leaf depth | root에서 해당 leaf까지 지나간 edge 수다. 일반 트리에서는 모든 leaf가 같은 depth일 필요가 없다. |
| B+Tree leaf | 실제 data entry 또는 그 위치가 모이고 옆 leaf와 연결될 수 있는 맨 아래 Node다. |

### 다시 정리

쉬운 설명은 이해를 위한 비유다. 실제 면접 답변에서는 위 정석 설명의 용어와
전제 조건으로 돌아와 설명한다.

## 2. 배우는 이유와 실제 쓰임

| 질문 | 먼저 잡을 답 |
|---|---|
| 왜 배우나 | 더 내려갈 child가 없는 지점을 알아야 재귀 종료, 탐색 실패·성공의 끝, height 계산의 기준을 정할 수 있다. |
| 판단 근거 | NIST는 leaf를 child가 없는 tree node로 정의한다. root도 child가 없다면 동시에 leaf가 될 수 있다. |
| 실제로 어디에 쓰이나 | tree 순회의 종료 조건, subtree 집계의 기저 사례, B+Tree의 leaf 순회처럼 가장 아래 level을 처리할 때 쓴다. |
| 기억할 장면 | 가지 끝에 더 갈 곳이 없는 마지막 잎이다. |

### 왜 필요한가

탐색 종료, 높이 계산, 트리 순회 종료 조건에서 leaf 개념이 중요하다.

## 3. 해결 범위와 비목표

이 문서가 직접 설명하는 범위는 Leaf: 리프 노드의 정의와 상위 자료구조에서의 역할이다. 정렬·동시성·영속성·특정 시간 복잡도는 별도 불변식이나 구현 계약이 있을 때만 보장된다.

쉽게 말하면 이 개념의 이름만 보고 다음 성질까지 자동으로 있다고 단정하면 안 된다.
`동시성`은 여러 실행 흐름이 함께 접근해도 안전한지, `영속성`은 프로그램이 끝난 뒤에도
파일이나 DB 등에 데이터가 남는지, `구현 계약`은 실제 클래스나 API가 무엇을 약속하는지를
뜻한다. 이 성질들은 사용하는 구현체의 공식 설명에서 각각 확인해야 한다.

## 4. 내부 구조와 핵심 원리

### 예

```text
A
├─ B
└─ C
```

B와 C에 자식이 없으면 B와 C는 leaf다.

### leaf를 판정하는 방법

일반 rooted tree에서는 children 수가 0인 node다. binary tree에서는 `left == null &&
right == null`, 배열 heap에서는 두 child index가 모두 size 범위 밖인지로 판단한다.
저장 표현은 달라도 논리적으로 아래에 더 진행할 node가 없다는 뜻은 같다.

### leaf가 알고리즘의 base case가 되는 이유

재귀 알고리즘은 더 내려갈 child가 없을 때 결과를 확정한다.

```java
int countLeaves(Node node) {
    if (node.children.isEmpty()) return 1;

    int count = 0;
    for (Node child : node.children) {
        count += countLeaves(child);
    }
    return count;
}
```

binary tree에서는 null 자체를 base case로 두는 방식도 있다.

```text
countLeaves(null) = 0
countLeaves(leaf) = 1
```

null child와 실제 leaf node를 같은 것으로 혼동하면 node 수와 leaf 수가 달라진다.

### tree 종류별 leaf 역할

| 구조 | leaf가 의미하는 것 |
|---|---|
| 일반 rooted tree | child가 없는 끝 node |
| BST | 검색 경로가 더 내려가지 못하는 위치 |
| Heap | 배열 뒤쪽의 child 없는 원소들 |
| B-Tree | 모든 leaf가 같은 level인 균형 경계 |
| B+Tree | 실제 record pointer가 집중되고 순차 연결될 수 있는 level |

## 5. 핵심 연산

### 연산에서의 역할

- 재귀 순회의 자연스러운 종료 지점이다.
- node 높이를 edge 수로 정의하면 leaf height는 0이다.
- decision tree에서는 최종 결과, B+Tree에서는 실제 entry가 모이는 level이 될 수 있다.

단, 모든 leaf가 같은 depth인지는 tree 종류에 따라 다르다. perfect binary tree와
B-Tree 계열은 관련 불변식이 있지만 일반 tree는 그렇지 않다.

## 7. 불변식

정상 상태에서 유지해야 할 구체 조건은 위 내부 구조와 연산 설명을 기준으로 확인한다. 상위 자료구조의 불변식과 충돌하는 변경은 허용하지 않는다.

여기서 `불변식`은 삽입·조회·삭제 전후에도 계속 참이어야 하는 규칙이다. 쉽게 말해
자료구조가 망가지지 않았다고 판단하는 필수 조건이며, 연산이 끝난 뒤에도 다시 맞아야 한다.

---

## 8. 실패·예외·경계 상황

경계값과 빈 구조를 먼저 확인하고, 상위 자료구조의 불변식이 깨졌을 때 발생하는 예외와 예외 없이 잘못되는 결과를 구분한다.

`경계값`은 첫 index, 마지막 index, 0개처럼 조건이 바뀌는 가장자리 값이다. 실패는
예외가 발생해 바로 드러나는 경우와, 예외 없이 틀린 값이나 깨진 연결이 남는 경우로 나눠 본다.

## 9. 성능과 비용

Leaf: 리프 노드 자체가 독립 연산을 모두 정의하는 것은 아니다. 접근 비용은 배열·Node·Tree 등 실제 저장 표현과 상위 자료구조의 연산 계약을 기준으로 판단한다.

이 개념만 떼어 자료구조를 선택하지 않는다. 필요한 조회·삽입·삭제·정렬·범위·순회 연산을 기준으로 인접 개념과 상위 자료구조를 비교한다.

## 14. 꼬리질문 방어

### 질문 지도

| 축 | 질문 | 답변 핵심 |
|---|---|---|
| 내부 | leaf height를 0으로 두는 근거는? | 가장 먼 leaf인 자기 자신까지 edge 수가 0이기 때문이다. |
| 실패 | 모든 leaf depth가 같다고 가정하면? | 일반 tree에서는 틀리며 특정 균형 tree의 추가 불변식이다. |
| 비교 | 무방향 tree leaf 정의는? | 보통 degree 1이며 rooted tree의 child 0 정의와 문맥을 구분한다. |

### Q1. root도 leaf가 될 수 있나?

<details><summary>답 보기</summary>

노드가 root 하나뿐인 트리라면 root는 자식이 없으므로 leaf이기도 하다.

</details>

### Q2. degree 1인 node도 leaf인가?

<details><summary>답 보기</summary>

rooted tree에서 child가 하나 있으면 leaf가 아니다. 무방향 tree 문맥에서는 degree가
1인 vertex를 leaf라고 부르기도 하므로 어떤 tree 정의인지 기준을 선언한다.

</details>

### Q3. leaf 수가 많으면 tree height도 큰가?

<details><summary>답 보기</summary>

반드시 그렇지 않다. 넓고 얕은 tree는 leaf가 많아도 height가 작을 수 있다. height는
가장 긴 경로이고 leaf 수는 끝 node 개수다.

</details>

### Q4. 완전 이진 트리의 모든 leaf depth는 같은가?

<details><summary>답 보기</summary>

아니다. complete binary tree의 마지막 level은 일부만 채워질 수 있어 leaf가 마지막
level과 그 바로 위 level에 존재할 수 있다. perfect binary tree와 구분한다.

</details>

### Q5. leaf를 삭제하면 parent도 leaf가 될 수 있는가?

<details><summary>답 보기</summary>

삭제 후 parent에게 남은 child가 없다면 그렇다. 삭제 연산은 node 수뿐 아니라 leaf
집합도 바꿀 수 있다.

</details>
