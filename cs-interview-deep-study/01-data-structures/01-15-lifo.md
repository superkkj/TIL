## 1. 정의

### 정석 설명

LIFO는 Last In First Out의 약자로, 마지막에 들어온 항목이 먼저 나가는 순서 정책이다.

### 쉬운 설명

접시 더미에서 맨 위 접시를 먼저 꺼내는 방식이다. 맨 위 접시는 가장 마지막에 올린 접시다.

### 어려운 말 먼저 풀기

| 용어 | 쉬운 뜻 |
|---|---|
| LIFO | 가장 마지막에 들어온 항목을 가장 먼저 내보내는 순서 규칙이다. |
| 순서 정책 | 데이터를 어떤 기준으로 먼저 처리할지 정한 규칙이다. LIFO 자체는 저장 코드가 아니다. |
| top | LIFO 구조에서 다음에 꺼낼 수 있는 가장 위의 위치다. |
| Stack | LIFO 정책을 `push`, `pop`, `peek` 같은 연산으로 제공하는 자료구조다. |

### 다시 정리

쉬운 설명은 이해를 위한 비유다. 실제 면접 답변에서는 위 정석 설명의 용어와
전제 조건으로 돌아와 설명한다.

## 2. 배우는 이유와 실제 쓰임

| 질문 | 먼저 잡을 답 |
|---|---|
| 왜 배우나 | LIFO는 클래스 이름이 아니라 처리 순서 규칙이다. 이 규칙을 알아야 Stack·Deque 구현과 문제의 처리 순서를 분리해 판단할 수 있다. |
| 판단 근거 | Java `Stack` 공식 계약은 마지막에 push한 항목을 pop으로 먼저 꺼내는 LIFO 동작을 정의한다. |
| 실제로 어디에 쓰이나 | 함수의 중첩된 복귀 순서를 이해하고, DFS·backtracking·undo·중첩 파싱에서 최근 작업을 먼저 이어갈 때 쓴다. |
| 기억할 장면 | “마지막으로 들어온 것부터 나간다”가 전부이며, 실제 상자는 Stack이나 Deque가 될 수 있다. |

### 왜 필요한가

최근 상태를 먼저 되돌리거나, 깊이 방향으로 탐색하거나, 함수 호출처럼 마지막에 시작한 작업을 먼저 끝내야 할 때 적합하다.

가장 최근의 미완료 작업으로 돌아가야 하는 문제와 맞는다. 함수 복귀, DFS의 다음
경로, undo 기록, 파서의 중첩 구조가 예다. 반대로 도착 순서의 공정성이 필요하면
FIFO Queue가 더 적합하다.

## 3. 해결 범위와 비목표

### LIFO가 맞지 않는 경우

```text
요청 A가 먼저 도착
요청 B가 나중에 도착
새 요청을 계속 먼저 처리
```

LIFO 작업 대기열에서는 오래된 A가 계속 밀리는 starvation 가능성이 있다. 도착 순서
공정성이 중요하면 FIFO, 긴급도가 중요하면 Priority Queue처럼 요구사항에 맞는 정책을
사용한다.

---

## 4. 내부 구조와 핵심 원리

### 예

```text
push A
push B
push C
pop -> C
pop -> B
pop -> A
```

### LIFO는 자료구조 이름이 아니라 제거 순서 정책이다

Stack은 LIFO를 제공하는 대표 ADT이고, LIFO 자체는 “가장 최근에 들어온 항목이
가장 먼저 나온다”는 관찰 가능한 순서다. 내부가 배열인지 연결 리스트인지는 LIFO
정의에 포함되지 않는다.

### 중첩 문제와 LIFO

LIFO가 필요한 근본 이유는 “가장 최근에 시작했지만 아직 끝나지 않은 작업”을 먼저
끝내야 하는 문제와 잘 맞기 때문이다.

```text
함수 A 시작
  함수 B 시작
    함수 C 시작
    함수 C 종료
  함수 B 종료
함수 A 종료
```

열린 괄호도 같다.

```text
입력: ([{}])
가장 마지막에 연 괄호 {를 가장 먼저 }로 닫는다.
```

### 제한된 undo 기록 설계

실무 undo는 무한 Stack이 아니다.

```text
push(newCommand)
  1. undoStack에 기록
  2. redoStack 비우기
  3. 최대 개수를 넘으면 가장 오래된 기록 제거
```

순수 Stack API만으로 가장 오래된 바닥 원소를 효율적으로 제거하기 어렵다면 Deque나
원형 buffer 등 다른 구현을 선택할 수 있다. 정책은 LIFO여도 저장 구조 선택은 별개다.

---

## 5. 핵심 연산

### LIFO를 연산 계약으로 표현하기

LIFO는 내부 구조가 아니라 삽입과 제거 결과 사이의 계약이다.

```text
push(x1), push(x2), ..., push(xn)
중간 삭제가 없으면 pop 결과는
xn, ..., x2, x1
```

배열과 연결 리스트 모두 이 결과를 만들 수 있다. 사용자가 관찰할 수 있는 제거 순서가
같으면 둘 다 LIFO 구현이지만, 확장 비용과 메모리 구조는 다를 수 있다.

---

## 6. 상태 추적과 구현

### 상태 추적

```text
push A: [A]
push B: [A, B]
push C: [A, B, C]
pop   : C
pop   : B
```

중간의 A를 먼저 제거할 수 있게 API를 열면 순수 Stack 계약이 아니다.

---

## 7. 불변식

정상 상태에서 유지해야 할 구체 조건은 위 내부 구조와 연산 설명을 기준으로 확인한다. 상위 자료구조의 불변식과 충돌하는 변경은 허용하지 않는다.

여기서 `불변식`은 삽입·조회·삭제 전후에도 계속 참이어야 하는 규칙이다. 쉽게 말해
자료구조가 망가지지 않았다고 판단하는 필수 조건이며, 연산이 끝난 뒤에도 다시 맞아야 한다.

---

## 8. 실패·예외·경계 상황

경계값과 빈 구조를 먼저 확인하고, 상위 자료구조의 불변식이 깨졌을 때 발생하는 예외와 예외 없이 잘못되는 결과를 구분한다.

`경계값`은 첫 index, 마지막 index, 0개처럼 조건이 바뀌는 가장자리 값이다. 실패는
예외가 발생해 바로 드러나는 경우와, 예외 없이 틀린 값이나 깨진 연결이 남는 경우로 나눠 본다.

---

## 9. 성능과 비용

LIFO: Last In First Out 자체가 독립 연산을 모두 정의하는 것은 아니다. 접근 비용은 배열·Node·Tree 등 실제 저장 표현과 상위 자료구조의 연산 계약을 기준으로 판단한다.

---

## 10. 대안 비교와 선택 기준

이 개념만 떼어 자료구조를 선택하지 않는다. 필요한 조회·삽입·삭제·정렬·범위·순회 연산을 기준으로 인접 개념과 상위 자료구조를 비교한다.

## 14. 꼬리질문 방어

### 질문 지도

| 축 | 질문 | 답변 핵심 |
|---|---|---|
| 내부 | LIFO가 구현 방식을 정하나? | 아니며 관찰 가능한 제거 순서만 정한다. |
| 실패 | undo 기록을 무한히 쌓으면? | 메모리가 계속 증가하므로 크기 제한·압축·영속화 정책이 필요하다. |
| 비교 | 작업 공정성이 중요하면? | 최근 작업 우선 LIFO보다 FIFO 또는 별도 scheduler가 적합하다. |

### Q1. LIFO와 FIFO 차이는?

<details><summary>답 보기</summary>

LIFO는 마지막에 들어온 것이 먼저 나가고, FIFO는 먼저 들어온 것이 먼저 나간다. Stack은 LIFO, Queue는 FIFO가 대표적이다.

</details>

### Q2. LIFO이면 항상 O(1)인가?

<details><summary>답 보기</summary>

아니다. LIFO는 순서 정책이고 시간 복잡도는 구현에 달렸다. 동적 배열 push는 평균이
아니라 상환 O(1)이며 확장 순간에는 O(n) 복사가 발생할 수 있다.

</details>

### Q3. LIFO와 Stack은 정확히 무엇이 다른가?

<details><summary>답 보기</summary>

LIFO는 제거 순서 정책이고 Stack은 그 정책을 제공하는 ADT다. Stack은 push, pop,
peek 같은 연산 계약까지 포함하지만 LIFO라는 말 자체는 순서만 가리킨다.

</details>

### Q4. 재귀가 LIFO인 이유는?

<details><summary>답 보기</summary>

가장 최근에 호출된 함수가 먼저 끝나야 그 함수를 호출한 이전 함수 위치로 돌아갈 수
있기 때문이다. 호출 frame과 복귀 순서가 LIFO를 따른다.

</details>

### Q5. LIFO가 FIFO보다 항상 빠른가?

<details><summary>답 보기</summary>

아니다. 두 용어는 제거 순서 정책이다. 성능은 배열, 연결 구조, 동시성 제어 등 실제
구현과 연산에 따라 달라진다.

</details>

### Q6. LIFO 작업 처리의 대표 실패는?

<details><summary>답 보기</summary>

새 작업이 계속 들어오면 오래된 작업이 처리되지 않는 starvation이다. 최대 대기 시간,
aging, FIFO 전환 같은 별도 스케줄링 정책이 필요할 수 있다.

</details>
