## 1. 정의

### 정석 설명

BFS는 시작점에서 가까운 노드부터 탐색하는 그래프/트리 탐색 방식이다. 일반적으로 queue를 사용한다.

### 쉬운 설명

친구의 친구를 보러 가기 전에, 먼저 내 친구들을 전부 보는 방식이다.

### 어려운 말 먼저 풀기

| 용어 | 쉬운 뜻 |
|---|---|
| BFS | 시작점에서 edge 1개 거리, 2개 거리, 3개 거리 순서로 범위를 넓혀 가는 탐색이다. |
| graph | 여러 Node와 그 Node들을 잇는 edge로 관계를 표현한 구조다. |
| frontier | 발견은 했지만 아직 이웃을 전부 확인하지 않은 다음 처리 대상들의 경계다. Queue에 들어 있다. |
| visited | 이미 발견한 Node를 다시 넣지 않도록 기록하는 집합이다. graph의 cycle에서 반복을 막는다. |
| unweighted graph | edge마다 별도 비용이 없거나 모두 같은 비용으로 보는 graph다. |
| shortest path | 시작점에서 도착점까지 거치는 edge 수가 가장 적은 경로다. BFS의 이 보장은 unweighted 조건에서 성립한다. |

### 다시 정리

쉬운 설명은 이해를 위한 비유다. 실제 면접 답변에서는 위 정석 설명의 용어와
전제 조건으로 돌아와 설명한다.

## 2. 배우는 이유와 실제 쓰임

| 질문 | 먼저 잡을 답 |
|---|---|
| 왜 배우나 | 시작점에서 가까운 node부터 넓게 탐색하면 왜 무가중 graph의 최소 edge 수 경로를 찾을 수 있는지 이해하기 위해 배운다. |
| 판단 근거 | NIST BFS는 발견한 vertex를 Queue로 처리해 시작점과의 거리가 작은 layer부터 방문하는 graph 탐색으로 정의된다. |
| 실제로 어디에 쓰이나 | graph의 도달 가능성, 무가중 최단 거리, 최소 단계 문제, tree의 level-order 순회에 쓴다. graph에서는 중복 방문을 막는 `visited`가 필요하다. |
| 기억할 장면 | 물결이 시작점에서 한 칸 거리, 두 칸 거리 순서로 퍼져 나간다. |

## 3. 해결 범위와 비목표

### 사용하면 안 되는 경우

| graph 조건 | BFS 판단 |
|---|---|
| 모든 edge 비용 동일 | 최소 edge 수/비용 경로 가능 |
| 비용이 0 또는 1 | 0-1 BFS 같은 변형 검토 |
| 서로 다른 비음수 비용 | Dijkstra 검토 |
| 음수 edge 포함 | 조건에 맞는 별도 알고리즘 필요 |

구체 알고리즘의 적용 조건을 확인하지 않고 “최단 거리니까 BFS”라고 고르면 틀릴 수
있다.

## 4. 내부 구조와 핵심 원리

### graph에서는 visited가 필수다

Tree는 parent를 제외하면 돌아오는 cycle이 없지만 일반 graph는 cycle과 여러 경로가
있다. node를 enqueue할 때 visited로 표시해야 같은 node가 Queue에 중복으로 많이
들어가는 것을 막을 수 있다.

```java
queue.offer(start);
visited.add(start);
distance.put(start, 0);

while (!queue.isEmpty()) {
    Node current = queue.poll();
    for (Node next : current.neighbors()) {
        if (visited.add(next)) {
            distance.put(next, distance.get(current) + 1);
            parent.put(next, current);
            queue.offer(next);
        }
    }
}
```

`parent`를 역으로 따라가면 실제 최단 경로를 복원할 수 있다.

### 왜 무가중 최단 거리인가

BFS는 시작점에서 edge 수가 0, 1, 2인 node 순서로 방문한다. 모든 edge 비용이
같다고 볼 수 있을 때 처음 발견한 경로가 최소 edge 수 경로다. 서로 다른 양의
가중치가 있으면 Dijkstra 같은 알고리즘이 필요하고, 음수 edge까지 있으면 추가
조건에 맞는 다른 알고리즘을 선택해야 한다.

## 5. 핵심 연산

### 내부 동작

```text
1. 시작 노드를 queue에 넣음
2. queue에서 꺼내 방문
3. 인접 노드 중 아직 방문하지 않은 노드를 queue에 넣음
4. queue가 빌 때까지 반복
```

### 연결되지 않은 graph

한 start에서 BFS하면 그 start가 속한 connected component만 방문한다. graph 전체를
순회하려면 아직 방문하지 않은 vertex마다 BFS를 다시 시작한다.

### multi-source BFS

여러 시작점에서 가장 가까운 거리를 구하려면 모든 source를 distance 0으로 Queue에
먼저 넣는다.

```text
for source in sources:
  visited.add(source)
  distance[source] = 0
  queue.offer(source)

이후 일반 BFS
```

각 source에서 BFS를 따로 n번 하는 것보다 모든 source의 frontier를 동시에 확장한다.

## 6. 상태 추적과 구현

### graph를 직접 추적하기

```text
A -- B -- D
|    |
C -- E
```

A에서 시작하고 인접 목록 순서를 B, C라고 하자.

```text
초기: visited={A}, Queue=[A], distance[A]=0

poll A
  B 발견: visited 추가, distance=1, parent=A, offer
  C 발견: visited 추가, distance=1, parent=A, offer
Queue=[B,C]

poll B
  D 발견: distance=2, parent=B
  E 발견: distance=2, parent=B
Queue=[C,D,E]

poll C
  A와 E는 이미 visited
Queue=[D,E]
```

E는 C를 통해서도 갈 수 있지만 처음 발견한 A-B-E 경로도 edge 수 2로 최단이다.
parent를 `E -> B -> A`로 역추적한 뒤 뒤집으면 실제 경로를 얻는다.

## 7. 불변식

### BFS의 핵심 불변식

```text
Queue에서 먼저 나오는 node의 distance는 뒤 node보다 크지 않다.
node는 최초 enqueue 시 visited가 된다.
처음 기록한 distance는 최소 edge 수 거리다.
```

이 불변식은 모든 edge 비용이 동일한 무가중 모델에서 성립한다.

## 8. 실패·예외·경계 상황

경계값과 빈 구조를 먼저 확인하고, 상위 자료구조의 불변식이 깨졌을 때 발생하는 예외와 예외 없이 잘못되는 결과를 구분한다.

`경계값`은 첫 index, 마지막 index, 0개처럼 조건이 바뀌는 가장자리 값이다. 실패는
예외가 발생해 바로 드러나는 경우와, 예외 없이 틀린 값이나 깨진 연결이 남는 경우로 나눠 본다.

## 9. 성능과 비용

### 복잡도

adjacency list와 visited를 사용하면 각 vertex를 한 번 enqueue하고 각 edge를 확인해
O(V+E) 시간, O(V) 추가 공간이 든다. adjacency matrix라면 각 vertex에서 V개 칸을
확인해 O(V^2)가 될 수 있다. 표현 방식 전제를 붙여야 한다.

## 14. 꼬리질문 방어

### 질문 지도

| 축 | 질문 | 답변 핵심 |
|---|---|---|
| 내부 | 최단 경로 자체는 어떻게 복원하나? | 첫 발견 때 parent를 저장하고 target에서 역추적한다. |
| 실패 | 가중 graph에 그대로 쓰면? | edge 수 최소와 비용 최소가 달라 잘못된 답이 될 수 있다. |
| 비교 | DFS와 메모리 차이는? | BFS는 frontier 폭, DFS는 탐색 depth에 주로 비례한다. |

### Q1. BFS는 왜 queue를 쓰나?

<details><summary>답 보기</summary>

먼저 발견한 가까운 노드를 먼저 처리해야 하므로 FIFO 구조인 queue가 맞다.

</details>

### Q2. visited를 dequeue할 때 표시하면 왜 비효율적일 수 있나?

<details><summary>답 보기</summary>

한 node가 dequeue되기 전에 여러 이웃이 같은 node를 각각 enqueue할 수 있다. enqueue
시점에 표시하면 Queue에 처음 한 번만 들어가고, 최초 발견 거리도 고정된다.

</details>

### Q3. target을 발견하자마자 종료해도 되는가?

<details><summary>답 보기</summary>

무가중 BFS에서 enqueue하며 처음 발견한 시점에 최소 거리 값은 정해진다. 필요한 결과가
거리/경로 하나뿐이면 종료할 수 있지만, 같은 거리의 모든 해나 전체 탐색이 목적이면
계속해야 한다.

</details>

### Q4. BFS 결과 순서는 항상 유일한가?

<details><summary>답 보기</summary>

아니다. 같은 depth 안의 순서는 adjacency iteration 순서에 따라 달라질 수 있다. 최단
거리 값은 같아도 선택된 parent와 구체 경로가 달라질 수 있다.

</details>

### Q5. adjacency list에서 O(V+E)인 이유는?

<details><summary>답 보기</summary>

각 vertex를 한 번 처리하고 각 adjacency entry를 확인한다. 무방향 graph에서는 한
edge가 두 목록에 나타나도 상수 배이므로 O(V+E)다.

</details>

### Q6. BFS Queue가 가장 커지는 시점은?

<details><summary>답 보기</summary>

frontier가 가장 넓은 level을 처리할 때다. 시간 복잡도와 별개로 graph의 폭에 따라
O(V) 추가 공간이 필요할 수 있다.

</details>
