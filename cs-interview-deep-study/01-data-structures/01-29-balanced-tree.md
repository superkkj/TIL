## 1. 정의

### 정석 설명

Balanced Tree는 트리 높이가 과도하게 커지지 않도록 균형 조건을 유지하는 트리 계열이다.

### 쉬운 설명

트리가 한쪽으로 기울면 탐색이 느려진다. 균형 트리는 중간중간 자세를 바로잡아 높이를 낮게 유지한다.

### 어려운 말 먼저 풀기

| 용어 | 쉬운 뜻 |
|---|---|
| balanced | 한쪽 가지가 지나치게 길어지지 않도록 높이를 제한한 상태다. 모든 좌우 높이가 완전히 같다는 뜻은 아니다. |
| height bound | Node가 `n`개일 때 트리 높이가 어느 정도를 넘지 않도록 정한 상한이다. |
| rotation | key의 정렬 순서는 유지하면서 부모·자식 연결을 바꿔 기울기를 줄이는 연산이다. |
| rebalance | 삽입이나 삭제로 균형 규칙이 깨진 뒤 회전·색 변경 등으로 다시 맞추는 작업이다. |
| self-balancing tree | 삽입·삭제 과정에 균형 복구 규칙이 포함된 트리다. AVL과 Red-Black Tree가 예다. |

### 다시 정리

쉬운 설명은 이해를 위한 비유다. 실제 면접 답변에서는 위 정석 설명의 용어와
전제 조건으로 돌아와 설명한다.

---

## 2. 배우는 이유와 실제 쓰임

| 질문 | 먼저 잡을 답 |
|---|---|
| 왜 배우나 | BST가 한쪽으로 늘어져 선형 탐색처럼 되는 문제를 막고, height를 제한하는 것이 왜 O(log n) 탐색으로 이어지는지 이해하기 위해 배운다. |
| 판단 근거 | NIST는 height-balanced tree를 subtree 높이 차이를 제한하는 tree로 정의하고, Java `TreeMap`은 red-black tree로서 기본 연산의 O(log n)을 보장한다. |
| 실제로 어디에 쓰이나 | 정렬된 key 조회·범위 탐색·삽입·삭제를 함께 제공하는 ordered map·set과 인메모리 검색 구조를 설계할 때 쓰인다. |
| 기억할 장면 | 가지가 한쪽으로만 길어지지 않게 돌려서 tree를 낮게 유지한다. |

### 왜 필요한가

BST는 균형이면 O(log n)에 탐색할 수 있지만, 한쪽으로 치우치면 O(n)이 된다. 균형 트리는 이 최악 상황을 줄이기 위해 필요하다.

---

## 3. 해결 범위와 비목표

이 문서가 직접 설명하는 범위는 Balanced Tree: 균형 트리의 정의와 상위 자료구조에서의 역할이다. 정렬·동시성·영속성·특정 시간 복잡도는 별도 불변식이나 구현 계약이 있을 때만 보장된다.

쉽게 말하면 이 개념의 이름만 보고 다음 성질까지 자동으로 있다고 단정하면 안 된다.
`동시성`은 여러 실행 흐름이 함께 접근해도 안전한지, `영속성`은 프로그램이 끝난 뒤에도
파일이나 DB 등에 데이터가 남는지, `구현 계약`은 실제 클래스나 API가 무엇을 약속하는지를
뜻한다. 이 성질들은 사용하는 구현체의 공식 설명에서 각각 확인해야 한다.

## 4. 내부 구조와 핵심 원리

### 대표 예

| 구조 | 특징 |
|---|---|
| AVL Tree | 균형 조건이 엄격 |
| Red-Black Tree | 균형 조건이 상대적으로 느슨, 실무 라이브러리에서 자주 등장 |
| B-Tree/B+Tree | multiway tree, DB 인덱스와 연결 |

### Balanced Tree는 하나의 고정 자료구조가 아니다

균형의 목적은 기본 연산 경로인 height를 n에 대해 O(log n)으로 제한하는 것이다.
어떤 상태를 허용하고 어떻게 복구하는지는 구조마다 다르다.

| 구조 | 균형 기준 | 복구 단위 |
|---|---|---|
| AVL | 모든 node의 좌우 height 차이 최대 1 | 회전 |
| Red-Black | 색과 black-height 불변식 | recolor와 회전 |
| B-Tree | node key/child 수 범위와 같은 leaf level | split, borrow, merge |

---

## 5. 핵심 연산

### 회전의 핵심

rotation은 BST의 inorder 순서를 보존하면서 parent-child 모양과 height를 바꾼다.
정렬 결과를 유지해야 하므로 node 값을 임의로 섞는 것이 아니다.

```text
    y              x
   /              / \
  x      ->       A   y
 / \                / \
A   B              B   C
```

### 회전은 왜 정렬 순서를 보존하는가?

right rotation 전 subtree의 inorder는 `A, x, B, y, C`다.

```text
        y                 x
       / \               / \
      x   C    ->        A   y
     / \                   / \
    A   B                 B   C
```

회전 후 inorder도 `A, x, B, y, C`다. 값의 상대 정렬 순서는 그대로 두고 height만
바꾼다. 링크를 잘못 연결해 B를 잃으면 정렬뿐 아니라 데이터도 유실된다.

### 복구가 필요한 시점

```text
BST 삽입/삭제
  -> subtree height 또는 색/점유 상태 변화
  -> 조상 방향으로 메타데이터 갱신
  -> 불변식 위반 발견
  -> rotation/recolor/split/merge
  -> 새 subtree root를 parent에 재연결
```

회전 함수가 내부 링크만 바꾸고 호출자가 새 subtree root를 받지 않으면 root 연결이
끊길 수 있다.

## 7. 불변식

정상 상태에서 유지해야 할 구체 조건은 위 내부 구조와 연산 설명을 기준으로 확인한다. 상위 자료구조의 불변식과 충돌하는 변경은 허용하지 않는다.

여기서 `불변식`은 삽입·조회·삭제 전후에도 계속 참이어야 하는 규칙이다. 쉽게 말해
자료구조가 망가지지 않았다고 판단하는 필수 조건이며, 연산이 끝난 뒤에도 다시 맞아야 한다.

---

## 8. 실패·예외·경계 상황

경계값과 빈 구조를 먼저 확인하고, 상위 자료구조의 불변식이 깨졌을 때 발생하는 예외와 예외 없이 잘못되는 결과를 구분한다.

`경계값`은 첫 index, 마지막 index, 0개처럼 조건이 바뀌는 가장자리 값이다. 실패는
예외가 발생해 바로 드러나는 경우와, 예외 없이 틀린 값이나 깨진 연결이 남는 경우로 나눠 본다.

---

## 9. 성능과 비용

### 비용의 대가

탐색 경로를 제한하는 대신 삽입·삭제 때 균형 정보 저장과 복구 작업이 필요하다.
읽기와 쓰기의 비율, 메모리/페이지 접근 방식에 따라 적합한 균형 규칙이 달라진다.

### 구조별 보장과 비용

| 구조 | height 목표 | 메타데이터 | 갱신 복구 |
|---|---|---|---|
| AVL | 엄격한 높이 균형 | height/BF | 단일·이중 회전 |
| Red-Black | 색 규칙으로 로그 height | color | recolor·회전 |
| B-Tree | leaf level 동일, node 점유 | key/child 수 | split·borrow·merge |

모두 “균형”이라는 목적은 같지만 불변식이 다르므로 복구 알고리즘을 섞어 말하면 안
된다.

## 14. 꼬리질문 방어

### 질문 지도

| 축 | 질문 | 답변 핵심 |
|---|---|---|
| 내부 | rotation이 key 순서를 바꾸나? | inorder 순서는 유지하고 parent-child 모양만 바꾼다. |
| 실패 | 균형 정보만 고치고 링크를 안 바꾸면? | 표시와 실제 height가 달라져 성능 보장이 깨진다. |
| 비교 | AVL과 Red-Black 선택 축은? | 더 엄격한 조회 height와 갱신 복구 비용의 절충이다. |

### Q1. 균형 트리는 항상 완벽히 균형인가?

<details><summary>답 보기</summary>

아니다. 구조마다 허용하는 균형 조건이 다르다. Red-Black Tree는 완전 균형보다 높이가 과도하게 커지지 않도록 제한하는 쪽에 가깝다.

</details>

### Q2. 완전 이진 트리여야 balanced tree인가?

<details><summary>답 보기</summary>

아니다. complete는 level을 왼쪽부터 채우는 모양 조건이고 balanced는 height를 제한하는
조건이다. Red-Black Tree는 complete가 아니어도 로그 height를 보장할 수 있다.

</details>

### Q3. 균형을 유지하면 모든 연산이 O(log n)인가?

<details><summary>답 보기</summary>

root-to-leaf 경로를 따르는 탐색·삽입·삭제 위치 찾기는 로그 height와 연결된다. 전체
순회나 value 조건 검색처럼 모든 node를 봐야 하는 연산은 O(n)이다.

</details>

### Q4. rotation 자체 비용은?

<details><summary>답 보기</summary>

고정된 수의 node 참조와 메타데이터를 바꾸는 한 번의 rotation은 O(1)이다. 어떤
조상에서 몇 번 수행하는지와 위치를 찾는 O(log n) 경로 비용은 별도다.

</details>

### Q5. 저장된 height나 color가 틀리면 바로 예외가 나는가?

<details><summary>답 보기</summary>

보통은 아니다. 잘못된 복구 판단으로 tree가 점차 기울거나 검색 경로가 깨지는 조용한
오동작이 될 수 있어 불변식 검증 테스트가 중요하다.

</details>
