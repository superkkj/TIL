## 1. 정의

### 정석 설명

Big-O는 입력 크기 `n`이 커질 때 알고리즘 비용이 어떤 성장률로 증가하는지 상한 관점에서 표현하는 표기다.

### 쉬운 설명

정확히 몇 초 걸리는지가 아니라, 데이터가 10배 커질 때 일이 얼마나 늘어나는지를 보는 방식이다.

### 어려운 말 먼저 풀기

| 용어 | 쉬운 뜻 |
|---|---|
| 입력 크기 `n` | 성능을 분석할 때 데이터가 얼마나 큰지를 나타내는 기준 값이다. 배열이면 원소 수일 수 있다. |
| 증가율 | `n`이 커질 때 연산 횟수가 얼마나 빠르게 늘어나는지를 말한다. |
| 점근적 | 작은 입력의 차이보다 `n`이 충분히 커졌을 때의 변화 모양을 본다는 뜻이다. |
| 상한 | 실제 증가가 그 기준보다 더 빠르게 커지지 않는다고 설명하는 한계다. Big-O는 점근적 상한 표기다. |
| 지배항 | `n`이 커질수록 전체 식에 가장 큰 영향을 주는 항이다. `3n² + 100n + 5`에서는 `n²`이다. |
| 상수 제거 | 컴퓨터나 구현에 따라 달라지는 고정 배수보다 증가 모양에 집중하는 것이다. O(1)이 실제 명령 한 개라는 뜻은 아니다. |

### 다시 정리

쉬운 설명은 이해를 위한 비유다. 실제 면접 답변에서는 위 정석 설명의 용어와
전제 조건으로 돌아와 설명한다.

## 2. 배우는 이유와 실제 쓰임

| 질문 | 먼저 잡을 답 |
|---|---|
| 왜 배우나 | 특정 컴퓨터에서 잰 시간보다 입력이 커질 때 일의 양이 얼마나 빨리 증가하는지를 비교해 확장 가능성을 판단하기 위해 배운다. |
| 판단 근거 | NIST는 Big-O를 입력 크기가 커질 때 함수 성장의 점근적 상한을 나타내는 표기로 정의한다. 상수와 낮은 차수보다 지배적인 성장률을 본다. |
| 실제로 어디에 쓰이나 | 알고리즘 비교, 자료구조 선택, 코드 리뷰, 병목 후보 선별에서 입력 크기와 핵심 연산의 증가 관계를 설명할 때 쓴다. 실제 시간 측정을 대체하지는 않는다. |
| 기억할 장면 | 데이터가 두 배가 됐을 때 일이 그대로인지, 두 배인지, 몇 단계만 늘어나는지를 묻는다. |

### 왜 필요한가

작은 입력에서는 차이가 안 보이지만, 데이터가 커질수록 성장률 차이가 성능 차이를 만든다.

## 3. 해결 범위와 비목표

이 문서가 직접 설명하는 범위는 Big-O: 시간 복잡도 표기의 정의와 상위 자료구조에서의 역할이다. 정렬·동시성·영속성·특정 시간 복잡도는 별도 불변식이나 구현 계약이 있을 때만 보장된다.

쉽게 말하면 이 개념의 이름만 보고 다음 성질까지 자동으로 있다고 단정하면 안 된다.
`동시성`은 여러 실행 흐름이 함께 접근해도 안전한지, `영속성`은 프로그램이 끝난 뒤에도
파일이나 DB 등에 데이터가 남는지, `구현 계약`은 실제 클래스나 API가 무엇을 약속하는지를
뜻한다. 이 성질들은 사용하는 구현체의 공식 설명에서 각각 확인해야 한다.

## 4. 내부 구조와 핵심 원리

### 예

| 표기 | 감각 |
|---|---|
| O(1) | 입력 크기와 거의 무관 |
| O(log n) | 매번 후보를 줄임 |
| O(n) | 전체를 훑음 |
| O(n log n) | 정렬에서 자주 등장 |
| O(n^2) | 모든 쌍 비교 |

### 수학적 정의

`f(n) = O(g(n))`은 충분히 큰 n에서 `f(n)`이 `c * g(n)`보다 커지지 않게 하는 양의
상수 `c`와 시작점 `n0`가 존재한다는 뜻이다.

```text
0 <= f(n) <= c * g(n), for all n >= n0
```

따라서 Big-O는 점근적 상한이다. 정확한 증가율을 위아래로 함께 제한하려면
Big-Theta, 하한은 Big-Omega를 사용한다. 실무 면접에서는 관례적으로 “복잡도”를
Big-O로 말하더라도 이 차이를 알고 있어야 한다.

### 입력 모델을 먼저 정한다

`n`이 배열 길이인지, vertex 수인지, 저장 entry 수인지 선언한다. graph는 `V`와
`E`, 문자열 알고리즘은 문자열 길이처럼 변수가 여러 개일 수 있다. 변수를 억지로
하나로 합치면 비용 원인을 숨긴다.

### 최악·평균·기대·상환을 구분한다

- worst-case: 같은 크기 입력 중 가장 비싼 경우
- average-case: 정의한 입력 분포에서의 평균
- expected: 확률적 선택이나 hash 분산 가정 아래 기댓값
- amortized: 긴 연산 열 전체 비용을 연산 수로 나눈 보장

HashMap 기대 O(1)과 동적 배열 append 상환 O(1)은 같은 “평균”이 아니다.

## 5. 핵심 연산

### 순차 연산은 더하고 중첩 연산은 곱한다

```text
O(n) 작업 후 O(n log n) 작업
  O(n + n log n) = O(n log n)

O(n) 반복 안에서 매번 O(log n) 작업
  O(n * log n) = O(n log n)
```

입력 변수가 다르면 함부로 하나로 합치지 않는다.

```text
두 독립 배열 순회: O(n + m)
두 배열의 모든 쌍: O(n * m)
```

## 6. 상태 추적과 구현

### 코드에서 연산 횟수를 세는 순서

```java
int sum = 0;                    // 1
for (int i = 0; i < n; i++) { // n에 비례
    sum += values[i];           // n번
}
```

정확한 기본 연산 수를 단순화하면 `an + b` 형태다. 충분히 큰 n에서 n 항이 지배하므로
O(n)이다.

```java
for (int i = 0; i < n; i++) {
    for (int j = 0; j < n; j++) {
        use(i, j);
    }
}
```

내부 연산은 n * n번이므로 O(n^2)다. 단순히 loop 개수만 세지 말고 각 loop 범위와
내부 연산 비용을 곱하거나 합한다.

## 7. 불변식

정상 상태에서 유지해야 할 구체 조건은 위 내부 구조와 연산 설명을 기준으로 확인한다. 상위 자료구조의 불변식과 충돌하는 변경은 허용하지 않는다.

여기서 `불변식`은 삽입·조회·삭제 전후에도 계속 참이어야 하는 규칙이다. 쉽게 말해
자료구조가 망가지지 않았다고 판단하는 필수 조건이며, 연산이 끝난 뒤에도 다시 맞아야 한다.

---

## 8. 실패·예외·경계 상황

### 항과 상수를 버리는 이유와 한계

`3n^2 + 100n + 5`는 충분히 큰 n에서 `n^2` 항이 지배해 O(n^2)로 표현한다. 그러나
작은 n, 메모리 접근, cache, 네트워크 I/O에서는 상수와 환경이 실제 시간을 좌우할
수 있다. Big-O 분석 뒤에 benchmark와 profiler가 필요한 이유다.

---

## 9. 성능과 비용

### 공간 복잡도도 같은 방식으로 본다

입력 자체 공간과 알고리즘이 추가로 사용하는 auxiliary space를 구분한다.

```text
in-place 배열 swap: 추가 공간 O(1)
길이 n 결과 배열:   추가 공간 O(n)
재귀 DFS:            호출 stack O(height)
```

시간을 줄이기 위해 hash index O(n) 공간을 추가하는 것처럼 시간-공간 trade-off가 있다.

### 상환 분석 예: 동적 배열 append

capacity가 찰 때 2배 배열로 복사한다고 하자.

```text
복사 크기: 1 + 2 + 4 + 8 + ... < 2n
일반 저장: n번
전체 작업: O(n)
n번 append당 상환 O(1)
```

특정 확장 append 한 번은 O(n)이다. 평균 입력 분포가 아니라 연산 열 전체 비용을
나누는 상환 분석이다.

## 14. 꼬리질문 방어

### 질문 지도

| 축 | 질문 | 답변 핵심 |
|---|---|---|
| 내부 | Big-O와 Theta 차이는? | O는 상한, Theta는 상·하한이 일치하는 타이트한 성장률이다. |
| 실패 | n의 의미를 안 정하면? | 문자열 길이·entry 수 같은 다른 비용 변수를 숨길 수 있다. |
| 실무 | Big-O 뒤에 왜 측정하나? | 상수, cache, I/O, 입력 분포와 런타임 구현 영향을 확인하기 위해서다. |

### Q1. Big-O가 실제 성능을 완전히 설명하나?

<details><summary>답 보기</summary>

아니다. 상수항, 캐시, 메모리, 구현체, 데이터 분포도 실제 성능에 영향을 준다. Big-O는 성장률을 보는 추상화다.

</details>

### Q2. O(n) 알고리즘은 동시에 O(n^2)이기도 한가?

<details><summary>답 보기</summary>

점근 상한 정의상 맞다. 하지만 가장 타이트한 유용한 상한을 말하는 것이 보통이다.
정확한 성장률을 강조하려면 Theta(n)이라고 표현한다.

</details>

### Q3. 최악 O(n)이고 평균 O(1)이라고 동시에 말할 수 있는가?

<details><summary>답 보기</summary>

가능하다. 서로 다른 비용 기준이다. 평균을 말하려면 입력 분포, 기대값을 말하려면
확률 가정 등 성립 조건을 함께 제시해야 한다.

</details>

### Q4. `n/2`번 반복하면 O(n/2)인가?

<details><summary>답 보기</summary>

Big-O에서는 상수 배를 제거해 O(n)이다. 정확한 실행 시간 비교에서는 1/2 상수도
의미가 있을 수 있다.

</details>

### Q5. 재귀식 `T(n)=T(n/2)+O(1)`은 왜 O(log n)인가?

<details><summary>답 보기</summary>

n을 절반으로 줄여 1이 될 때까지 약 log2(n)번 호출하고 각 level 작업이 O(1)이기
때문이다.

</details>

### Q6. Big-O가 같으면 두 구현 성능도 같은가?

<details><summary>답 보기</summary>

아니다. 상수, 메모리 접근, cache, JIT, I/O, 데이터 분포가 다를 수 있다. Big-O로
확장성을 비교한 뒤 benchmark와 profiler로 실제 병목을 확인한다.

</details>
