## 1. 정의

### 정석 설명

일반 graph 이론에서 tree는 모든 vertex가 연결되어 있고 cycle이 없는 graph다. root를
하나 정해 방향과 계층을 부여하면 rooted tree가 되며 parent-child 관계로 설명한다.

### 쉬운 설명

회사 조직도나 폴더 구조처럼 위아래 관계가 있는 데이터 구조다.

### 어려운 말 먼저 풀기

| 용어 | 쉬운 뜻 |
|---|---|
| Node | 값과 다른 Node로 이어지는 연결을 담은 트리의 한 지점이다. |
| graph | 여러 지점(vertex)과 그 지점을 잇는 연결(edge)로 관계를 표현한 구조다. |
| vertex | graph를 이루는 한 지점이다. 트리에서는 보통 Node라고 부른다. |
| edge | 부모와 자식 Node 사이의 연결선 하나다. |
| subtree | 어떤 Node와 그 아래 모든 후손을 하나의 작은 트리로 본 것이다. |
| cycle | 연결을 따라가다 출발한 Node로 다시 돌아오는 고리다. 일반적인 rooted tree에는 없어야 한다. |
| rooted tree | 여러 Node 중 하나를 root로 정해 위아래 parent-child 방향을 부여한 트리다. |
| 경로(path) | 한 Node에서 다른 Node까지 edge를 따라가는 연결 순서다. |
| 불변식 | 트리 연산 전후에도 반드시 유지해야 하는 규칙이다. “root를 제외한 각 Node는 부모가 하나”가 예다. |

### 다시 정리

쉬운 설명은 이해를 위한 비유다. 실제 면접 답변에서는 위 정석 설명의 용어와
전제 조건으로 돌아와 설명한다.

## 3. 해결 범위와 비목표

이 문서가 직접 설명하는 범위는 Tree: 트리의 정의와 상위 자료구조에서의 역할이다. 정렬·동시성·영속성·특정 시간 복잡도는 별도 불변식이나 구현 계약이 있을 때만 보장된다.

쉽게 말하면 이 개념의 이름만 보고 다음 성질까지 자동으로 있다고 단정하면 안 된다.
`동시성`은 여러 실행 흐름이 함께 접근해도 안전한지, `영속성`은 프로그램이 끝난 뒤에도
파일이나 DB 등에 데이터가 남는지, `구현 계약`은 실제 클래스나 API가 무엇을 약속하는지를
뜻한다. 이 성질들은 사용하는 구현체의 공식 설명에서 각각 확인해야 한다.

## 4. 내부 구조와 핵심 원리

### 내부 구성 요소

| 용어 | 의미 |
|---|---|
| root | 시작 노드 |
| parent | 위 노드 |
| child | 아래 노드 |
| leaf | 자식 없는 끝 노드 |
| edge | 노드 사이 연결 |

### rooted tree로 바꾸면 생기는 관계

root를 정하면 각 edge의 위·아래 방향이 생긴다. root를 제외한 node는 root로 가는
유일한 경로에서 바로 앞 node 하나를 parent로 갖는다. 한 node와 모든 descendant로
이루어진 부분은 subtree다. parent-child 관계는 저장 방식이 아니라 논리 관계이며,
구현은 child 목록만 또는 parent 포인터까지 함께 저장할 수 있다.

### 트리는 무엇을 빠르게 해 주는가?

“tree라서 검색이 O(log n)”은 틀린 일반화다. Tree는 계층과 유일 경로를 표현하는
구조이고, 검색 성능은 추가 규칙에 달려 있다.

| tree 종류 | 추가 불변식 | 강해지는 연산 |
|---|---|---|
| 일반 rooted tree | parent-child 계층 | subtree 순회, 계층 표현 |
| Binary Search Tree | 왼쪽 < node < 오른쪽 | key 비교 탐색 |
| 균형 BST | BST + height 제한 | 탐색·삽입·삭제 O(log n) 유지 |
| Heap | complete shape + parent 우선순위 | 최소/최대 head 접근 |
| B-Tree | 다분기 + 점유·균형 규칙 | page I/O를 줄이는 탐색 |

### 저장 표현 세 가지

#### child 목록

```java
class Node<T> {
    T value;
    List<Node<T>> children;
}
```

아래 방향 순회가 직접적이다. 위로 이동하려면 호출 Stack이나 별도 parent가 필요하다.

#### parent 배열

```text
node:    A  B  C  D
parent: -1  A  A  B
```

각 node의 parent를 빠르게 알 수 있지만 child를 찾으려면 역 index가 없을 때 전체를
확인해야 한다.

#### 인접 리스트 graph

```text
A: B, C
B: A, D
C: A
D: B
```

무방향 edge를 양쪽에 저장하므로 DFS에서 직전 parent 또는 visited를 기억해야 되돌아간
edge를 다시 타지 않는다.

### tree 여부를 검증하는 기준

유한 무방향 graph가 tree인지 확인하는 대표 기준은 다음과 같다.

```text
모든 node가 연결되어 있고 cycle이 없다.
```

DFS/BFS 중 이미 방문한 node를 parent가 아닌 edge로 다시 만나면 cycle이 있다. 순회가
끝난 후 방문 수가 전체 node 수보다 작으면 연결되지 않았다.

## 5. 핵심 연산

Tree: 트리 자체가 독립 ADT 연산을 정의하지 않는 경우에는 상위 자료구조의 삽입·조회·삭제 과정에서 이 개념이 사용되는 지점을 기준으로 설명한다.

`ADT`는 내부 코드를 정하는 말이 아니라, 사용할 수 있는 연산과 그 연산이 지켜야 할
규칙을 정한 사용 계약이다. 이 용어 자체에 독립 연산이 없다면 실제 자료구조의 어느
동작에 참여하는지를 따라가며 이해한다.

---

## 6. 상태 추적과 구현

### 최소 구조

```java
class Node<T> {
    T value;
    List<Node<T>> children = new ArrayList<>();
}
```

그래프 입력으로 tree를 순회한다면 parent 포인터가 없어도 `visited` 또는 이전 node를
기억해 되돌아가는 edge를 다시 방문하지 않아야 한다.

### 순회 상태 추적

```text
        A
      /   \
     B     C
    / \
   D   E
```

preorder DFS라면:

```text
visit A
  visit B
    visit D
    visit E
  visit C

결과: A, B, D, E, C
```

모든 node를 정확히 한 번 방문한다는 불변식을 지키려면 graph 표현에서는 parent나
visited가 필요하다.

## 7. 불변식

### 정의에서 빠지면 안 되는 두 조건

- connected: 어떤 두 node 사이에도 경로가 있다.
- acyclic: 자기 자신으로 돌아오는 cycle이 없다.

node가 `n`개인 유한 tree의 edge는 `n-1`개이고, 임의의 두 node 사이에는 단순 경로가
정확히 하나다. edge 하나를 끊으면 두 component로 나뉘고, cycle을 만들지 않는 새
edge 하나로 다시 연결할 수 있다.

---

## 8. 실패·예외·경계 상황

### 실패 상황

| 실패 | 증상 |
|---|---|
| 조상을 child로 다시 연결 | cycle, 재귀 순회 무한 반복 가능 |
| 한 node를 두 parent에 연결 | root 경로 유일성 붕괴, DAG 형태 |
| root 참조 유실 | 외부 참조가 없다면 전체 tree에 접근 불가 |
| parent/children 양방향 불일치 | 위·아래 순회 결과가 달라짐 |
| 균형 규칙 없이 빠른 탐색 기대 | 치우친 높이에서 O(n) 가능 |

---

## 9. 성능과 비용

### 비용

모든 node를 한 번 방문하는 순회는 O(n), 재귀 DFS의 추가 stack은 최악 O(height),
level-order의 Queue는 한 level의 최대 너비에 비례한다. 탐색이 빠른지는 tree라는
사실만으로 결정되지 않고 정렬·균형 같은 추가 불변식에 달려 있다.

## 14. 꼬리질문 방어

### 질문 지도

| 축 | 질문 | 답변 핵심 |
|---|---|---|
| 내부 | node n개의 edge가 왜 n-1인가? | 연결을 유지하는 최소 edge 수이며 하나 더 있으면 cycle이 생긴다. |
| 실패 | visited 없이 graph를 tree처럼 돌면? | cycle에서 무한 반복하거나 node를 중복 방문할 수 있다. |
| 비교 | tree와 DAG 차이는? | tree는 root 경로와 parent가 유일하지만 DAG는 여러 parent를 허용한다. |

### Q1. 트리와 그래프 차이는?

<details><summary>답 보기</summary>

트리는 계층 구조를 가진 특수한 형태로 볼 수 있고, 일반적으로 root와 부모-자식 관계를 중심으로 설명한다. 그래프는 더 일반적인 노드-간선 구조다.

</details>

### Q2. DAG는 cycle이 없으니 tree인가?

<details><summary>답 보기</summary>

아니다. DAG의 한 node는 여러 parent 경로를 가질 수 있고 임의 두 node 사이 경로가
유일하지 않을 수 있다. tree는 연결성과 유일 경로 조건까지 만족해야 한다.

</details>

### Q3. node n개인 tree의 edge가 n-1보다 적으면?

<details><summary>답 보기</summary>

모든 node를 연결할 수 없다. n개의 독립 node를 하나의 connected component로 만들려면
최소 n-1개의 edge가 필요하다.

</details>

### Q4. edge가 n-1개면 무조건 tree인가?

<details><summary>답 보기</summary>

연결 조건도 확인해야 한다. 전체 graph가 connected이고 edge가 n-1개라면 tree로 볼
수 있다. 연결되지 않은 graph는 한 component에 cycle이 있고 다른 component가 분리된
형태도 가능하다.

</details>

### Q5. subtree도 tree인가?

<details><summary>답 보기</summary>

rooted tree에서 어떤 node와 모든 descendant를 모으면 그 node를 root로 하는 tree다.
원래 tree의 parent 방향 edge는 subtree 범위에서 제외된다.

</details>

### Q6. Tree 순회에 visited가 항상 필요한가?

<details><summary>답 보기</summary>

child 방향만 저장된 올바른 rooted tree라면 cycle이 없어 필수는 아니다. 무방향 인접
리스트로 저장했거나 입력이 정말 tree인지 신뢰할 수 없다면 parent/visited가 필요하다.

</details>
