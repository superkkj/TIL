## 1. 정의

### 정석 설명

O(n)은 입력 크기 `n`에 비례해 비용이 증가하는 성장률이다.

### 쉬운 설명

처음부터 끝까지 하나씩 확인하는 방식이다. 데이터가 10배 늘면 할 일도 대략 10배 늘어난다.

### 어려운 말 먼저 풀기

| 용어 | 쉬운 뜻 |
|---|---|
| linear time | 입력이 `n`개 늘어날 때 확인하거나 처리할 일도 `n`에 비례해 늘어나는 형태다. |
| full scan | 처음부터 끝까지 모든 원소를 한 번씩 확인하는 작업이다. |
| early exit | 원하는 값을 중간에 찾으면 순회를 일찍 끝내는 것이다. 일부 입력은 빨라도 최악에는 끝까지 볼 수 있다. |
| coefficient | `3n`의 3처럼 `n` 앞에 붙는 고정 배수다. Big-O에서는 증가 차수가 같으면 제거한다. |
| nested loop | 반복문 안에 반복문이 있는 구조다. 항상 O(n²)은 아니며 각 반복 횟수가 입력과 어떻게 연결되는지 계산해야 한다. |

### 다시 정리

쉬운 설명은 이해를 위한 비유다. 실제 면접 답변에서는 위 정석 설명의 용어와
전제 조건으로 돌아와 설명한다.

## 2. 탄생 배경과 필요성

### 왜 중요한가

O(n)은 작은 데이터에서는 괜찮지만, 대량 데이터에서 반복 호출되면 병목이 될 수 있다.

---

## 3. 해결 범위와 비목표

이 문서가 직접 설명하는 범위는 O(n): 선형 시간의 정의와 상위 자료구조에서의 역할이다. 정렬·동시성·영속성·특정 시간 복잡도는 별도 불변식이나 구현 계약이 있을 때만 보장된다.

쉽게 말하면 이 개념의 이름만 보고 다음 성질까지 자동으로 있다고 단정하면 안 된다.
`동시성`은 여러 실행 흐름이 함께 접근해도 안전한지, `영속성`은 프로그램이 끝난 뒤에도
파일이나 DB 등에 데이터가 남는지, `구현 계약`은 실제 클래스나 API가 무엇을 약속하는지를
뜻한다. 이 성질들은 사용하는 구현체의 공식 설명에서 각각 확인해야 한다.

---

## 4. 내부 구조와 핵심 원리

### 예

```text
배열 전체 순회
리스트에서 특정 값 찾기
HashMap에서 value 조건 검색
트리의 모든 노드 방문
```

### 선형이 생기는 두 형태

1. 결과를 만들기 위해 모든 입력을 반드시 한 번 보는 경우
2. 더 좋은 탐색 기준이 없어 target을 찾을 때까지 순차 비교하는 경우

첫 번째는 합계·최댓값처럼 입력 전체가 결과에 영향을 주므로 O(n)이 최적일 수 있다.
두 번째는 정렬, hash, index 같은 추가 구조로 개선 가능할 수 있지만 전처리·메모리·
갱신 비용을 함께 지불한다.

---

## 5. 핵심 연산

O(n): 선형 시간 자체가 독립 ADT 연산을 정의하지 않는 경우에는 상위 자료구조의 삽입·조회·삭제 과정에서 이 개념이 사용되는 지점을 기준으로 설명한다.

`ADT`는 내부 코드를 정하는 말이 아니라, 사용할 수 있는 연산과 그 연산이 지켜야 할
규칙을 정한 사용 계약이다. 이 용어 자체에 독립 연산이 없다면 실제 자료구조의 어느
동작에 참여하는지를 따라가며 이해한다.

---

## 6. 상태 추적과 구현

위 핵심 연산의 예시를 입력부터 결과까지 따라가며 값·index·Node·Stack·Queue 중 실제로 변하는 상태를 확인한다.

---

## 7. 불변식

정상 상태에서 유지해야 할 구체 조건은 위 내부 구조와 연산 설명을 기준으로 확인한다. 상위 자료구조의 불변식과 충돌하는 변경은 허용하지 않는다.

여기서 `불변식`은 삽입·조회·삭제 전후에도 계속 참이어야 하는 규칙이다. 쉽게 말해
자료구조가 망가지지 않았다고 판단하는 필수 조건이며, 연산이 끝난 뒤에도 다시 맞아야 한다.

---

## 8. 실패·예외·경계 상황

경계값과 빈 구조를 먼저 확인하고, 상위 자료구조의 불변식이 깨졌을 때 발생하는 예외와 예외 없이 잘못되는 결과를 구분한다.

`경계값`은 첫 index, 마지막 index, 0개처럼 조건이 바뀌는 가장자리 값이다. 실패는
예외가 발생해 바로 드러나는 경우와, 예외 없이 틀린 값이나 깨진 연결이 남는 경우로 나눠 본다.

---

## 9. 성능과 비용

### 조기 종료와 최악 비용

선형 검색은 첫 원소가 답이면 O(1), 마지막 또는 부재면 O(n)이다. 입력 분포를 모르면
최악 O(n)이라고 말하고, 평균을 주장하려면 target 위치와 부재 확률 같은 분포 가정이
필요하다.

### 숨은 중첩 비용

겉으로 loop가 하나여도 내부 연산이 O(n)이면 전체는 O(n^2)이 될 수 있다.

```java
for (String x : list) {       // n번
    otherList.contains(x);    // LinkedList라면 최대 n번
}
```

반대로 중첩 loop라도 전체 pointer가 각각 한 번만 전진하는 two-pointer 방식은 O(n)일
수 있다. 문법 모양이 아니라 총 연산 횟수를 센다.

### 연산 수를 직접 세기

```java
int max = values[0];
for (int i = 1; i < values.length; i++) {
    if (values[i] > max) max = values[i];
}
```

길이 n에서 비교는 n-1번이다.

```text
T(n) = a(n-1) + b
     = an + (b-a)
     = O(n)
```

상수와 낮은 차수 항을 제거해도 모든 원소를 확인한다는 핵심은 남는다.

### O(n)이 최적인 경우

정렬되지 않은 n개 값의 최댓값을 정확히 구하려면 확인하지 않은 값이 더 클 수 있다.
따라서 일반 비교 모델에서는 모든 값을 한 번은 봐야 해 Ω(n) 하한이 있고, 위 알고리즘은
O(n)이므로 Θ(n)이다.

### 전처리로 조회를 줄이는 trade-off

```text
매번 선형 검색
  준비 비용 없음
  조회 O(n)

HashSet index 구축
  구축 O(n), 추가 공간 O(n)
  이후 membership 기대 O(1)
  데이터 변경 때 index도 갱신
```

조회 한 번이면 전처리가 손해일 수 있고, 같은 데이터에 조회가 반복되면 이득일 수 있다.

### 두 포인터가 중첩 loop처럼 보여도 O(n)인 예

```java
int left = 0;
int right = 0;
while (right < values.length) {
    right++;
    while (mustShrink(left, right)) {
        left++;
    }
}
```

left와 right가 각각 최대 n번만 증가하고 뒤로 가지 않는다면 총 이동은 2n 수준이어서
O(n)일 수 있다. 각 원소가 몇 번 처리되는지 합으로 센다.

## 14. 꼬리질문 방어

### 질문 지도

| 축 | 질문 | 답변 핵심 |
|---|---|---|
| 내부 | 조기 종료가 있으면 O(1)인가? | best case만 O(1)이고 최악은 여전히 O(n)일 수 있다. |
| 실패 | loop 하나만 보고 O(n)이라 하면? | 내부 연산 비용을 놓쳐 O(n^2)일 수 있다. |
| 선택 | O(n)을 개선할 수 없는 경우는? | 모든 입력이 결과에 영향을 주어 반드시 읽어야 하는 문제다. |

### Q1. O(n)은 항상 나쁜가?

<details><summary>답 보기</summary>

아니다. 모든 데이터를 반드시 봐야 하는 문제라면 O(n)이 자연스럽다. 문제는 O(1)이나 O(log n)으로 풀 수 있는 요구를 불필요하게 O(n)으로 처리하는 경우다.

</details>

### Q2. HashMap의 모든 value를 조건 검색하면 왜 O(n)인가?

<details><summary>답 보기</summary>

HashMap은 key를 bucket으로 보내는 index만 갖고 value 조건의 위치 정보는 없다. 어떤
entry의 value가 조건을 만족하는지 알려면 최악에는 모든 mapping을 확인해야 한다.

</details>

### Q3. 두 배열을 각각 순회하면 O(n^2)인가?

<details><summary>답 보기</summary>

순차적으로 길이 n 배열 두 개를 순회하면 O(n+n)=O(n)이다. 한 배열의 각 원소마다
다른 배열 전체를 순회해야 O(n^2)이다.

</details>

### Q4. 길이가 다른 두 입력 순회 비용은?

<details><summary>답 보기</summary>

각 길이를 n, m이라고 하면 O(n+m)이다. 근거 없이 둘 다 n으로 놓으면 입력 관계를
숨길 수 있다.

</details>

### Q5. 병렬로 n개를 처리하면 시간 복잡도가 O(1)이 되는가?

<details><summary>답 보기</summary>

무한 processor를 가정하지 않는다. 총 work는 여전히 O(n)일 수 있고 실제 wall-clock은
worker 수, 분할, 동기화, I/O에 따라 달라진다. work와 병렬 실행 시간을 구분한다.

</details>
