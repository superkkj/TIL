## 1. 정의

### 정석 설명

root는 트리에서 시작점이 되는 노드다. 일반적으로 부모가 없는 최상위 노드다.

### 쉬운 설명

조직도에서 대표, 파일 시스템에서 최상위 폴더 같은 위치다.

### 어려운 말 먼저 풀기

| 용어 | 쉬운 뜻 |
|---|---|
| root | 부모가 없는 트리의 시작 Node다. |
| root reference | Tree 객체가 시작 Node를 찾아갈 수 있게 보관하는 참조 값이다. |
| depth 0 | root에서 root까지 지나가는 edge가 0개라는 뜻이다. |
| subtree root | 전체 트리의 중간 Node를 그 아래 부분 트리의 시작점으로 볼 때의 root다. |
| rotation | BST의 정렬 순서는 유지하면서 부모·자식 연결을 바꿔 높이를 조정하는 연산이다. 전체 root에서 일어나면 root 참조도 바꿔야 한다. |

### 다시 정리

쉬운 설명은 이해를 위한 비유다. 실제 면접 답변에서는 위 정석 설명의 용어와
전제 조건으로 돌아와 설명한다.

### 왜 필요한가

트리 순회와 탐색은 보통 root에서 시작한다. root가 있어야 아래 subtree들의 기준점이 생긴다.

## 3. 해결 범위와 비목표

이 문서가 직접 설명하는 범위는 Root: 루트의 정의와 상위 자료구조에서의 역할이다. 정렬·동시성·영속성·특정 시간 복잡도는 별도 불변식이나 구현 계약이 있을 때만 보장된다.

쉽게 말하면 이 개념의 이름만 보고 다음 성질까지 자동으로 있다고 단정하면 안 된다.
`동시성`은 여러 실행 흐름이 함께 접근해도 안전한지, `영속성`은 프로그램이 끝난 뒤에도
파일이나 DB 등에 데이터가 남는지, `구현 계약`은 실제 클래스나 API가 무엇을 약속하는지를
뜻한다. 이 성질들은 사용하는 구현체의 공식 설명에서 각각 확인해야 한다.

---

## 4. 내부 구조와 핵심 원리

### root는 구조에서 어떤 역할을 하나

rooted tree에서 root는 모든 node로 가는 경로의 출발점이며 parent가 없는 유일한
node다. 저장 구조에서는 tree 객체가 root 참조 하나를 보관하고, 빈 tree라면 root를
null로 표현할 수 있다.

```java
final class Tree<T> {
    Node<T> root; // null이면 empty tree
}
```

### root를 바꾸면 무엇이 바뀌나

같은 무방향 tree에서도 다른 node를 root로 선택할 수 있다. edge 자체는 같지만
parent, child, depth, subtree 범위가 바뀐다. 따라서 root는 단순히 “그림의 가장 위”가
아니라 계층 방향을 결정하는 기준이다.

### root가 결정하는 것

같은 무방향 tree라도 root를 어디로 정하느냐에 따라 다음 값이 바뀐다.

```text
무방향 edge: A - B - C

A를 root로 선택       B를 root로 선택
A depth 0             B depth 0
└ B depth 1          ├ A depth 1
  └ C depth 2        └ C depth 1
```

edge 집합은 같지만 parent, child, depth, subtree, traversal 시작점이 달라진다. 따라서
root는 그림을 위아래로 그리기 위한 장식이 아니라 계층 관계의 기준이다.

---

## 5. 핵심 연산

### 연결 개념

```text
root
 ├─ child
 └─ child
```

### root가 연산에서 사용되는 지점

- 전체 탐색: root에서 DFS/BFS를 시작한다.
- BST 검색: target 비교를 root에서 시작해 한 subtree를 선택한다.
- Heap: root가 최소값 또는 최대값이다.
- 균형 회전: root가 바뀌면 Tree 객체의 root 참조도 갱신해야 한다.
- 삭제: root가 유일한 node면 삭제 후 empty tree가 된다.

#### root 교체 실패

```text
회전 전 Tree.root = G

    G                 P
   /                 / \
  P       ->        X   G
 /
X
```

링크 회전은 했지만 `Tree.root`를 여전히 G로 두면 새 root P에서 시작할 수 없어 구조
일부에 접근하지 못하는 조용한 버그가 생긴다.

## 7. 불변식

정상 상태에서 유지해야 할 구체 조건은 위 내부 구조와 연산 설명을 기준으로 확인한다. 상위 자료구조의 불변식과 충돌하는 변경은 허용하지 않는다.

여기서 `불변식`은 삽입·조회·삭제 전후에도 계속 참이어야 하는 규칙이다. 쉽게 말해
자료구조가 망가지지 않았다고 판단하는 필수 조건이며, 연산이 끝난 뒤에도 다시 맞아야 한다.

---

## 8. 실패·예외·경계 상황

### 경계 사례

- node 하나인 tree에서는 그 node가 root이면서 leaf다.
- 빈 tree의 root는 null로 표현할 수 있으며 height 규약은 자료마다 다르다.
- root가 여러 개라면 하나의 rooted tree가 아니라 여러 tree의 forest로 본다.

이 개념만 떼어 자료구조를 선택하지 않는다. 필요한 조회·삽입·삭제·정렬·범위·순회 연산을 기준으로 인접 개념과 상위 자료구조를 비교한다.

## 14. 꼬리질문 방어

### 질문 지도

| 축 | 질문 | 답변 핵심 |
|---|---|---|
| 내부 | root를 바꾸면 edge도 바뀌나? | 무방향 edge는 같고 parent-child 방향과 depth가 바뀐다. |
| 실패 | root 참조를 잃으면? | 다른 외부 참조가 없으면 tree 전체 접근 경로를 잃는다. |
| 비교 | 빈 tree의 root는? | 구현에서는 보통 null이며 height 기준은 별도 선언한다. |

### Q1. root가 여러 개인 구조도 트리인가?

<details><summary>답 보기</summary>

root가 여러 개인 구조는 보통 forest라고 부른다. 여러 트리의 집합으로 볼 수 있다.

</details>

### Q2. root를 찾는 비용은 항상 O(1)인가?

<details><summary>답 보기</summary>

Tree 객체가 root 참조를 보관하면 O(1)이다. 임의 node에서 parent만 따라 root를
찾는다면 node depth만큼 O(depth)가 든다. 표현 방식의 전제를 먼저 말해야 한다.

</details>

### Q3. root는 항상 값이 가장 작은 node인가?

<details><summary>답 보기</summary>

아니다. 일반 tree와 BST의 root는 최소값일 필요가 없다. min-heap은 추가 heap
불변식 때문에 root가 최소값이다.

</details>

### Q4. node 하나인 tree의 height와 root 성질은?

<details><summary>답 보기</summary>

edge 수 기준이면 root height는 0이고, 그 node는 root이면서 leaf다. 빈 tree height는
관례가 다르므로 별도로 선언한다.

</details>

### Q5. forest를 하나의 tree처럼 순회하려면?

<details><summary>답 보기</summary>

각 root에서 순회를 시작하거나, 실제 데이터에 없는 가상 super-root를 두고 각 tree
root를 child로 연결하는 모델을 사용할 수 있다.

</details>
