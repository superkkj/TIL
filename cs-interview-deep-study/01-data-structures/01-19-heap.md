## 1. 정의

### 정석 설명

자료구조의 heap은 부모와 자식 사이에 우선순위 규칙을 유지하는 트리 기반 구조다. priority queue 구현에 자주 사용된다.

### 쉬운 설명

가장 중요한 값이 항상 꼭대기에 오도록 관리하는 구조다. 전체가 완벽히 정렬된 것은 아니지만, 꼭대기 값은 빠르게 꺼낼 수 있다.

### 용어 경계

여기서 Heap은 우선순위 자료구조인 binary heap이다. JVM Heap이라는 객체 메모리
영역과 이름만 같고 다른 개념이다.

### 어려운 말 먼저 풀기

| 용어 | 쉬운 뜻 |
|---|---|
| complete binary tree | 마지막 층을 제외한 층은 꽉 채우고, 마지막 층은 왼쪽부터 빈틈없이 채운 이진 트리다. |
| heap invariant | 부모와 자식 사이에 항상 지켜야 하는 우선순위 규칙이다. min-heap에서는 부모가 자식보다 작거나 같다. |
| sift-up | 새 원소를 배열 끝에 넣은 뒤 부모와 비교하며 위로 올리는 복구 과정이다. |
| sift-down | root를 제거한 뒤 끝 원소를 root로 옮기고 자식과 비교하며 아래로 내리는 복구 과정이다. |
| heapify | 여러 원소를 heap 불변식을 만족하는 배치로 만드는 작업이다. |
| stable 정렬 | 비교 결과가 같은 원소들의 기존 상대 순서를 보존하는 정렬이다. 일반적인 heap sort는 이를 보장하지 않는다. |

### 다시 정리

쉬운 설명은 이해를 위한 비유다. 실제 면접 답변에서는 위 정석 설명의 용어와
전제 조건으로 돌아와 설명한다.

## 3. 해결 범위와 비목표

이 문서가 직접 설명하는 범위는 Heap: 힙의 정의와 상위 자료구조에서의 역할이다. 정렬·동시성·영속성·특정 시간 복잡도는 별도 불변식이나 구현 계약이 있을 때만 보장된다.

쉽게 말하면 이 개념의 이름만 보고 다음 성질까지 자동으로 있다고 단정하면 안 된다.

뜻한다. 이 성질들은 사용하는 구현체의 공식 설명에서 각각 확인해야 한다.

## 4. 내부 구조와 핵심 원리

### 종류

| 종류 | 의미 |
|---|---|
| Min Heap | 부모가 자식보다 작거나 같음 |
| Max Heap | 부모가 자식보다 크거나 같음 |

### 배열 표현

```text
parent index = (i - 1) / 2
left child = 2i + 1
right child = 2i + 2
```

### complete binary tree가 필요한 이유

heap order만 있고 모양 제한이 없다면 한쪽으로 긴 tree가 될 수 있다. binary heap은
마지막 level을 제외한 level을 모두 채우고 마지막 level도 왼쪽부터 채우는 complete
binary tree 모양을 유지한다.

```text
가능한 heap 모양              complete가 아닌 모양

        1                            1
      /   \                            \
     3     2                            2
    / \   /                              \
   8  7  5                                3
```

complete 모양 덕분에 높이가 O(log n)이고 빈 포인터 없이 배열 index로 부모·자식을
계산할 수 있다.

## 5. 핵심 연산

### 삽입: sift-up

1. complete tree 모양을 유지하도록 배열 끝에 넣는다.
2. 부모보다 작으면 교환한다.
3. root에 도달하거나 부모가 더 작을 때까지 반복한다.

height가 O(log n)이므로 삽입도 O(log n)이다.

### head 제거: sift-down

1. root 값을 결과로 보관한다.
2. 배열 마지막 값을 root로 옮긴다.
3. 두 child 중 더 작은 쪽과 비교해 필요하면 교환한다.
4. 불변식이 복구될 때까지 반복한다.

peek는 root만 읽어 O(1), poll은 O(log n), 임의 값 검색은 정렬 구조가 아니므로
O(n)이다.

### 임의 값 갱신과 삭제

heap은 root가 아닌 임의 key의 위치를 바로 찾지 못한다. 위치를 찾는 데 O(n)이 들 수
있다. 별도 `key -> heap index` map을 유지하면 위치를 빠르게 찾을 수 있지만 swap할
때마다 map index도 함께 갱신해야 한다.

값 변경 후에는 방향을 판단한다.

```text
min-heap에서 값이 작아짐 -> parent 방향 sift-up 가능
min-heap에서 값이 커짐   -> child 방향 sift-down 가능
```

## 6. 상태 추적과 구현

### 최소 sift-up 코드

```java
void siftUp(int[] heap, int i) {
    while (i > 0) {
        int p = (i - 1) / 2;
        if (heap[p] <= heap[i]) return;
        int tmp = heap[p];
        heap[p] = heap[i];
        heap[i] = tmp;
        i = p;
    }
}
```

### 삽입과 삭제 전체 추적

```text
초기 min-heap: [2, 5, 4, 9, 7, 8]

1 삽입
배열 끝:       [2, 5, 4, 9, 7, 8, 1]
부모 4와 교환: [2, 5, 1, 9, 7, 8, 4]
부모 2와 교환: [1, 5, 2, 9, 7, 8, 4]

root 1 삭제
마지막 4 이동: [4, 5, 2, 9, 7, 8]
작은 child 2:  [2, 5, 4, 9, 7, 8]
```

각 단계에서 배열 끝을 사용하므로 shape가 유지된다.

## 7. 불변식

### 구조와 불변식

binary heap은 complete binary tree이며 보통 0-based 배열에 저장한다.

```text
parent(i) = (i - 1) / 2
left(i)   = 2 * i + 1
right(i)  = 2 * i + 2
```

min-heap의 불변식은 모든 edge에서 `parent <= child`다. 이 규칙은 root가 전체
최솟값임을 보장하지만, 형제나 서로 다른 subtree 사이의 전체 순서는 보장하지 않는다.

### heap의 두 불변식

```text
1. shape invariant
   complete binary tree다.

2. order invariant
   min-heap이면 모든 parent <= child다.
   max-heap이면 모든 parent >= child다.
```

삽입은 배열 끝에 넣어 shape를 먼저 지킨 뒤 sift-up으로 order를 복구한다. root 삭제는
마지막 값을 root로 옮겨 shape를 지킨 뒤 sift-down으로 order를 복구한다.

## 9. 성능과 비용

### heapify가 O(n)인 이유

각 element를 차례로 삽입하면 O(n log n)이지만, 배열의 마지막 내부 node부터
sift-down하는 bottom-up heapify는 O(n)이다. 많은 leaf는 이동할 높이가 0이고,
위로 갈수록 node 수가 급격히 줄어 전체 작업량의 합이 선형으로 제한된다.

### bottom-up heapify 비용을 합으로 보기

leaf는 sift-down할 필요가 없다. 전체 node의 절반가량이 leaf이고, 높이 1만큼 움직일
수 있는 node는 더 적으며, root처럼 많이 움직일 수 있는 node는 매우 적다.

```text
대략적인 작업량
n/2개 * 0단계
n/4개 * 1단계
n/8개 * 2단계
...
```

이 합은 O(n)으로 제한된다. 각 원소를 빈 heap에 하나씩 삽입하는 O(n log n) 방식과
구분해야 한다.

## 14. 꼬리질문 방어

### 질문 지도

| 축 | 질문 | 답변 핵심 |
|---|---|---|
| 내부 | poll에서 어느 child와 교환하나? | min-heap이면 두 child 중 더 작은 쪽과 비교해야 불변식이 복구된다. |
| 실패 | root에 마지막 값을 옮기고 끝내면? | 부모-자식 순서가 깨질 수 있어 sift-down이 필요하다. |
| 비교 | heap과 JVM Heap 차이는? | 전자는 우선순위 자료구조, 후자는 객체·배열 런타임 메모리 영역이다. |

### Q1. heap은 BST인가?

<details><summary>답 보기</summary>

아니다. heap은 부모-자식 우선순위만 보장하고, 왼쪽 subtree가 작고 오른쪽 subtree가 크다는 BST 정렬 규칙을 보장하지 않는다.

</details>

### Q2. BST 대신 heap으로 값을 빠르게 검색할 수 있나?

<details><summary>답 보기</summary>

최솟값 또는 최댓값 head 조회에는 heap이 적합하다. 임의 key 검색이나 범위 순회는
전체 정렬 관계가 없어 O(n)이 될 수 있으므로 BST 계열이 더 적합하다.

</details>

### Q3. heap의 왼쪽 child가 오른쪽 child보다 작아야 하는가?

<details><summary>답 보기</summary>

아니다. min-heap은 각 parent가 child보다 작거나 같다는 관계만 요구한다. 형제끼리의
순서는 제한하지 않는다.

</details>

### Q4. heap 높이가 O(log n)인 근거는?

<details><summary>답 보기</summary>

complete binary tree이기 때문이다. level이 하나 늘 때 담을 수 있는 node 수가 대략
두 배가 되어 n개를 담는 높이가 로그 규모로 제한된다.

</details>

### Q5. 배열 중간에 빈 칸이 있어도 heap인가?

<details><summary>답 보기</summary>

일반적인 binary heap의 complete shape invariant를 깨뜨린다. 배열 `0..size-1`을
연속으로 사용해야 부모·자식 index 계산과 O(log n) 높이 보장이 자연스럽게 유지된다.

</details>

### Q6. heap sort의 시간과 공간은?

<details><summary>답 보기</summary>

배열을 O(n)에 heapify한 뒤 root 제거를 n번 수행해 O(n log n) 시간이다. in-place
구현은 O(1) 추가 배열 공간으로 만들 수 있지만 일반적인 heap sort는 stable하지 않다.

</details>
