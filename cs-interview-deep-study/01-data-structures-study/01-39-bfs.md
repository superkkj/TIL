### [가까운 친구부터 전부 만나기] 너비 우선 탐색 (Breadth-First Search)
> 🔑 핵심 키워드: BFS, graph, frontier, visited, unweighted graph, shortest path, Queue, multi-source BFS

웨이드님~ 트리 순회에서 한 발 넓혀서 그래프 탐색으로 가보자. 😊 BFS는 "친구의 친구를 보러 가기 전에, 먼저 내 친구들을 전부 보는 방식"이야. level-order의 그래프 확장판이라고 봐도 돼.

***

### 🌐 핵심 원리 (Core Principles & Concepts)

BFS는 시작점에서 가까운 노드부터 탐색하는 그래프/트리 탐색 방식이고, 일반적으로 queue를 사용해. 표준 근거는 NIST DADS의 Breadth-First Search 항목이야. [출처: 01-39-bfs.md §1, §0(NIST DADS Breadth-First Search 재인용)]

BFS가 필요한 이유는 이 정의가 참여하는 문제에서 이전 저장·탐색 방식의 한계를 줄이기 위해서고, 구체적인 필요성은 상위 자료구조의 사용 목적과 함께 판단해. [출처: 01-39-bfs.md §2]

특히 이 문서에서 중요한 건 "사용하면 안 되는 경우" 표야. 구체 알고리즘의 적용 조건을 확인하지 않고 "최단 거리니까 BFS"라고 고르면 틀릴 수 있어. [출처: 01-39-bfs.md §3]

| graph 조건 | BFS 판단 |
|---|---|
| 모든 edge 비용 동일 | 최소 edge 수/비용 경로 가능 |
| 비용이 0 또는 1 | 0-1 BFS 같은 변형 검토 |
| 서로 다른 비음수 비용 | Dijkstra 검토 |
| 음수 edge 포함 | 조건에 맞는 별도 알고리즘 필요 |

용어 표 원형 보존! 🤓 [출처: 01-39-bfs.md §1]

| 용어 | 쉬운 뜻 |
|---|---|
| BFS | 시작점에서 edge 1개 거리, 2개 거리, 3개 거리 순서로 범위를 넓혀 가는 탐색이다. |
| graph | 여러 Node와 그 Node들을 잇는 edge로 관계를 표현한 구조다. |
| frontier | 발견은 했지만 아직 이웃을 전부 확인하지 않은 다음 처리 대상들의 경계다. Queue에 들어 있다. |
| visited | 이미 발견한 Node를 다시 넣지 않도록 기록하는 집합이다. graph의 cycle에서 반복을 막는다. |
| unweighted graph | edge마다 별도 비용이 없거나 모두 같은 비용으로 보는 graph다. |
| shortest path | 시작점에서 도착점까지 거치는 edge 수가 가장 적은 경로다. BFS의 이 보장은 unweighted 조건에서 성립한다. |

쉬운 설명은 비유고, 면접에서는 정석 용어와 전제 조건으로 돌아와 설명하자. [출처: 01-39-bfs.md §1 다시 정리]

### 🚀 내부 구조와 핵심 연산

#### 🛂 graph에서는 visited가 필수다

Tree는 parent를 제외하면 돌아오는 cycle이 없지만 일반 graph는 cycle과 여러 경로가 있어. 그래서 node를 enqueue할 때 visited로 표시해야 같은 node가 Queue에 중복으로 많이 들어가는 것을 막을 수 있어. [출처: 01-39-bfs.md §4]

```java
queue.offer(start);
visited.add(start);
distance.put(start, 0);

while (!queue.isEmpty()) {
    Node current = queue.poll();
    for (Node next : current.neighbors()) {
        if (visited.add(next)) {
            distance.put(next, distance.get(current) + 1);
            parent.put(next, current);
            queue.offer(next);
        }
    }
}
```

`parent`를 역으로 따라가면 실제 최단 경로를 복원할 수 있어. [출처: 01-39-bfs.md §4]

왜 무가중 최단 거리냐면 — BFS는 시작점에서 edge 수가 0, 1, 2인 node 순서로 방문하고, 모든 edge 비용이 같다고 볼 수 있을 때 처음 발견한 경로가 최소 edge 수 경로이기 때문이야. 서로 다른 양의 가중치가 있으면 Dijkstra 같은 알고리즘이 필요하고, 음수 edge까지 있으면 추가 조건에 맞는 다른 알고리즘을 선택해야 해. [출처: 01-39-bfs.md §4]

> 😒 **Bailey**: 흥. "최단 경로 = BFS" 라고 반사적으로 답하는 사람, 열에 아홉이야. 가중치가 서로 다른 순간 BFS의 '처음 발견 = 최단' 보장은 무너져. §3 표를 못 외우겠으면 최소한 "unweighted 전제"라는 말은 입에 붙여 둬. 😠 [출처: 01-39-bfs.md §3, §4]

#### 🔄 기본 동작과 연결되지 않은 graph

내부 동작 네 단계. [출처: 01-39-bfs.md §5]

```text
1. 시작 노드를 queue에 넣음
2. queue에서 꺼내 방문
3. 인접 노드 중 아직 방문하지 않은 노드를 queue에 넣음
4. queue가 빌 때까지 반복
```

한 start에서 BFS하면 그 start가 속한 connected component만 방문해. graph 전체를 순회하려면 아직 방문하지 않은 vertex마다 BFS를 다시 시작해야 해. [출처: 01-39-bfs.md §5]

#### 🧲 multi-source BFS

여러 시작점에서 가장 가까운 거리를 구하려면 모든 source를 distance 0으로 Queue에 먼저 넣어. [출처: 01-39-bfs.md §5]

```text
for source in sources:
  visited.add(source)
  distance[source] = 0
  queue.offer(source)

이후 일반 BFS
```

각 source에서 BFS를 따로 n번 하는 것보다 모든 source의 frontier를 동시에 확장하는 방식이야. [출처: 01-39-bfs.md §5]

#### 🖐️ graph를 직접 추적하기

[출처: 01-39-bfs.md §6]

```text
A -- B -- D
|    |
C -- E
```

A에서 시작하고 인접 목록 순서를 B, C라고 하자.

```text
초기: visited={A}, Queue=[A], distance[A]=0

poll A
  B 발견: visited 추가, distance=1, parent=A, offer
  C 발견: visited 추가, distance=1, parent=A, offer
Queue=[B,C]

poll B
  D 발견: distance=2, parent=B
  E 발견: distance=2, parent=B
Queue=[C,D,E]

poll C
  A와 E는 이미 visited
Queue=[D,E]
```

E는 C를 통해서도 갈 수 있지만 처음 발견한 A-B-E 경로도 edge 수 2로 최단이야. parent를 `E -> B -> A`로 역추적한 뒤 뒤집으면 실제 경로를 얻어. [출처: 01-39-bfs.md §6]

### 🧭 불변식과 실패·경계 상황

BFS의 핵심 불변식 세 줄. [출처: 01-39-bfs.md §7]

```text
Queue에서 먼저 나오는 node의 distance는 뒤 node보다 크지 않다.
node는 최초 enqueue 시 visited가 된다.
처음 기록한 distance는 최소 edge 수 거리다.
```

이 불변식은 모든 edge 비용이 동일한 무가중 모델에서 성립해. [출처: 01-39-bfs.md §7]

실패·경계는 경계값(첫 index, 마지막 index, 0개처럼 조건이 바뀌는 가장자리 값)과 빈 구조를 먼저 확인하고, 불변식이 깨졌을 때 발생하는 예외와 예외 없이 잘못되는 결과를 구분해서 봐. 가중 graph에 BFS를 그대로 쓰면 edge 수 최소와 비용 최소가 달라 예외 없이 잘못된 답이 나오는 대표적인 "조용한 실패"야. [출처: 01-39-bfs.md §8, §14 질문 지도]

### ⚡ 성능·비용과 대안 비교

복잡도는 표현 방식 전제를 붙여 말해야 해. adjacency list와 visited를 사용하면 각 vertex를 한 번 enqueue하고 각 edge를 확인해 O(V+E) 시간, O(V) 추가 공간이 들어. adjacency matrix라면 각 vertex에서 V개 칸을 확인해 O(V^2)가 될 수 있어. [출처: 01-39-bfs.md §9]

대표 쓰임. [출처: 01-39-bfs.md §10]

- 트리 level-order 순회
- 가중치 없는 그래프 최단 거리
- 가까운 노드부터 탐색

DFS와 비교 표는 원형 유지. [출처: 01-39-bfs.md §10]

| 기준 | BFS | DFS |
|---|---|---|
| 자료구조 | Queue | Stack/재귀 |
| 방향 | 가까운 곳 먼저 | 깊은 곳 먼저 |
| 최단 거리 | 가중치 없는 그래프에 적합 | 직접적이지 않음 |

근거 기준(표준 자료 / Oracle Java 17 문서 / OpenJDK 소스 연결 시 구현 세부)과 실무 판단 기준(API 계약, 입력 크기, 데이터 분포, 변경 빈도, 동시 접근)도 함께. [출처: 01-39-bfs.md §11, §12]

### Q1. BFS는 왜 queue를 쓰나?

<details><summary>답 보기</summary>

먼저 발견한 가까운 노드를 먼저 처리해야 하므로 FIFO 구조인 queue가 맞다. [출처: 01-39-bfs.md §14 Q1]

</details>

### Q2. visited를 dequeue할 때 표시하면 왜 비효율적일 수 있나?

<details><summary>답 보기</summary>

한 node가 dequeue되기 전에 여러 이웃이 같은 node를 각각 enqueue한다. enqueue 시점에 표시하면 Queue에 처음 한 번만 들어가고, 최초 발견 거리도 고정된다. [출처: 01-39-bfs.md §14 Q2]

</details>

### Q3. target을 발견하자마자 종료해도 되는가?

<details><summary>답 보기</summary>

무가중 BFS에서 enqueue하며 처음 발견한 시점에 최소 거리 값은 정해진다. 필요한 결과가 거리/경로 하나뿐이면 종료할 수 있지만, 같은 거리의 모든 해나 전체 탐색이 목적이면 계속해야 한다. [출처: 01-39-bfs.md §14 Q3]

</details>

### Q4. BFS 결과 순서는 항상 유일한가?

<details><summary>답 보기</summary>

아니다. 같은 depth 안의 순서는 adjacency iteration 순서에 따라 달라질 수 있다. 최단 거리 값은 같아도 선택된 parent와 구체 경로가 달라질 수 있다. [출처: 01-39-bfs.md §14 Q4]

</details>

### Q5. adjacency list에서 O(V+E)인 이유는?

<details><summary>답 보기</summary>

각 vertex를 한 번 처리하고 각 adjacency entry를 확인한다. 무방향 graph에서는 한 edge가 두 목록에 나타나도 상수 배이므로 O(V+E)다. [출처: 01-39-bfs.md §14 Q5]

</details>

### Q6. BFS Queue가 가장 커지는 시점은?

<details><summary>답 보기</summary>

frontier가 가장 넓은 level을 처리할 때다. 시간 복잡도와 별개로 graph의 폭에 따라 O(V) 추가 공간이 필요하다. [출처: 01-39-bfs.md §14 Q6]

</details>

### 📝 요약 (Summary)

- BFS = 시작점에서 가까운 노드부터 탐색, Queue(FIFO) 사용. 트리에 적용하면 level-order. [출처: 01-39-bfs.md §1, §13]
- graph에서는 enqueue 시점 visited 표시가 필수 — 중복 삽입을 막고 최초 발견 거리를 고정한다. [출처: 01-39-bfs.md §4, §14 Q2]
- '처음 발견 = 최단'은 무가중(모든 edge 비용 동일) 전제에서 성립. 비음수 상이 가중치는 Dijkstra, 0/1 가중치는 0-1 BFS 변형 검토. [출처: 01-39-bfs.md §3, §7]
- parent 기록 후 역추적하면 경로 복원. connected component 밖은 방문하지 않으므로 전체 순회는 미방문 vertex마다 재시작. [출처: 01-39-bfs.md §4, §5]
- adjacency list 기준 O(V+E) 시간 / O(V) 공간, matrix 기준 O(V^2) — 표현 전제를 붙여 말한다. [출처: 01-39-bfs.md §9]

## 사실 (F)
F1. BFS는 시작점에서 가까운 노드부터 탐색하는 그래프/트리 탐색 방식이며 일반적으로 queue를 사용한다. [출처: 01-39-bfs.md §1; NIST DADS Breadth-First Search (원본 §0 재인용)]
F2. BFS의 최단 경로 보장은 unweighted(모든 edge 비용 동일) 조건에서 성립하고, 서로 다른 비음수 비용은 Dijkstra, 음수 edge 포함은 별도 알고리즘이 필요하다. [출처: 01-39-bfs.md §1, §3]
F3. 핵심 불변식: Queue 앞 node의 distance는 뒤 node보다 크지 않고, node는 최초 enqueue 시 visited가 되며, 처음 기록한 distance가 최소 edge 수 거리다(무가중 모델). [출처: 01-39-bfs.md §7]
F4. adjacency list + visited 기준 O(V+E) 시간, O(V) 추가 공간이고, adjacency matrix는 O(V^2)가 될 수 있다. [출처: 01-39-bfs.md §9]
F5. 한 start의 BFS는 그 start가 속한 connected component만 방문하며, 전체 순회는 미방문 vertex마다 재시작한다. [출처: 01-39-bfs.md §5]
F6. multi-source BFS는 모든 source를 distance 0으로 먼저 enqueue해 frontier를 동시에 확장한다. [출처: 01-39-bfs.md §5]
F7. 예시 graph(A-B-D, A-C, B-E, C-E)에서 A 시작 BFS의 E 최단 거리는 2이고 parent 역추적 경로는 A-B-E다. [출처: 01-39-bfs.md §6]

## 모름 (U)
