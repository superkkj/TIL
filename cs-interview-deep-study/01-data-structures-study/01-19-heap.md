### 🧠 핵심 원리 (Core Principles & Concepts)

정석 정의부터. 자료구조의 heap은 **부모와 자식 사이에 우선순위 규칙을 유지하는 트리 기반 구조**야. priority queue 구현에 자주 사용돼. [출처: 01-19-heap.md §1, NIST DADS Heap (원본 §0 재인용)]

쉽게 말하면 가장 중요한 값이 항상 꼭대기에 오도록 관리하는 구조야. 전체가 전부 정렬된 것은 아니지만, 꼭대기 값은 빠르게 꺼낼 수 있어. 이 비유는 이해용이고 면접에서는 정석 용어로 돌아와 답해야 해. [출처: 01-19-heap.md §1]

용어 경계 하나 짚고 가자. 여기서 Heap은 우선순위 자료구조인 binary heap이야. JVM Heap이라는 객체 메모리 영역과는 이름만 같고 다른 개념이야. [출처: 01-19-heap.md §1 용어 경계, §8]

heap이 필요한 이유는 이 정의가 참여하는 문제에서 이전 저장·탐색 방식의 한계를 줄이기 위해서야. 구체적인 필요성은 상위 자료구조의 사용 목적과 함께 판단해. [출처: 01-19-heap.md §2] 그리고 이 문서의 범위는 heap의 정의와 상위 자료구조에서의 역할까지고, 정렬·동시성·영속성·특정 시간 복잡도는 별도 불변식이나 구현 계약이 있을 때만 보장돼. 이름만 보고 성질을 단정하지 말고 구현체의 공식 설명에서 각각 확인해야 해. [출처: 01-19-heap.md §3]

### ⚙️ 내부 구조와 핵심 연산 (Shape + Order Machine)

#### 🏗️ 종류와 배열 표현

**정의.** heap은 방향에 따라 두 종류로 나뉘어. Min Heap은 부모가 자식보다 작거나 같고, Max Heap은 부모가 자식보다 크거나 같아. 원본 표 그대로야. [출처: 01-19-heap.md §4]

| 종류 | 의미 |
|---|---|
| Min Heap | 부모가 자식보다 작거나 같음 |
| Max Heap | 부모가 자식보다 크거나 같음 |

**메커니즘.** binary heap은 보통 0-based 배열에 저장하고 index 계산으로 부모·자식을 오가. 포인터 없이 산술 연산만으로 이동이 되는 게 배열 표현의 힘이야. 공식은 아래와 같아. 이 공식이 성립하려면 배열이 연속으로 채워져 있어야 해. [출처: 01-19-heap.md §4, §7]

```text
parent index = (i - 1) / 2
left child = 2i + 1
right child = 2i + 2
```

**예시 — 완전 이진 트리가 필요한 이유.** heap order만 있고 모양 제한이 없다면 한쪽으로 긴 tree가 될 수 있어. 그래서 binary heap은 마지막 level을 제외한 level을 모두 채우고 마지막 level도 왼쪽부터 채우는 완전 이진 트리 모양을 유지해. 이 모양은 높이를 O(log n)으로 제한하고 빈 포인터 없이 배열 index로 부모·자식을 계산하게 해. [출처: 01-19-heap.md §4]

```text
완전 이진 트리 heap 모양       완전 이진 트리가 아닌 모양

        1                            1
      /   \                            \
     3     2                            2
    / \   /                              \
   8  7  5                                3
```

#### 🔄 삽입과 제거 — sift-up / sift-down

**정의.** 삽입은 sift-up, head 제거는 sift-down으로 처리해. 둘 다 "shape 먼저, order 복구 나중" 순서야. 원본 절차를 그대로 옮겼어. [출처: 01-19-heap.md §5]

삽입(sift-up) 절차:
1. 완전 트리 모양을 유지하도록 배열 끝에 넣는다.
2. 부모보다 작으면 교환한다.
3. root에 도달하거나 부모가 더 작을 때까지 반복한다.

height가 O(log n)이므로 삽입도 O(log n)이야. [출처: 01-19-heap.md §5]

head 제거(sift-down) 절차:
1. root 값을 결과로 보관한다.
2. 배열 마지막 값을 root로 옮긴다.
3. 두 child 중 더 작은 쪽과 비교해 필요하면 교환한다.
4. 불변식이 복구될 때까지 반복한다.

**비용 정리.** peek는 root만 읽어 O(1), poll은 O(log n), 임의 값 검색은 정렬 구조가 아니므로 O(n)이야. [출처: 01-19-heap.md §5]

**예시 — 임의 값 갱신과 삭제.** heap은 root가 아닌 임의 key의 위치를 바로 찾지 못해서 위치를 찾는 데 O(n)이 들 수 있어. 별도 `key -> heap index` map을 유지하면 위치를 빠르게 찾을 수 있지만 swap할 때마다 map index도 함께 갱신해야 해. 값 변경 후에는 방향을 판단해. [출처: 01-19-heap.md §5]

```text
min-heap에서 값이 작아짐 -> parent 방향 sift-up 가능
min-heap에서 값이 커짐   -> child 방향 sift-down 가능
```

#### 🔬 상태 추적과 최소 코드

**정의.** 원본의 최소 sift-up 코드와 삽입·삭제 전체 추적을 그대로 보존할게. 손으로 배열 상태를 굴려 보는 게 면접 대비의 핵심이야. 각 단계에서 배열 끝을 사용하므로 shape가 유지된다는 점을 눈으로 확인해 봐. [출처: 01-19-heap.md §6]

```java
void siftUp(int[] heap, int i) {
    while (i > 0) {
        int p = (i - 1) / 2;
        if (heap[p] <= heap[i]) return;
        int tmp = heap[p];
        heap[p] = heap[i];
        heap[i] = tmp;
        i = p;
    }
}
```

```text
초기 min-heap: [2, 5, 4, 9, 7, 8]

1 삽입
배열 끝:       [2, 5, 4, 9, 7, 8, 1]
부모 4와 교환: [2, 5, 1, 9, 7, 8, 4]
부모 2와 교환: [1, 5, 2, 9, 7, 8, 4]

root 1 삭제
마지막 4 이동: [4, 5, 2, 9, 7, 8]
작은 child 2:  [2, 5, 4, 9, 7, 8]
```

### 🧷 두 불변식 (Shape & Order Invariants)

heap의 불변식은 두 개야. 원본 정리를 그대로 가져올게. [출처: 01-19-heap.md §7]

```text
1. shape invariant
   완전 이진 트리다.

2. order invariant
   min-heap이면 모든 parent <= child다.
   max-heap이면 모든 parent >= child다.
```

min-heap의 불변식은 모든 edge에서 `parent <= child`야. 이 규칙은 root가 전체 최솟값임을 보장하지만, 형제나 서로 다른 subtree 사이의 전체 순서는 보장하지 않아. [출처: 01-19-heap.md §7]

삽입은 배열 끝에 넣어 shape를 먼저 지킨 뒤 sift-up으로 order를 복구하고, root 삭제는 마지막 값을 root로 옮겨 shape를 지킨 뒤 sift-down으로 order를 복구해. 두 불변식을 "동시에 완성"하는 게 아니라 "shape 먼저, order 복구"라는 순서로 지킨다는 게 포인트야. [출처: 01-19-heap.md §7]

경계 상황 주의 하나 더 — JVM heap 메모리와 heap 자료구조는 다른 개념이야. [출처: 01-19-heap.md §8]

### 📊 성능·비용과 대안 비교 (Cost & Alternatives)

#### heapify가 O(n)인 이유

각 element를 차례로 삽입하면 O(n log n)이지만, 배열의 마지막 내부 node부터 sift-down하는 bottom-up heapify는 O(n)이야. 많은 leaf는 이동할 높이가 0이고, 위로 갈수록 node 수가 급격히 줄어 전체 작업량의 합이 선형으로 제한되기 때문이야. [출처: 01-19-heap.md §9]

leaf는 sift-down할 필요가 없어. 전체 node의 절반가량이 leaf이고, 높이 1만큼 움직일 수 있는 node는 더 적으며, root처럼 많이 움직일 수 있는 node는 매우 적어. 작업량을 합으로 보면 이래. [출처: 01-19-heap.md §9]

```text
대략적인 작업량
n/2개 * 0단계
n/4개 * 1단계
n/8개 * 2단계
상위 level로 갈수록 node 수는 절반씩 감소
```

이 합은 O(n)으로 제한돼. 각 원소를 빈 heap에 하나씩 삽입하는 O(n log n) 방식과 구분해야 해. [출처: 01-19-heap.md §9]

#### Heap, BST, 정렬 배열 비교 (원본 표 유지)

| 요구사항 | Heap | 균형 BST | 정렬 배열 |
|---|---:|---:|---:|
| 최소값 확인 | O(1) | O(log n) 또는 끝 참조 시 더 빠름 | O(1) |
| 최소값 제거 | O(log n) | O(log n) | 원소 이동 O(n) |
| 임의 key 탐색 | O(n) | O(log n) | O(log n) |
| 정렬 순회 | 직접 보장 안 함 | O(n) | 이미 정렬 |
| 삽입 | O(log n) | O(log n) | O(n) |

[출처: 01-19-heap.md §10]

근거를 말하는 기준도 같이 기억해. 일반 자료구조·알고리즘 성질은 §0에 연결된 표준·공식 자료 기준, Java API 동작은 Oracle Java 17 문서가 명시한 범위에서만 보장, OpenJDK 내부 세부는 소스가 연결된 경우에만 구현 세부로 구분해. 실무에서는 개념 이름보다 구현체의 API 계약, 입력 크기, 데이터 분포, 변경 빈도와 동시 접근 조건을 함께 확인하고. [출처: 01-19-heap.md §11, §12]

#### Q1. heap은 BST인가?

<details><summary>답 보기</summary>

아니다. heap은 부모-자식 우선순위만 보장하고, 왼쪽 subtree가 작고 오른쪽 subtree가 크다는 BST 정렬 규칙을 보장하지 않는다. [출처: 01-19-heap.md §14 Q1]

</details>

#### Q2. BST 대신 heap으로 값을 빠르게 검색할 수 있나?

<details><summary>답 보기</summary>

최솟값 또는 최댓값 head 조회에는 heap이 적합하다. 임의 key 검색이나 범위 순회는 전체 정렬 관계가 없어 O(n)이 될 수 있으므로 BST 계열이 더 적합하다. [출처: 01-19-heap.md §14 Q2]

</details>

#### Q3. heap의 왼쪽 child가 오른쪽 child보다 작아야 하는가?

<details><summary>답 보기</summary>

아니다. min-heap은 각 parent가 child보다 작거나 같다는 관계만 요구한다. 형제끼리의 순서는 제한하지 않는다. [출처: 01-19-heap.md §14 Q3]

</details>

#### Q4. heap 높이가 O(log n)인 근거는?

<details><summary>답 보기</summary>

완전 이진 트리이기 때문이다. level이 하나 늘 때 담을 수 있는 node 수가 대략 두 배가 되어 n개를 담는 높이가 로그 규모로 제한된다. [출처: 01-19-heap.md §14 Q4]

</details>

#### Q5. 배열 중간에 빈 칸이 있어도 heap인가?

<details><summary>답 보기</summary>

일반적인 binary heap의 완전 이진 트리 형태 invariant를 깨뜨린다. 배열 `0..size-1`을 연속으로 사용해야 부모·자식 index 계산과 O(log n) 높이 보장이 자연스럽게 유지된다. [출처: 01-19-heap.md §14 Q5]

</details>

#### Q6. heap sort의 시간과 공간은?

<details><summary>답 보기</summary>

배열을 O(n)에 heapify한 뒤 root 제거를 n번 수행해 O(n log n) 시간이다. in-place 구현은 O(1) 추가 배열 공간으로 만들 수 있지만 일반적인 heap sort는 stable하지 않다. [출처: 01-19-heap.md §14 Q6]

</details>

### 📌 요약 (Summary)

- heap은 부모-자식 사이 우선순위 규칙을 유지하는 트리 기반 구조로, priority queue 구현에 자주 쓰인다. [출처: 01-19-heap.md §1]
- 불변식은 shape(완전 이진 트리)와 order(min이면 parent <= child) 두 개다. 삽입·삭제 모두 shape를 먼저 지키고 sift로 order를 복구한다. [출처: 01-19-heap.md §7]
- order invariant는 root가 최솟값(min-heap 기준)임을 보장할 뿐, 형제·subtree 간 전체 순서는 보장하지 않는다. [출처: 01-19-heap.md §7]
- peek O(1), 삽입·poll O(log n), 임의 key 검색 O(n)이다. bottom-up heapify는 O(n)이고 개별 삽입 반복(O(n log n))과 구분한다. [출처: 01-19-heap.md §5, §9]
- 자료구조 heap과 JVM Heap 메모리는 이름만 같은 다른 개념이다. [출처: 01-19-heap.md §1, §8]

## 사실 (F)

F1. heap은 부모-자식 사이 우선순위 규칙(heap invariant)을 유지하는 트리 기반 구조다. [출처: NIST DADS Heap (원본 01-19-heap.md §0·§1 재인용)]
F2. binary heap은 완전 이진 트리(shape invariant)와 parent-child 순서(order invariant) 두 불변식을 유지하며, 배열에서 `parent=(i-1)/2, left=2i+1, right=2i+2`로 이동한다. [출처: 01-19-heap.md §4, §7]
F3. 완전 이진 트리 모양은 높이를 O(log n)으로 제한하며 삽입·poll의 비용은 O(log n)이다. [출처: 01-19-heap.md §4, §5]
F4. peek는 O(1), 임의 key 검색은 O(n)이다. 임의 key 위치를 빠르게 찾으려면 별도 key→index map과 swap 시 동기 갱신이 필요하다. [출처: 01-19-heap.md §5]
F5. bottom-up heapify는 O(n)이며, 원소를 하나씩 삽입하는 O(n log n) 방식과 구분된다. [출처: 01-19-heap.md §9]
F6. heap sort는 O(n log n) 시간이고 일반적인 구현은 stable하지 않다. [출처: 01-19-heap.md §14 Q6]

## 모름 (U)

U2. d-ary heap, Fibonacci heap 등 binary heap 외 변형의 성질은 원본에 없어 본문에서 제외했다.
U3. heap의 역사적 기원(heapsort 고안 맥락 등)은 원본에 근거가 없어 다루지 않았다.
