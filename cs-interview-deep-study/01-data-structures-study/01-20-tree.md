### 🧠 핵심 원리 (Core Principles & Concepts)

정석 정의부터. 일반 graph 이론에서 tree는 **모든 vertex가 연결되어 있고 cycle이 없는 graph**야. 여기서 root를 하나 정해 방향과 계층을 부여하면 rooted tree가 되고, parent-child 관계로 설명하게 돼. [출처: 01-20-tree.md §1, NIST DADS Tree (원본 §0 재인용)]

쉽게 말하면 회사 조직도나 폴더 구조처럼 위아래 관계가 있는 데이터 구조야. 이 비유는 이해용이고, 면접에서는 connected·acyclic 같은 정석 용어와 전제 조건으로 돌아와 설명해야 해. [출처: 01-20-tree.md §1]

tree가 필요한 이유는 계층 관계, 정렬된 탐색, 범위 탐색, 우선순위 구조, 파싱 구조 등을 표현하기 위해서야. [출처: 01-20-tree.md §2]

범위 주의! 이 문서가 직접 설명하는 범위는 tree의 정의와 상위 자료구조에서의 역할이야. 정렬·동시성·영속성·특정 시간 복잡도는 별도 불변식이나 구현 계약이 있을 때만 보장돼. `동시성`은 여러 실행 흐름의 안전성, `영속성`은 종료 후 데이터 보존, `구현 계약`은 실제 클래스·API의 약속을 뜻하고, 각각 구현체의 공식 설명에서 확인해야 해. [출처: 01-20-tree.md §3]

#### 📖 어려운 말 먼저 풀기 (원본 용어 표 보존)

| 용어 | 쉬운 뜻 |
|---|---|
| Node | 값과 다른 Node로 이어지는 연결을 담은 트리의 한 지점이다. |
| graph | 여러 지점(vertex)과 그 지점을 잇는 연결(edge)로 관계를 표현한 구조다. |
| vertex | graph를 이루는 한 지점이다. 트리에서는 보통 Node라고 부른다. |
| edge | 부모와 자식 Node 사이의 연결선 하나다. |
| subtree | 어떤 Node와 그 아래 모든 후손을 하나의 작은 트리로 본 것이다. |
| cycle | 연결을 따라가다 출발한 Node로 다시 돌아오는 고리다. 일반적인 rooted tree에는 없어야 한다. |
| rooted tree | 여러 Node 중 하나를 root로 정해 위아래 parent-child 방향을 부여한 트리다. |
| 경로(path) | 한 Node에서 다른 Node까지 edge를 따라가는 연결 순서다. |
| 불변식 | 트리 연산 전후에도 반드시 유지해야 하는 규칙이다. "root를 제외한 각 Node는 부모가 하나"가 예다. |

### 🏗️ 내부 구조와 저장 표현 (Structure & Representations)

#### 🧩 구성 요소와 rooted tree의 관계

**정의.** tree의 내부 구성 요소는 다섯 가지로 정리돼. root는 시작 노드, parent는 위 노드, child는 아래 노드, leaf는 자식 없는 끝 노드, edge는 노드 사이 연결이야. 원본 표 그대로 가져왔어. [출처: 01-20-tree.md §4]

| 용어 | 의미 |
|---|---|
| root | 시작 노드 |
| parent | 위 노드 |
| child | 아래 노드 |
| leaf | 자식 없는 끝 노드 |
| edge | 노드 사이 연결 |

**메커니즘.** root를 정하면 각 edge의 위·아래 방향이 생겨. root를 제외한 node는 root로 가는 유일한 경로에서 바로 앞 node 하나를 parent로 갖게 돼. 한 node와 모든 descendant로 이루어진 부분은 subtree야. 그리고 parent-child 관계는 저장 방식이 아니라 논리 관계라서, 구현은 child 목록만 저장할 수도 있고 parent 포인터까지 함께 저장할 수도 있어. [출처: 01-20-tree.md §4]

**예시 — "tree라서 O(log n)"은 틀린 일반화.** Tree는 계층과 유일 경로를 표현하는 구조이고, 검색 성능은 추가 규칙에 달려 있어. 원본 표를 보면 어떤 불변식을 얹느냐에 따라 강해지는 연산이 달라진다는 게 한눈에 보여. [출처: 01-20-tree.md §4]

| tree 종류 | 추가 불변식 | 강해지는 연산 |
|---|---|---|
| 일반 rooted tree | parent-child 계층 | subtree 순회, 계층 표현 |
| Binary Search Tree | 왼쪽 < node < 오른쪽 | key 비교 탐색 |
| 균형 BST | BST + height 제한 | 탐색·삽입·삭제 O(log n) 유지 |
| Heap | 완전 이진 트리 형태 + parent 우선순위 | 최소/최대 head 접근 |
| B-Tree | 다분기 + 점유·균형 규칙 | page I/O를 줄이는 탐색 |

> 😒 **Bailey**: 흥. "트리니까 탐색 O(log n)이죠"라고 답하면 바로 꼬리질문 맞는다? 균형 규칙 없이 빠른 탐색을 기대하면 치우친 높이에서 O(n)이 될 수 있다고 원본 실패 표에 떡하니 적혀 있어. 😠 [출처: 01-20-tree.md §4, §8]

#### 🗂️ 저장 표현 세 가지

**정의.** 같은 논리 tree라도 저장 표현은 세 가지로 나뉘어. child 목록, parent 배열, 인접 리스트 graph야. 각각 장단점이 달라. [출처: 01-20-tree.md §4]

**child 목록** — 아래 방향 순회가 직접적이야. 위로 이동하려면 호출 Stack이나 별도 parent가 필요해. [출처: 01-20-tree.md §4]

```java
class Node<T> {
    T value;
    List<Node<T>> children;
}
```

**parent 배열** — 각 node의 parent를 빠르게 알 수 있지만 child를 찾으려면 역 index가 없을 때 전체를 확인해야 해. [출처: 01-20-tree.md §4]

```text
node:    A  B  C  D
parent: -1  A  A  B
```

**인접 리스트 graph** — 무방향 edge를 양쪽에 저장하므로 DFS에서 직전 parent 또는 visited를 기억해야 되돌아간 edge를 다시 타지 않아. [출처: 01-20-tree.md §4]

```text
A: B, C
B: A, D
C: A
D: B
```

#### ✅ tree 여부를 검증하는 기준

**정의.** 유한 무방향 graph가 tree인지 확인하는 대표 기준은 "모든 node가 연결되어 있고 cycle이 없다"야. [출처: 01-20-tree.md §4]

**메커니즘.** DFS/BFS 중 이미 방문한 node를 parent가 아닌 edge로 다시 만나면 cycle이 있는 거야. 순회가 끝난 후 방문 수가 전체 node 수보다 작으면 연결되지 않은 거고. 이 두 검사를 통과해야 tree라고 말할 수 있어. [출처: 01-20-tree.md §4]

**연산 관점 예시.** tree 자체가 독립 ADT 연산을 정의하지 않는 경우에는 상위 자료구조의 삽입·조회·삭제 과정에서 tree가 사용되는 지점을 기준으로 설명해. `ADT`는 내부 코드를 정하는 말이 아니라 사용할 수 있는 연산과 그 연산이 지켜야 할 규칙을 정한 사용 계약이야. [출처: 01-20-tree.md §5]

#### 🔬 순회 상태 추적

최소 구조와 preorder DFS 추적을 원형 그대로 보존할게. graph 입력으로 tree를 순회한다면 parent 포인터가 없어도 `visited` 또는 이전 node를 기억해 되돌아가는 edge를 다시 방문하지 않아야 해. [출처: 01-20-tree.md §6]

```java
class Node<T> {
    T value;
    List<Node<T>> children = new ArrayList<>();
}
```

```text
        A
      /   \
     B     C
    / \
   D   E
```

preorder DFS라면:

```text
visit A
  visit B
    visit D
    visit E
  visit C

결과: A, B, D, E, C
```

모든 node를 정확히 한 번 방문한다는 불변식을 지키려면 graph 표현에서는 parent나 visited가 필요해. [출처: 01-20-tree.md §6]

### 🧷 불변식과 실패 상황 (Invariants & Failures)

정의에서 빠지면 안 되는 두 조건은 이거야. [출처: 01-20-tree.md §7]

- connected: 어떤 두 node 사이에도 경로가 있다.
- acyclic: 자기 자신으로 돌아오는 cycle이 없다.

node가 `n`개인 유한 tree의 edge는 `n-1`개이고, 임의의 두 node 사이에는 단순 경로가 정확히 하나야. edge 하나를 끊으면 두 component로 나뉘고, cycle을 만들지 않는 새 edge 하나로 다시 연결할 수 있어. [출처: 01-20-tree.md §7]

실패 상황은 원본 표 그대로 기억해 두자. [출처: 01-20-tree.md §8]

| 실패 | 증상 |
|---|---|
| 조상을 child로 다시 연결 | cycle, 재귀 순회 무한 반복 가능 |
| 한 node를 두 parent에 연결 | root 경로 유일성 붕괴, DAG 형태 |
| root 참조 유실 | 외부 참조가 없다면 전체 tree에 접근 불가 |
| parent/children 양방향 불일치 | 위·아래 순회 결과가 달라짐 |
| 균형 규칙 없이 빠른 탐색 기대 | 치우친 높이에서 O(n) 가능 |

### 📊 성능·비용과 선택 기준 (Cost & Selection)

모든 node를 한 번 방문하는 순회는 O(n)이야. 재귀 DFS의 추가 stack은 최악 O(height)이고, level-order의 Queue는 한 level의 최대 너비에 비례해. 탐색이 빠른지는 tree라는 사실만으로 결정되지 않고 정렬·균형 같은 추가 불변식에 달려 있어. [출처: 01-20-tree.md §9]

선택 기준은 이 개념만 떼어 정하지 않아. 필요한 조회·삽입·삭제·정렬·범위·순회 연산을 기준으로 인접 개념과 상위 자료구조를 비교해. [출처: 01-20-tree.md §10] 근거는 §0 연결 표준·공식 자료 기준으로 말하고, Java API 동작은 Oracle Java 17 문서 명시 범위에서만 보장으로 말해. 실무에서는 구현체의 API 계약, 입력 크기, 데이터 분포, 변경 빈도와 동시 접근 조건을 함께 확인하고. [출처: 01-20-tree.md §11, §12]

#### Q1. 트리와 그래프 차이는?

<details><summary>답 보기</summary>

트리는 계층 구조를 가진 특수한 형태로 볼 수 있고, 일반적으로 root와 부모-자식 관계를 중심으로 설명한다. 그래프는 더 일반적인 노드-간선 구조다. [출처: 01-20-tree.md §14 Q1]

</details>

#### Q2. DAG는 cycle이 없으니 tree인가?

<details><summary>답 보기</summary>

아니다. DAG의 한 node는 여러 parent 경로를 가질 수 있고 임의 두 node 사이 경로가 유일하지 않을 수 있다. tree는 연결성과 유일 경로 조건까지 만족해야 한다. [출처: 01-20-tree.md §14 Q2]

</details>

#### Q3. node n개인 tree의 edge가 n-1보다 적으면?

<details><summary>답 보기</summary>

모든 node를 연결할 수 없다. n개의 독립 node를 하나의 connected component로 만들려면 최소 n-1개의 edge가 필요하다. [출처: 01-20-tree.md §14 Q3]

</details>

#### Q4. edge가 n-1개면 무조건 tree인가?

<details><summary>답 보기</summary>

연결 조건도 확인해야 한다. 전체 graph가 connected이고 edge가 n-1개라면 tree로 볼 수 있다. 연결되지 않은 graph는 한 component에 cycle이 있고 다른 component가 분리된 형태도 가능하다. [출처: 01-20-tree.md §14 Q4]

</details>

#### Q5. subtree도 tree인가?

<details><summary>답 보기</summary>

rooted tree에서 어떤 node와 모든 descendant를 모으면 그 node를 root로 하는 tree다. 원래 tree의 parent 방향 edge는 subtree 범위에서 제외된다. [출처: 01-20-tree.md §14 Q5]

</details>

#### Q6. Tree 순회에 visited가 항상 필요한가?

<details><summary>답 보기</summary>

child 방향만 저장된 올바른 rooted tree라면 cycle이 없어 필수는 아니다. 무방향 인접 리스트로 저장했거나 입력이 정말 tree인지 신뢰할 수 없다면 parent/visited가 필요하다. [출처: 01-20-tree.md §14 Q6]

</details>

### 📌 요약 (Summary)

- tree는 connected + acyclic graph이고, root를 정하면 rooted tree가 되어 parent-child 계층이 생긴다. [출처: 01-20-tree.md §1]
- node n개인 유한 tree의 edge는 n-1개, 임의 두 node 사이 단순 경로는 정확히 하나다. [출처: 01-20-tree.md §7]
- "tree라서 검색이 O(log n)"은 틀린 일반화다. 검색 성능은 BST·균형·heap 같은 추가 불변식이 결정한다. [출처: 01-20-tree.md §4]
- 저장 표현은 child 목록 / parent 배열 / 인접 리스트 세 가지이고, 무방향 표현에서는 parent 또는 visited 기억이 필요하다. [출처: 01-20-tree.md §4, §6]
- 순회는 O(n), 재귀 stack은 최악 O(height), level-order Queue는 한 level 최대 너비에 비례한다. [출처: 01-20-tree.md §9]

## 사실 (F)

F1. graph 이론에서 tree는 모든 vertex가 연결되어 있고 cycle이 없는 graph이며, root를 정하면 rooted tree가 된다. [출처: NIST DADS Tree (원본 01-20-tree.md §0·§1 재인용)]
F2. node n개인 유한 tree의 edge는 n-1개이고 임의 두 node 사이 단순 경로는 정확히 하나다. [출처: 01-20-tree.md §7]
F3. tree의 검색 성능은 tree라는 사실이 아니라 BST 정렬·균형·heap 우선순위 같은 추가 불변식이 결정한다. [출처: 01-20-tree.md §4, §9]
F4. 저장 표현은 child 목록, parent 배열, 인접 리스트 graph 세 가지가 있고 각각 위/아래 방향 이동 비용이 다르다. [출처: 01-20-tree.md §4]
F5. 무방향 graph 입력에서 tree 검증은 "연결 + 무cycle" 확인이며, DFS/BFS에서 parent가 아닌 edge로 방문 node를 다시 만나면 cycle이다. [출처: 01-20-tree.md §4]
F6. 전체 순회는 O(n), 재귀 DFS 추가 stack은 최악 O(height)다. [출처: 01-20-tree.md §9]

U3. tree 개념의 역사적 기원은 원본에 없어 다루지 않았다.
