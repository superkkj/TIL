### [열쇠를 숫자 지문으로] 해시 (Hash)

> 🔑 핵심 키워드: `hash function` · `결정적(deterministic)` · `분산성` · `비트 섞기` · `bucket index` · `비둘기집 원리` · `암호학적 hash`

웨이드님~ 이번엔 "key를 숫자로 바꾸는 과정" — hash 그 자체야! 😊 앞 문서에서 `key -> hash -> index` 흐름을 봤다면, 이 문서는 그 첫 화살표를 현미경으로 보는 시간이야.

***

### 💠 핵심 원리 (Core Principles & Concepts)

정의부터. hash는 입력 값을 해시 테이블의 위치 계산 등에 쓰기 쉬운 값으로 변환한 결과야. Java HashMap에서는 key의 `hashCode()`와 내부 보정 `hash()`가 bucket index 계산에 사용돼. [출처: 01-04-hash.md §1 정석 설명]

쉬운 비유로는 — 긴 이름표를 숫자 지문처럼 바꾼 거야. 이 숫자 지문으로 사물함 위치 후보를 계산하는 거지. 🤓 [출처: 01-04-hash.md §1 쉬운 설명]

왜 필요할까? 문자열, 객체, 이메일 같은 key를 배열 index로 바로 쓸 수 없기 때문이야. hash는 다양한 key를 배열 위치 계산이 가능한 숫자 세계로 옮겨줘. [출처: 01-04-hash.md §2 왜 필요한가]

범위 주의. 이 문서가 다루는 건 hash의 정의와 상위 자료구조에서의 역할이야. 정렬·동시성·영속성·특정 시간 복잡도는 별도 불변식이나 구현 계약이 있을 때만 보장되고, 개념 이름만 보고 이런 성질까지 자동으로 있다고 단정하면 안 돼. 이 성질들은 사용하는 구현체의 공식 설명에서 각각 확인해야 해. [출처: 01-04-hash.md §3 해결 범위와 비목표]

용어 표는 그대로 보존! 😊

| 용어 | 쉬운 뜻 |
|---|---|
| hash | 원래 key를 계산하기 쉬운 정수 값으로 바꾼 결과다. 원본 그 자체나 배열 위치는 아니다. |
| hash function | key를 넣으면 hash를 돌려주는 계산 규칙이다. |
| 결정적(deterministic) | 같은 실행 조건에서 같은 key를 넣으면 같은 결과가 나오는 성질이다. |
| 비트 섞기 | hash의 위쪽 비트 정보도 아래쪽 비트에 반영되도록 섞어 bucket 쏠림을 줄이는 계산이다. |
| bucket index | hash를 현재 table 크기에 맞는 실제 칸 번호로 줄인 최종 위치 후보다. |

[출처: 01-04-hash.md §1 어려운 말 먼저 풀기]

쉬운 설명은 비유고, 면접에서는 정석 설명의 용어와 전제 조건으로 돌아와 답하는 것! [출처: 01-04-hash.md §1 다시 정리]

### 🧮 hash의 내부 여정 (From Key to Index)

#### 🔀 세 단계 변환과 좋은 hash의 성질

① 정의. 입력에서 bucket까지는 세 단계야. 원본 key가 hash function을 지나 hash value가 되고, compression(압축)을 거쳐 bucket index가 돼. Java `HashMap`에서는 key의 `hashCode()`가 첫 정수 값을 만들고, OpenJDK 구현의 내부 `hash()`가 비트를 섞은 뒤, table 길이에 맞춰 index를 계산해. **`hashCode`, 내부 hash, bucket index는 같은 값이 아니야.** [출처: 01-04-hash.md §4 입력에서 bucket까지의 세 단계]

```text
원본 key -> hash function -> hash value -> compression -> bucket index
```

[출처: 01-04-hash.md §4]

② 메커니즘 — 좋은 비암호학적 hash의 성질 네 가지. 결정성(같은 상태의 같은 key는 같은 값), 분산성(실제 입력 집합을 bucket 전체에 가능한 한 고르게), 계산 비용(hash 계산이 조회보다 지나치게 비싸지 않게), 그리고 작은 입력 차이가 특정 bucket에 반복적으로 몰리지 않게 하는 것. 주의 — "충돌이 전혀 없다"는 조건은 여기 없어. 유한한 출력 공간에 더 많은 입력을 매핑하면 비둘기집 원리에 의해 충돌은 피할 수 없기 때문이야. [출처: 01-04-hash.md §4 좋은 비암호학적 hash의 성질]

③ 예시 — OpenJDK의 비트 섞기. 실제 코드는 이렇게 생겼어. [출처: OpenJDK 17u HashMap.java (원본 01-04-hash.md §4 재인용)]

```java
static int hash(Object key) {
    int h;
    return key == null ? 0 : (h = key.hashCode()) ^ (h >>> 16);
}
```

최종 index 계산이 table 길이에 따라 hash의 일부 낮은 비트를 사용하니까, 높은 비트도 낮은 쪽 계산에 참여시키려는 구현이야. 이 식은 Java API 보장이 아니라 OpenJDK 17u 구현 세부야. [출처: 01-04-hash.md §4 OpenJDK 내부 비트 섞기]

#### 🔐 암호학적 hash와의 구분

① 정의. HashMap용 hash는 빠른 bucket 분산이 목적이야. SHA-256 같은 암호학적 hash는 역상·충돌 공격에 대한 보안 성질이 목적이야. 목적이 다른 두 도구인 거지. [출처: 01-04-hash.md §4 암호학적 hash와 구분]

② 메커니즘. 둘 다 입력을 고정 크기 값으로 바꾼다는 겉모습은 같아. 하지만 요구사항과 비용이 달라. HashMap 쪽은 계산이 싸고 분산이 좋으면 충분하고, 암호학 쪽은 공격 저항이라는 훨씬 무거운 성질을 요구해. 그래서 "hash"라는 단어가 나오면 어느 쪽 hash인지 먼저 구분하고 답해야 해 — 면접에서 이 구분 자체가 채점 포인트야. [출처: 01-04-hash.md §4]

③ 예시. 비유하자면 HashMap의 hash는 우편번호(빠른 분류용)고, 암호학적 hash는 위조 방지 봉인(변조 검증용)이야. 우편번호로 봉인을 대신할 수 없고, 봉인 검사로 우편 분류를 하면 비용이 과해. 도구는 목적에 맞게! 😊

#### 🧵 String hashCode 계산과 손 추적

① 정의. Java 17 `String.hashCode()` 계약은 문자열 문자값을 31을 사용한 다항식으로 계산해. 개념적으로 왼쪽부터 `h = 31 * h + 현재 문자 값`으로 누적하는 방식이야. [출처: Oracle Java 17 문서 String.hashCode 계약 (원본 01-04-hash.md §5 재인용)]

```text
h = 0
h = 31 * h + 현재 문자 값
```

② 메커니즘 — `"ABC"`를 손으로 계산해보자. [출처: 01-04-hash.md §5 String hashCode 계산]

```text
'A'=65: 0 * 31 + 65 = 65
'B'=66: 65 * 31 + 66 = 2081
'C'=67: 2081 * 31 + 67 = 64578
```

그런데 서로 다른 문자열도 같은 결과를 만들 수 있어. 이게 hash 충돌의 실물 증거야.

```text
"Aa" = 65 * 31 + 97 = 2112
"BB" = 66 * 31 + 66 = 2112

hashCode 같음
equals false
```

[출처: 01-04-hash.md §5]

③ 예시 — index까지 손 추적(§6 상태 추적). hash 35, capacity 16이면 mask는 15고, `35 & 15 = 3`이라서 bucket index는 3이야. 여기서 `35`는 hash고 `3`은 bucket index — capacity가 바뀌면 같은 hash도 다른 index로 갈 수 있어. Java key의 전체 여정은 세 단계로 정리돼. [출처: 01-04-hash.md §6 손 추적, Java key가 bucket에 도착하는 세 단계]

```text
hash = 35
capacity = 16
mask = 15
index = 35 & 15 = 3
```

```text
key
 ↓ key.hashCode()
객체가 계산한 int
 ↓ h ^ (h >>> 16)
HashMap 내부에서 비트를 섞은 hash
 ↓ hash & (table.length - 1)
현재 table의 bucket index
```

`hashCode`, 내부 hash, bucket index는 같은 값이 아니라는 것, 세 번째 강조! [출처: 01-04-hash.md §6]

> 😒 **Bailey**: 흥. "Aa"랑 "BB"가 hashCode 2112로 같다는 거, 계산기 두드려서 직접 확인해봤어? 65×31+97=2112, 66×31+66=2112. 원본이 굳이 숫자까지 적어준 이유는 "hashCode 같음 ≠ 같은 key"를 몸으로 기억하라는 거야. equals가 최종 판정이라고. [출처: 01-04-hash.md §5]

### 🧷 불변식과 경계 (Invariants & Boundaries)

결정성과 불변 key. 저장 때와 조회 때 같은 key 상태가 같은 hash를 만들어야 같은 시작 위치로 갈 수 있어. key의 동등성 관련 필드를 저장 후 변경하면 다른 hash가 나와서 기존 entry를 못 찾을 수 있어. 이게 hash 세계의 핵심 불변식이야. [출처: 01-04-hash.md §7 결정성과 불변 key]

경계 주의사항 하나 더. 일반적인 hash는 원본의 대소 관계를 보존하도록 설계하지 않아. 그래서 hash 값의 대소만 보고 원본 key의 대소나 인접 관계를 판단할 수 없어. [출처: 01-04-hash.md §8 주의]

### 🗺️ 비용과 선택의 자리 (Cost & Positioning)

hash 자체가 독립 연산을 모두 정의하는 건 아니야. 접근 비용은 배열·Node·Tree 등 실제 저장 표현과 상위 자료구조의 연산 계약을 기준으로 판단해. [출처: 01-04-hash.md §9 성능과 비용]

대안 비교도 마찬가지 — 이 개념만 떼어 자료구조를 선택하지 않아. 필요한 조회·삽입·삭제·정렬·범위·순회 연산을 기준으로 인접 개념과 상위 자료구조를 비교해. [출처: 01-04-hash.md §10]

근거를 말하는 기준: 일반 자료구조·알고리즘 성질은 표준·공식 자료(NIST 등) 기준, Java API 동작은 Oracle Java 17 문서가 명시한 범위에서만 보장, OpenJDK 내부 필드·상수·계산식은 소스가 연결된 경우에만 구현 세부로 구분. [출처: 01-04-hash.md §0, §11]

실무에서는 개념 이름보다 사용하는 구현체의 API 계약, 입력 크기, 데이터 분포, 변경 빈도와 동시 접근 조건을 함께 확인해. [출처: 01-04-hash.md §12]

### Q1. hash만 같으면 같은 key인가?

<details><summary>답 보기</summary>

아니다. 서로 다른 key도 같은 hash를 가질 수 있다. hash는 위치 후보를 찾는 값이고, 진짜 같은 key인지는 equals로 확인해야 한다. [출처: 01-04-hash.md §14 Q1]

</details>

### Q2. hash는 원본을 압축한 값인데 복원할 수 있나?

<details><summary>답 보기</summary>

일반적으로 hash만으로 원본을 유일하게 복원할 수 없다. 여러 입력이 같은 출력으로 갈 수 있기 때문이다. HashMap이 실제 entry에 원본 key도 보관하는 이유 중 하나다. [출처: 01-04-hash.md §14 Q2]

</details>

### Q3. 좋은 hash면 충돌이 전혀 없는가?

<details><summary>답 보기</summary>

아니다. 일반적인 큰 입력 공간을 유한한 출력에 매핑하면 충돌은 가능하다. 좋은 hash는 실제 입력을 충분히 고르게 분산해 특정 bucket 집중을 줄이는 것이 핵심이다. [출처: 01-04-hash.md §14 Q3]

</details>

### Q4. hash 값의 대소로 원본 key를 정렬할 수 있는가?

<details><summary>답 보기</summary>

일반 hash는 원본 대소 관계를 보존하지 않는다. 범위·정렬이 필요하면 comparator 기반 tree나 별도 정렬 index를 사용한다. [출처: 01-04-hash.md §14 Q4]

</details>

### Q5. HashMap용 hash와 SHA-256을 같은 목적으로 쓰는가?

<details><summary>답 보기</summary>

아니다. HashMap용 hash는 빠른 bucket 분산이 목적이고 암호학적 hash는 역상·충돌 공격 저항 같은 보안 성질이 중요하다. [출처: 01-04-hash.md §14 Q5]

</details>

### 📝 요약 (Summary)

- hash = key를 위치 계산에 쓰기 쉬운 값으로 바꾼 결과. 원본도 아니고 배열 위치도 아니다. [출처: 01-04-hash.md §1]
- 세 단계: key → hash function → hash value → compression → bucket index. hashCode·내부 hash·index는 서로 다른 값. [출처: §4, §6]
- 좋은 비암호학적 hash: 결정성 + 분산성 + 낮은 계산 비용 + 쏠림 방지. 충돌 제로는 목표가 아니다(비둘기집 원리). [출처: §4]
- OpenJDK 내부 `h ^ (h >>> 16)`은 API 보장이 아닌 구현 세부. 낮은 비트만 쓰는 index 계산에 높은 비트를 참여시키는 장치. [출처: §4]
- String.hashCode는 31 다항식. "Aa"와 "BB"는 둘 다 2112 — hashCode 같아도 equals false 가능. [출처: §5]
- 결정성 불변식: 저장과 조회의 key 상태가 같아야 같은 시작점. 저장 후 동등성 필드 변경 금지. [출처: §7]
- hash 대소는 원본 대소와 무관. 정렬·범위는 다른 구조의 몫. [출처: §8]

## 사실 (F)

F1. hash는 입력 값을 해시 테이블의 위치 계산 등에 쓰기 쉬운 값으로 변환한 결과이며, Java HashMap에서는 key의 hashCode()와 내부 보정 hash()가 bucket index 계산에 사용된다. [출처: 01-04-hash.md §1]
F2. Java 17 String.hashCode() 계약은 문자열 문자값을 31을 사용한 다항식으로 계산한다. [출처: Oracle Java 17 문서 (원본 01-04-hash.md §5 재인용)]
F3. "Aa"(65×31+97)와 "BB"(66×31+66)는 hashCode가 2112로 같지만 equals는 false다. [출처: 01-04-hash.md §5]
F4. OpenJDK 17u 내부 hash는 `key == null ? 0 : (h = key.hashCode()) ^ (h >>> 16)`이며, 이 식은 Java API 보장이 아니라 구현 세부다. [출처: OpenJDK 17u HashMap.java (원본 01-04-hash.md §4 재인용)]
F5. 유한한 출력 공간에 더 많은 입력을 매핑하면 비둘기집 원리에 의해 충돌을 피할 수 없다. [출처: 01-04-hash.md §4]
F6. 일반 hash는 원본의 대소 관계를 보존하도록 설계되지 않으므로 hash 대소로 원본 key의 대소·인접 관계를 판단할 수 없다. [출처: 01-04-hash.md §8]
F7. hash 35, capacity 16(mask 15)일 때 bucket index는 35 & 15 = 3이며, capacity가 바뀌면 같은 hash도 다른 index로 갈 수 있다. [출처: 01-04-hash.md §6]

## 모름 (U)

U1. 좋은 hash function을 설계하는 구체 알고리즘 — 원본은 성질(결정성·분산성·비용·쏠림 방지)만 나열했고 설계법은 제시하지 않았다.
U2. String.hashCode에 31이라는 상수가 선택된 이유 — 원본에 근거 없음.
U3. SHA-256 등 암호학적 hash의 내부 구조와 보안 성질의 정량 기준 — 원본은 목적 구분(분산 vs 보안)까지만 서술했다.
U4. HashMap 외 다른 언어/라이브러리 해시 구현의 비트 섞기 방식 — 원본 범위 밖.
