### [처음부터 끝까지, 딱 한 바퀴의 값] O(n) 선형 시간 (Linear Time)
> 🔑 핵심 키워드: linear time, full scan, early exit, coefficient, nested loop, two-pointer, Ω(n) 하한

웨이드님~ 복잡도 3부작 세 번째, O(n)이야. 😊 "처음부터 끝까지 하나씩 확인 — 데이터 10배면 일도 대략 10배"라는 정직한 비용이지. 근데 loop 모양만 보고 O(n)을 판정하다가 틀리는 함정이 은근히 많아. 오늘 그걸 다 걷어내자.

***

### 📚 핵심 원리 (Core Principles & Concepts)

O(n)은 입력 크기 `n`에 비례해 비용이 증가하는 성장률이야. 쉬운 설명으로는 처음부터 끝까지 하나씩 확인하는 방식이고, 데이터가 10배 늘면 할 일도 대략 10배 늘어나. 표준 근거는 NIST DADS의 Big-O Notation 항목이야. [출처: 01-42-o-n.md §1, §0(NIST DADS Big-O Notation 재인용)]

왜 중요하냐면, O(n)은 작은 데이터에서는 괜찮지만 대량 데이터에서 반복 호출되면 병목이 될 수 있기 때문이야. [출처: 01-42-o-n.md §2] 문서 범위는 O(n)의 정의와 상위 자료구조에서의 역할까지고, 정렬·동시성·영속성·특정 시간 복잡도는 별도 불변식이나 구현 계약이 있을 때만 보장돼. [출처: 01-42-o-n.md §3]

용어 표 원형 보존! 🤓 [출처: 01-42-o-n.md §1]

| 용어 | 쉬운 뜻 |
|---|---|
| linear time | 입력이 `n`개 늘어날 때 확인하거나 처리할 일도 `n`에 비례해 늘어나는 형태다. |
| full scan | 처음부터 끝까지 모든 원소를 한 번씩 확인하는 작업이다. |
| early exit | 원하는 값을 중간에 찾으면 순회를 일찍 끝내는 것이다. 일부 입력은 빨라도 최악에는 끝까지 볼 수 있다. |
| coefficient | `3n`의 3처럼 `n` 앞에 붙는 고정 배수다. Big-O에서는 증가 차수가 같으면 제거한다. |
| nested loop | 반복문 안에 반복문이 있는 구조다. 항상 O(n²)은 아니며 각 반복 횟수가 입력과 어떻게 연결되는지 계산해야 한다. |

쉬운 설명은 비유고, 면접에서는 정석 용어와 전제 조건으로 돌아와 설명하자. [출처: 01-42-o-n.md §1 다시 정리]

### 🔎 내부 구조와 핵심 연산

#### 🧵 선형이 생기는 두 형태

대표 예. [출처: 01-42-o-n.md §4]

```text
배열 전체 순회
리스트에서 특정 값 찾기
HashMap에서 value 조건 검색
트리의 모든 노드 방문
```

선형이 생기는 형태는 두 가지야. [출처: 01-42-o-n.md §4]

1. 결과를 만들기 위해 모든 입력을 반드시 한 번 보는 경우
2. 더 좋은 탐색 기준이 없어 target을 찾을 때까지 순차 비교하는 경우

첫 번째는 합계·최댓값처럼 입력 전체가 결과에 영향을 주므로 O(n)이 최적일 수 있어. 두 번째는 정렬, hash, index 같은 추가 구조로 개선 가능할 수 있지만 전처리·메모리·갱신 비용을 함께 지불해. [출처: 01-42-o-n.md §4]

O(n)이라는 개념 자체가 독립 ADT 연산을 정의하지 않는 경우에는 상위 자료구조의 삽입·조회·삭제 과정에서 이 개념이 쓰이는 지점을 기준으로 설명하고, 상태 추적은 예시를 입력부터 결과까지 따라가며 실제로 변하는 상태를 확인하는 방식으로 연습해. [출처: 01-42-o-n.md §5, §6]

#### ✍️ 연산 수를 직접 세기

[출처: 01-42-o-n.md §9]

```java
int max = values[0];
for (int i = 1; i < values.length; i++) {
    if (values[i] > max) max = values[i];
}
```

길이 n에서 비교는 n-1번이야.

```text
T(n) = a(n-1) + b
     = an + (b-a)
     = O(n)
```

상수와 낮은 차수 항을 제거해도 모든 원소를 확인한다는 핵심은 남아. [출처: 01-42-o-n.md §9]

#### 🪤 숨은 중첩 비용과 two-pointer

겉으로 loop가 하나여도 내부 연산이 O(n)이면 전체는 O(n^2)이 될 수 있어. [출처: 01-42-o-n.md §9]

```java
for (String x : list) {       // n번
    otherList.contains(x);    // LinkedList라면 최대 n번
}
```

반대로 중첩 loop라도 전체 pointer가 각각 한 번만 전진하는 two-pointer 방식은 O(n)일 수 있어. 문법 모양이 아니라 총 연산 횟수를 세는 거야. [출처: 01-42-o-n.md §9]

```java
int left = 0;
int right = 0;
while (right < values.length) {
    right++;
    while (mustShrink(left, right)) {
        left++;
    }
}
```

left와 right가 각각 최대 n번만 증가하고 뒤로 가지 않는다면 총 이동은 2n 수준이어서 O(n)일 수 있어. 각 원소가 몇 번 처리되는지 합으로 세는 거지. [출처: 01-42-o-n.md §9]

> 😒 **Bailey**: 흥. "loop 한 개니까 O(n)이요, 두 개니까 O(n²)이요" — 그건 문법 감상문이지 복잡도 분석이 아니야. contains 한 줄에 숨은 n을 못 보면 위 코드 두 개의 판정이 전부 뒤집힌다고. 😠 [출처: 01-42-o-n.md §9]

### 🚨 불변식과 실패·경계 상황

불변식은 삽입·조회·삭제 전후에도 계속 참이어야 하는 규칙이고, 정상 상태에서 유지해야 할 구체 조건은 내부 구조와 연산 설명을 기준으로 확인해. [출처: 01-42-o-n.md §7]

실패·경계는 경계값(첫 index, 마지막 index, 0개처럼 조건이 바뀌는 가장자리 값)과 빈 구조를 먼저 확인하고, 불변식이 깨졌을 때 발생하는 예외와 예외 없이 잘못되는 결과를 구분해서 봐. [출처: 01-42-o-n.md §8]

조기 종료와 최악 비용도 여기서 짚자. 선형 검색은 첫 원소가 답이면 O(1), 마지막 또는 부재면 O(n)이야. 입력 분포를 모르면 최악 O(n)이라고 말하고, 평균을 주장하려면 target 위치와 부재 확률 같은 분포 전제가 필요해. [출처: 01-42-o-n.md §9]

### 🏗️ 성능·비용과 대안 비교

#### O(n)이 최적인 경우

정렬되지 않은 n개 값의 최댓값을 정확히 구하려면 확인하지 않은 값이 더 클 수 있어. 따라서 일반 비교 모델에서는 모든 값을 한 번은 봐야 해 Ω(n) 하한이 있고, 위 최댓값 알고리즘은 O(n)이므로 Θ(n)이야. [출처: 01-42-o-n.md §9]

#### 전처리로 조회를 줄이는 trade-off

[출처: 01-42-o-n.md §9]

```text
매번 선형 검색
  준비 비용 없음
  조회 O(n)

HashSet index 구축
  구축 O(n), 추가 공간 O(n)
  이후 membership 기대 O(1)
  데이터 변경 때 index도 갱신
```

조회 한 번이면 전처리가 손해일 수 있고, 같은 데이터에 조회가 반복되면 이득일 수 있어. [출처: 01-42-o-n.md §9]

대안 선택은 이 개념만 떼어 하지 않고 필요한 조회·삽입·삭제·정렬·범위·순회 연산 기준으로 비교해. [출처: 01-42-o-n.md §10] 근거 기준(표준 자료 / Oracle Java 17 문서 / OpenJDK 소스 연결 시 구현 세부)과 실무 판단 기준(API 계약, 입력 크기, 데이터 분포, 변경 빈도, 동시 접근)도 원본 그대로. [출처: 01-42-o-n.md §11, §12]

### Q1. O(n)은 항상 나쁜가?

<details><summary>답 보기</summary>

아니다. 모든 데이터를 반드시 봐야 하는 문제라면 O(n)이 자연스럽다. 문제는 O(1)이나 O(log n)으로 풀 수 있는 요구를 불필요하게 O(n)으로 처리하는 경우다. [출처: 01-42-o-n.md §14 Q1]

</details>

### Q2. HashMap의 모든 value를 조건 검색하면 왜 O(n)인가?

<details><summary>답 보기</summary>

HashMap은 key를 bucket으로 보내는 index만 갖고 value 조건의 위치 정보는 없다. 어떤 entry의 value가 조건을 만족하는지 알려면 최악에는 모든 mapping을 확인해야 한다. [출처: 01-42-o-n.md §14 Q2]

</details>

### Q3. 두 배열을 각각 순회하면 O(n^2)인가?

<details><summary>답 보기</summary>

순차적으로 길이 n 배열 두 개를 순회하면 O(n+n)=O(n)이다. 한 배열의 각 원소마다 다른 배열 전체를 순회해야 O(n^2)이다. [출처: 01-42-o-n.md §14 Q3]

</details>

### Q4. 길이가 다른 두 입력 순회 비용은?

<details><summary>답 보기</summary>

각 길이를 n, m이라고 하면 O(n+m)이다. 근거 없이 둘 다 n으로 놓으면 입력 관계를 숨길 수 있다. [출처: 01-42-o-n.md §14 Q4]

</details>

### Q5. 병렬로 n개를 처리하면 시간 복잡도가 O(1)이 되는가?

<details><summary>답 보기</summary>

무한 processor를 전제하지 않는다. 총 work는 여전히 O(n)일 수 있고 실제 wall-clock은 worker 수, 분할, 동기화, I/O에 따라 달라진다. work와 병렬 실행 시간을 구분한다. [출처: 01-42-o-n.md §14 Q5]

</details>

### 📝 요약 (Summary)

- O(n) = 입력 크기에 비례해 비용이 늘어나는 선형 시간. full scan과 순차 탐색이 대표 예. [출처: 01-42-o-n.md §1, §13]
- 선형의 두 형태: 전 입력이 결과에 필요한 경우(개선 불가 후보)와 더 좋은 탐색 기준이 없는 경우(전처리로 개선 후보). [출처: 01-42-o-n.md §4]
- 조기 종료는 best만 바꾼다. 분포를 모르면 최악 O(n)으로 말한다. [출처: 01-42-o-n.md §9]
- loop 개수가 아니라 총 연산 횟수를 센다 — 단일 loop 안 contains는 O(n^2), two-pointer 중첩 loop는 O(n)일 수 있다. [출처: 01-42-o-n.md §9]
- 정렬 안 된 n개의 최댓값은 Ω(n) 하한이 있어 순회 알고리즘이 Θ(n)이다. [출처: 01-42-o-n.md §9]
- HashSet index 구축 trade-off: 구축 O(n)·공간 O(n)·이후 조회 기대 O(1)·변경 시 갱신 비용. [출처: 01-42-o-n.md §9]


## 사실 (F)
F1. O(n)은 입력 크기 n에 비례해 비용이 증가하는 성장률이다. [출처: 01-42-o-n.md §1; NIST DADS Big-O Notation (원본 §0 재인용)]
F2. 길이 n 배열의 최댓값 순회는 비교 n-1번, T(n) = a(n-1)+b = O(n)이다. [출처: 01-42-o-n.md §9]
F3. 정렬되지 않은 n개 값의 최댓값 문제는 일반 비교 모델에서 Ω(n) 하한이 있고, 순회 알고리즘은 Θ(n)이다. [출처: 01-42-o-n.md §9]
F4. 단일 loop 안의 LinkedList contains 호출은 전체 O(n^2)이 될 수 있고, two-pointer 중첩 loop는 포인터가 각각 최대 n번만 전진하면 O(n)일 수 있다. [출처: 01-42-o-n.md §9]
F5. 선형 검색은 첫 원소가 답이면 O(1), 마지막 또는 부재면 O(n)이며, 분포를 모르면 최악 O(n)으로 말한다. [출처: 01-42-o-n.md §9]
F6. HashMap value 조건 검색이 O(n)인 이유는 key→bucket index만 있고 value 위치 정보가 없어 최악에는 모든 mapping을 확인해야 하기 때문이다. [출처: 01-42-o-n.md §14 Q2]
F7. 서로 다른 길이의 두 입력 순회는 O(n+m)으로 표기하며, 둘 다 n으로 합치면 입력 관계를 숨긴다. [출처: 01-42-o-n.md §14 Q4]

## 모름 (U)
