### 🧭 핵심 원리 (Core Principles & Concepts)

자료구조는 데이터를 저장하고 접근하는 방식을 정한 구조야. 같은 데이터라도 배열, 리스트, 해시 테이블, 트리, 힙 중 어디에 놓느냐에 따라 조회, 삽입, 삭제, 순회, 정렬, 범위 검색 비용이 달라져. [출처: 01-01-data-structure.md §1 정의]

비유로 감을 잡아볼까? 🤓 책을 아무 상자에나 넣으면 넣기는 쉽지만 찾기가 어렵고, 가나다순 책장에 꽂으면 찾기는 쉽지만 중간에 새 책을 넣을 때 자리를 조정해야 해. 자료구조는 "어떤 일을 쉽게 만들기 위해 데이터를 어떻게 놓을 것인가"의 선택이야. [출처: 01-01-data-structure.md §1 쉬운 설명]

왜 필요할까? 데이터가 적을 때는 아무 구조나 써도 차이가 작지만, 데이터가 많아지거나 요청이 많아지면 핵심 연산 비용이 서비스 성능을 결정하기 때문이야. [출처: 01-01-data-structure.md §2 왜 필요한가] 컴퓨터 메모리는 결국 바이트의 연속인데, 프로그램이 원하는 연산은 서로 달라. `i`번째 값을 즉시 읽고 싶은 경우, key로 값을 찾는 경우, 가장 작은 값을 계속 꺼내는 경우를 같은 배치로 모두 최적화할 수는 없어. 그래서 자료구조는 데이터의 **논리적 관계**, 메모리의 **물리적 배치**, 외부에 제공하는 **연산 계약**을 묶은 선택이야. [출처: 01-01-data-structure.md §2 문제의 뿌리]

```text
요구사항 -> 빠르게 해야 할 연산 -> 필요한 불변식 -> 자료구조 선택
```

[출처: 01-01-data-structure.md §2]

범위도 분명히 하자. 이 문서가 다루는 건 자료구조의 정의와 상위 자료구조에서의 역할이고, 정렬·동시성·영속성·특정 시간 복잡도는 별도 불변식이나 구현 계약이 있을 때만 보장돼. 개념 이름만 보고 이런 성질까지 자동으로 있다고 단정하면 안 돼. `동시성`은 여러 실행 흐름이 함께 접근해도 안전한지, `영속성`은 프로그램이 끝난 뒤에도 파일이나 DB 등에 데이터가 남는지, `구현 계약`은 실제 클래스나 API가 무엇을 약속하는지를 뜻하고, 이 성질들은 사용하는 구현체의 공식 설명에서 각각 확인해야 해. [출처: 01-01-data-structure.md §3 해결 범위와 비목표]

> 😒 **Bailey**: 흥. "Map이니까 thread-safe겠지" 하고 이름만 보고 단정하는 사람 꼭 있더라? 동시성·영속성·복잡도는 전부 구현체 공식 문서에서 따로 확인할 일이야. 원본 §3이 콕 집어 경고한 부분이라고. [출처: 01-01-data-structure.md §3]

용어부터 정리하고 가자. 원본의 "어려운 말 먼저 풀기" 표는 면접 대비 핵심이니까 그대로 보존할게. 😊

| 용어 | 쉬운 뜻 |
|---|---|
| 원소(element) | 구조 안에 저장하는 데이터 한 개다. 배열의 한 칸에 든 값이 원소다. |
| 연산(operation) | 데이터를 넣기, 찾기, 지우기처럼 자료구조에 시키는 일이다. |
| 불변식(invariant) | 연산 전후에도 반드시 지켜져야 하는 규칙이다. 예를 들어 Stack은 중간이 아니라 맨 위에서 꺼낸다. |
| ADT(추상 자료형) | 내부 코드보다 "무슨 일을 어떤 규칙으로 제공하는가"를 정의한 사용 설명서다. |
| 구현체 | ADT의 규칙을 실제 배열, Node, 코드로 만든 것이다. |

[출처: 01-01-data-structure.md §1 어려운 말 먼저 풀기]

그리고 잊지 마 — 쉬운 설명은 이해를 위한 비유고, 실제 면접 답변에서는 정석 설명의 용어와 전제 조건으로 돌아와 설명하는 거야. [출처: 01-01-data-structure.md §1 다시 정리]

### 🏗️ 내부 구조를 보는 눈 (Layers & Contracts)

#### 🔍 내부를 보는 네 층

① 정의부터. 자료구조의 내부는 한 덩어리가 아니라 네 개의 층으로 나눠 질문하는 게 원본의 틀이야. 논리 모델, 저장 단위, 연결 방식, 연산 계약 — 이 네 층이 각각 다른 질문에 답해. 층을 나누면 "이 구조가 실제로 무엇을 보장하는가"를 빠뜨리지 않고 점검하게 돼. [출처: 01-01-data-structure.md §4 내부를 보는 네 층]

② 메커니즘은 원본 표 그대로 볼게.

| 층 | 확인할 질문 | 예 |
|---|---|---|
| 논리 모델 | 데이터가 어떤 관계인가? | 순서, 계층, key-value |
| 저장 단위 | 무엇을 한 덩어리로 저장하는가? | element, node, entry |
| 연결 방식 | 다음 데이터를 어떻게 찾는가? | index 계산, 참조, 비교 |
| 연산 계약 | 무엇을 보장하는가? | 중복, null, 순서, 복잡도 |

[출처: 01-01-data-structure.md §4]

③ 예시로 감 잡기. 예를 들어 key-value 관계(논리 모델)를 entry 단위(저장 단위)로 저장하고 index 계산(연결 방식)으로 찾아가는 구조라면, 마지막으로 중복·null·순서·복잡도 계약(연산 계약)까지 확인해야 그 구조를 다 본 거야. 다음 문서의 HashMap이 정확히 이 네 층으로 분해되는 대표 사례야. 표의 질문 네 개를 면접 답변 뼈대로 그대로 써도 돼. 🤓

#### 📐 ADT · 자료구조 원리 · 구현체 — 세 층위 구분

① 정의. ADT/인터페이스는 "무엇을 할 수 있는가"를 정한 층이야(List, Queue, Map). 자료구조 원리는 "어떤 불변식으로 구현하는가"의 층이야(dynamic array, linked list, hash table, tree). 구현체는 그 원리를 구체 코드와 계약으로 만든 층이야(ArrayList, LinkedList, HashMap, TreeMap). [출처: 01-01-data-structure.md §4 ADT, 자료구조, 구현체 구분]

```text
ADT/인터페이스
  무엇을 할 수 있는가: List, Queue, Map

자료구조 원리
  어떤 불변식으로 구현하는가: dynamic array, linked list, hash table, tree

구현체
  구체 코드와 계약: ArrayList, LinkedList, HashMap, TreeMap
```

[출처: 01-01-data-structure.md §4]

② 메커니즘의 함정. `Queue`라고 해서 모두 FIFO는 아니야. `PriorityQueue`는 우선순위로 head를 정해. 인터페이스 이름만 보고 내부 비용을 확정하면 안 돼. 이름은 계약의 존재를 알려줄 뿐이고, 실제 비용은 자료구조 원리와 구현체 계약에서 나와. [출처: 01-01-data-structure.md §4]

> 😠 **Bailey**: "Queue니까 당연히 FIFO죠"라고 답하는 순간 바로 꼬리질문 들어온다? PriorityQueue는 우선순위로 head를 정한다고 원본에 딱 적혀 있어. 인터페이스 이름 보고 내부 비용 확정하지 마. [출처: 01-01-data-structure.md §4]

③ 예시. 어떤 개념이 독립 ADT 연산을 정의하지 않는 경우에는, 상위 자료구조의 삽입·조회·삭제 과정에서 그 개념이 사용되는 지점을 기준으로 설명해. `ADT`는 내부 코드를 정하는 말이 아니라 사용할 수 있는 연산과 그 연산이 지켜야 할 규칙을 정한 사용 계약이거든. 그리고 연산 예시는 입력부터 결과까지 따라가며 값·index·Node·Stack·Queue 중 실제로 변하는 상태를 확인하는 방식으로 이해해. [출처: 01-01-data-structure.md §5 핵심 연산, §6 상태 추적과 구현]

#### 🧪 구조 손상을 검증하는 방법

① 정의. 자료구조 구현 테스트는 반환값만 확인하면 부족해. 매 연산 뒤 내부 불변식이 유지되는지 함께 확인해야 해. 결과가 맞아 보여도 내부가 이미 손상됐을 수 있다는 게 원본의 경고야. [출처: 01-01-data-structure.md §4 구조 손상을 검증하는 방법]

② 메커니즘. 구조별 확인 항목은 원본 그대로:

```text
Linked List
  forward/backward link가 서로 일치하는가?

BST
  모든 subtree 허용 범위가 유지되는가?

Heap
  shape와 parent-child 우선순위가 유지되는가?

Hash Table
  모든 entry를 자신의 hash 탐색 경로로 다시 찾을 수 있는가?
```

[출처: 01-01-data-structure.md §4]

③ 예시 — 테스트 입력 케이스. 정상 입력, 빈 구조, 원소 하나, 중복, 최대 용량, 삭제 후 재삽입, 최악 모양을 포함해. 이 일곱 케이스를 체크리스트로 외워두면 구현 문제에서도, 면접의 "테스트는 어떻게 하실 건가요?" 질문에서도 바로 꺼내 쓸 수 있어. 😊 [출처: 01-01-data-structure.md §4]

### ⚖️ 불변식과 실패 상황 (Invariants & Failure)

`불변식`은 삽입·조회·삭제 전후에도 계속 참이어야 하는 규칙이야. 쉽게 말해 자료구조가 망가지지 않았다고 판단하는 필수 조건이고, 연산이 끝난 뒤에도 다시 맞아야 해. 정상 상태에서 유지해야 할 구체 조건은 내부 구조와 연산 설명을 기준으로 확인하고, 상위 자료구조의 불변식과 충돌하는 변경은 허용하지 않아. [출처: 01-01-data-structure.md §7 불변식]

구조별 불변식의 예를 보면 — 배열은 유효 index 범위, 연결 리스트는 끊어지지 않은 링크, BST는 정렬 관계, heap은 부모-자식 우선순위가 불변식이야. 연산 결과만 맞아 보여도 불변식이 깨지면 다음 연산에서 누락, 무한 순회, 잘못된 탐색이 발생해. [출처: 01-01-data-structure.md §8 불변식과 실패]

즉 "지금 결과가 맞다"와 "구조가 건강하다"는 다른 명제야. 실패는 대개 불변식이 깨진 다음 연산에서 드러나. [출처: 01-01-data-structure.md §8]

### 💰 비용 언어와 선택 기준 (Cost & Selection)

#### 🗣️ 비용을 말하는 법

시간 복잡도만 말하면 안 돼. 추가 메모리, 캐시 지역성, 재배치 비용, 동시 접근, 구현 복잡도도 함께 봐야 해. 그리고 평균 O(1)과 최악 O(n), O(1)과 상환 O(1)은 서로 다른 주장이라서 반드시 전제를 붙여야 해. [출처: 01-01-data-structure.md §9 비용을 말하는 법]

근거를 말하는 기준도 세 줄로 정리돼 있어. 일반 자료구조·알고리즘 성질은 표준·공식 자료(NIST DADS 등)를 기준으로 하고, Java API 동작은 Oracle Java 17 문서가 명시한 범위에서만 보장으로 말하고, OpenJDK 내부 필드·상수·계산식은 소스가 연결된 경우에만 구현 세부로 구분해. [출처: 01-01-data-structure.md §0, §11 공식 보장과 구현 세부]

#### 🧮 자료구조 선택 질문

| 질문 | 후보 |
|---|---|
| index로 바로 접근해야 하는가? | Array, ArrayList |
| key로 value를 찾아야 하는가? | Hash Table, HashMap |
| 정렬/범위가 필요한가? | Tree, B+Tree |
| 우선순위가 필요한가? | Heap, Priority Queue |
| 먼저 들어온 순서가 중요한가? | Queue |
| 나중에 들어온 것을 먼저 꺼내야 하는가? | Stack |

[출처: 01-01-data-structure.md §10 자료구조 선택 질문]

선택을 손으로 추적해보자. `회원 ID로 회원 한 명 조회`는 정확한 key 조회라서 HashMap이 후보가 되고, `점수 80~90점 회원 조회`는 범위 조건이라서 정렬 트리나 DB 인덱스가 후보가 돼. 같은 회원 데이터라도 질문이 달라지면 적합한 자료구조도 달라지는 거지. [출처: 01-01-data-structure.md §10 선택을 손으로 추적하는 예]

#### 📋 실무 판단 — 연산 비율로 구체화하기

"조회가 많다"만으로는 부족해. 어떤 조회인지와 쓰기 비용을 함께 적어야 해. [출처: 01-01-data-structure.md §12]

```text
요구사항 예
  key 단건 조회: 초당 10,000회
  신규 삽입:     초당 10회
  key 범위 조회: 없음
  정렬 순회:     없음

후보
  HashMap: 단건 key 조회에 적합
  TreeMap: 필요 없는 정렬 유지 비용 추가
  List: 조회마다 전체 순회 가능
```

[출처: 01-01-data-structure.md §12]

반대로 범위 조회가 핵심이면 "평균 O(1)"이라는 이유만으로 HashMap을 선택하면 안 돼. 실무에서는 개념 이름보다 사용하는 구현체의 API 계약, 입력 크기, 데이터 분포, 변경 빈도와 동시 접근 조건을 함께 확인하는 거야. [출처: 01-01-data-structure.md §12]

| 내부 | 자료구조와 알고리즘은 어떻게 다른가? | 자료구조는 데이터 관계와 연산 계약, 알고리즘은 그 구조를 이용하는 절차다. |
| 실패 | Big-O만 보고 고르면 왜 위험한가? | 메모리, cache, 입력 분포, 동시성, 구현 상수가 빠진다. |
| 선택 | 읽기와 쓰기 요구가 모두 강하면? | 한 구조로 강제하지 말고 보조 index와 정합성 비용을 함께 설계한다. |

> 😎 **Bailey**: 자, 웨이드님. 내가 면접관이라 치고 던진다? "Big-O만 보고 자료구조 고르면 왜 위험한데?" — 메모리, cache, 입력 분포, 동시성, 구현 상수가 빠진다는 답이 3초 안에 나와야지. 흥, 막히면 아래 답 보기 열어봐. [출처: 01-01-data-structure.md §14]

### Q1. 자료구조를 고르는 첫 기준은?

<details><summary>답 보기</summary>

저장할 데이터 모양보다 먼저 "가장 자주 하는 연산"을 본다. 조회, 삽입, 삭제, 순회, 정렬, 범위 검색 중 무엇이 핵심인지가 선택 기준이다. [출처: 01-01-data-structure.md §14 Q1]

</details>

### Q2. 같은 데이터를 두 자료구조에 중복 저장해도 되나?

<details><summary>답 보기</summary>

가능하지만 쓰기 때 두 구조를 원자적으로 갱신해야 한다. 조회 성능을 얻는 대신 메모리와 정합성 책임을 지는 공간-시간 교환이다. [출처: 01-01-data-structure.md §14 Q2]

</details>

### Q3. 자료구조가 다르면 같은 알고리즘 복잡도도 달라지는가?

<details><summary>답 보기</summary>

그렇다. binary search는 중간 index O(1) 접근이 가능한 정렬 배열에서는 O(log n)이지만, LinkedList는 중간 node 이동 비용 때문에 같은 방식의 전체 비용이 O(log n)이 아니다. [출처: 01-01-data-structure.md §14 Q3]

</details>

### Q4. 가장 빠른 자료구조 하나로 통일하면 안 되는가?

<details><summary>답 보기</summary>

연산마다 강점이 다르다. HashMap은 단건 key 조회, TreeMap은 정렬·범위, Heap은 head 우선순위에 강하다. 한 연산을 빠르게 만들면 메모리·갱신·순서 비용을 지불한다. [출처: 01-01-data-structure.md §14 Q4]

</details>

### Q5. 보조 index를 추가하면 무엇이 어려워지는가?

<details><summary>답 보기</summary>

쓰기 때 원본과 index를 함께 갱신해야 한다. 중간 실패나 동시 변경에서 둘이 달라지지 않도록 정합성·원자성·복구 전략이 필요하다. [출처: 01-01-data-structure.md §14 Q5]

</details>

### Q6. Big-O 외에 실무에서 확인할 것은?

<details><summary>답 보기</summary>

실제 입력 크기와 분포, 메모리, cache locality, 객체 할당, 동시성, I/O, 구현 상수, 운영 측정값을 확인한다. [출처: 01-01-data-structure.md §14 Q6]

</details>

### 📝 요약 (Summary)

- 자료구조 = 데이터를 저장하고 접근하는 방식의 선택. 핵심은 자주 하는 연산의 비용을 낮추는 것. [출처: 01-01-data-structure.md §1, §13]
- 내부는 논리 모델 → 저장 단위 → 연결 방식 → 연산 계약의 네 층으로 나눠 확인한다. [출처: §4]
- ADT(계약) / 자료구조 원리(불변식) / 구현체(구체 코드)는 서로 다른 층위다. 이름만 보고 내부 비용을 확정하지 않는다. [출처: §4]
- 불변식은 연산 전후 항상 참이어야 하고, 깨지면 다음 연산에서 누락·무한 순회·잘못된 탐색이 발생한다. [출처: §7, §8]
- 비용은 시간 복잡도 + 메모리·캐시·재배치·동시성·구현 복잡도. 평균/최악/상환은 전제를 붙여 구분해 말한다. [출처: §9]
- 선택 기준은 "가장 자주 하는 연산"이 먼저다. 요구사항을 연산 비율로 적어 후보를 비교한다. [출처: §10, §12, §14 Q1]

## 사실 (F)

F1. 자료구조는 데이터를 저장하고 접근하는 방식을 정한 구조이며, 배치에 따라 조회·삽입·삭제·순회·정렬·범위 검색 비용이 달라진다. [출처: 01-01-data-structure.md §1]
F2. 원본 문서의 근거 기준은 NIST Dictionary of Algorithms and Data Structures와 Oracle Java 17 Collections Framework 문서다. [출처: NIST DADS / Oracle Java 17 문서 (원본 01-01-data-structure.md §0 재인용)]
F3. ADT는 연산과 규칙을 정한 사용 계약, 자료구조 원리는 불변식 구현 방식, 구현체는 구체 코드와 계약이다. [출처: 01-01-data-structure.md §4]
F4. Queue라고 모두 FIFO가 아니며 PriorityQueue는 우선순위로 head를 정한다. [출처: 01-01-data-structure.md §4]
F5. 불변식은 연산 전후에도 계속 참이어야 하는 규칙이고, 깨지면 다음 연산에서 누락·무한 순회·잘못된 탐색이 발생한다. [출처: 01-01-data-structure.md §7, §8]
F6. binary search는 정렬 배열에서 O(log n)이지만 LinkedList에서는 중간 node 이동 비용이 더해져 같은 방식의 전체 비용이 O(log n)이 아니다. [출처: 01-01-data-structure.md §14 Q3]
F7. 자료구조 구현 테스트는 반환값과 함께 매 연산 뒤 내부 불변식을 확인하고, 정상 입력·빈 구조·원소 하나·중복·최대 용량·삭제 후 재삽입·최악 모양을 포함한다. [출처: 01-01-data-structure.md §4]
