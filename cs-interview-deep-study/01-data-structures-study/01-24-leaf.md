> 🔑 핵심 키워드: `leaf` · `internal node` · `base case` · `leaf depth` · `B+Tree leaf` · `height 0` [출처: 01-24-leaf.md §1]

### 🧭 핵심 원리 (Core Principles & Concepts)

정석 정의부터. leaf는 **자식이 없는 노드**야. 트리의 끝에 위치해. [출처: 01-24-leaf.md §1, NIST DADS Leaf (원본 §0 재인용)]

쉽게 말하면 더 이상 아래로 내려갈 곳이 없는 마지막 폴더나 조직도 맨 아래 팀원이야. 비유는 이해용이고 면접에서는 정석 용어로 돌아와 답해. [출처: 01-24-leaf.md §1]

leaf가 왜 중요하냐면, 탐색 종료·높이 계산·트리 순회 종료 조건에서 leaf 개념이 핵심이기 때문이야. [출처: 01-24-leaf.md §2]

이 문서의 범위는 leaf의 정의와 상위 자료구조에서의 역할까지야. 정렬·동시성·영속성·특정 시간 복잡도는 별도 불변식이나 구현 계약이 있을 때만 보장돼. [출처: 01-24-leaf.md §3]

#### 📖 어려운 말 먼저 풀기 (원본 용어 표 보존)

| 용어 | 쉬운 뜻 |
|---|---|
| leaf | child가 하나도 없는 끝 Node다. |
| internal node | leaf가 아니며 하나 이상의 child를 가진 Node다. |
| 종료 조건(base case) | 재귀 호출을 더 하지 않고 답을 반환하는 조건이다. leaf는 트리 재귀에서 자주 종료 지점이 된다. |
| leaf depth | root에서 해당 leaf까지 지나간 edge 수다. 일반 트리에서는 모든 leaf가 같은 depth일 필요가 없다. |
| B+Tree leaf | 실제 data entry 또는 그 위치가 모이고 옆 leaf와 연결될 수 있는 맨 아래 Node다. |

[출처: 01-24-leaf.md §1 어려운 말 먼저 풀기]

### 🏗️ 판정과 역할 (Detection & Roles)

#### 🔍 leaf를 판정하는 방법

기본 예시부터. B와 C에 자식이 없으면 B와 C는 leaf야. [출처: 01-24-leaf.md §4]

```text
A
├─ B
└─ C
```

판정 기준은 저장 표현마다 달라. 일반 rooted tree에서는 children 수가 0인 node, binary tree에서는 `left == null && right == null`, 배열 heap에서는 두 child index가 모두 size 범위 밖인지로 판단해. 저장 표현은 달라도 논리적으로 "아래에 더 진행할 node가 없다"는 뜻은 같아. [출처: 01-24-leaf.md §4]

#### 🧮 leaf가 알고리즘의 base case가 되는 이유

재귀 알고리즘은 더 내려갈 child가 없을 때 결과를 확정해. 원본 코드 그대로 보존할게. [출처: 01-24-leaf.md §4]

```java
int countLeaves(Node node) {
    if (node.children.isEmpty()) return 1;

    int count = 0;
    for (Node child : node.children) {
        count += countLeaves(child);
    }
    return count;
}
```

binary tree에서는 null 자체를 base case로 두는 방식도 있어. [출처: 01-24-leaf.md §4]

```text
countLeaves(null) = 0
countLeaves(leaf) = 1
```

null child와 실제 leaf node를 같은 것으로 혼동하면 node 수와 leaf 수가 달라져. [출처: 01-24-leaf.md §4]

> 😒 **Bailey**: 흥. `countLeaves(null)`과 `countLeaves(leaf)`를 똑같이 취급하는 코드, 리뷰에서 한두 번 본 게 아니야. null은 "자리 없음"이고 leaf는 "진짜 node"라고. 혼동하면 개수부터 틀려. 😠 [출처: 01-24-leaf.md §4]

#### 🗺️ tree 종류별 leaf 역할 (원본 표 보존)

| 구조 | leaf가 의미하는 것 |
|---|---|
| 일반 rooted tree | child가 없는 끝 node |
| BST | 검색 경로가 더 내려가지 못하는 위치 |
| Heap | 배열 뒤쪽의 child 없는 원소들 |
| B-Tree | 모든 leaf가 같은 level인 균형 경계 |
| B+Tree | 실제 record pointer가 집중되고 순차 연결될 수 있는 level |

[출처: 01-24-leaf.md §4]

#### ⚙️ 연산에서의 역할

- 재귀 순회의 자연스러운 종료 지점이다. [출처: 01-24-leaf.md §5]
- node 높이를 edge 수로 정의하면 leaf height는 0이다. [출처: 01-24-leaf.md §5]
- decision tree에서는 최종 결과, B+Tree에서는 실제 entry가 모이는 level이 될 수 있다. [출처: 01-24-leaf.md §5]

단, 모든 leaf가 같은 depth인지는 tree 종류에 따라 달라. 포화 이진 트리와 B-Tree 계열은 관련 불변식이 있지만 일반 tree는 그렇지 않아. [출처: 01-24-leaf.md §5]

상태 추적은 위 연산 예시를 입력부터 결과까지 따라가며 실제로 변하는 상태를 확인하는 방식으로 해. [출처: 01-24-leaf.md §6]

### 🧷 불변식과 경계 상황 (Invariants & Boundaries)

불변식은 삽입·조회·삭제 전후에도 계속 참이어야 하는 규칙이야. 자료구조가 망가지지 않았다고 판단하는 필수 조건이고, 연산이 끝난 뒤에도 다시 맞아야 해. leaf 관련 조건은 위 판정 기준을 근거로 확인하고, 상위 자료구조의 불변식과 충돌하는 변경은 허용하지 않아. [출처: 01-24-leaf.md §7]

경계값은 첫 index, 마지막 index, 0개처럼 조건이 바뀌는 가장자리 값이야. 실패는 예외가 발생해 바로 드러나는 경우와, 예외 없이 틀린 값이나 깨진 연결이 남는 경우로 나눠 봐야 해. [출처: 01-24-leaf.md §8]

### 📊 성능과 선택 기준 (Cost & Selection)

leaf 자체가 독립 연산을 모두 정의하는 것은 아니야. 접근 비용은 배열·Node·Tree 등 실제 저장 표현과 상위 자료구조의 연산 계약을 기준으로 판단해. [출처: 01-24-leaf.md §9] 자료구조 선택도 이 개념만 떼어 하지 않고 필요한 조회·삽입·삭제·정렬·범위·순회 연산 기준으로 비교해. [출처: 01-24-leaf.md §10] 근거는 표준·공식 자료 기준, Java API는 Oracle Java 17 문서 명시 범위에서만 보장으로 말하고, 실무에서는 구현체 API 계약·입력 크기·데이터 분포·변경 빈도·동시 접근 조건을 함께 확인해. [출처: 01-24-leaf.md §11, §12]

#### Q1. root도 leaf가 될 수 있나?

<details><summary>답 보기</summary>

노드가 root 하나뿐인 트리라면 root는 자식이 없으므로 leaf이기도 하다. [출처: 01-24-leaf.md §14 Q1]

</details>

#### Q2. degree 1인 node도 leaf인가?

<details><summary>답 보기</summary>

rooted tree에서 child가 하나 있으면 leaf가 아니다. 무방향 tree 문맥에서는 degree가 1인 vertex를 leaf라고 부르기도 하므로 어떤 tree 정의인지 기준을 선언한다. [출처: 01-24-leaf.md §14 Q2]

</details>

#### Q3. leaf 수가 많으면 tree height도 큰가?

<details><summary>답 보기</summary>

반드시 그렇지 않다. 넓고 얕은 tree는 leaf가 많아도 height가 작을 수 있다. height는 가장 긴 경로이고 leaf 수는 끝 node 개수다. [출처: 01-24-leaf.md §14 Q3]

</details>

#### Q4. 완전 이진 트리의 모든 leaf depth는 같은가?

<details><summary>답 보기</summary>

아니다. 완전 이진 트리의 마지막 level은 일부만 채워질 수 있어 leaf가 마지막 level과 그 바로 위 level에 존재한다. 포화 이진 트리와 구분한다. [출처: 01-24-leaf.md §14 Q4]

</details>

#### Q5. leaf를 삭제하면 parent도 leaf가 될 수 있는가?

<details><summary>답 보기</summary>

삭제 후 parent에게 남은 child가 없다면 그렇다. 삭제 연산은 node 수뿐 아니라 leaf 집합도 바꿀 수 있다. [출처: 01-24-leaf.md §14 Q5]

</details>

### 📌 요약 (Summary)

- leaf는 child가 하나도 없는 끝 node이고, 판정 방법은 저장 표현(children 목록/null 필드/index 범위)마다 다르다. [출처: 01-24-leaf.md §1, §4]
- leaf는 재귀 알고리즘의 base case가 된다. null child와 실제 leaf node를 혼동하면 개수 계산이 틀린다. [출처: 01-24-leaf.md §4]
- edge 수 기준이면 leaf height는 0이다. [출처: 01-24-leaf.md §5]
- 모든 leaf의 depth가 같다는 것은 일반 성질이 아니라 포화 이진 트리·B-Tree 계열의 추가 불변식이다. [출처: 01-24-leaf.md §5, §14]
- B+Tree에서는 leaf level에 실제 record pointer가 집중되고 순차 연결될 수 있다. [출처: 01-24-leaf.md §4]

## 사실 (F)

F1. leaf는 자식이 없는 노드이며 트리의 끝에 위치한다. [출처: NIST DADS Leaf (원본 01-24-leaf.md §0·§1 재인용)]
F2. leaf 판정은 표현마다 다르다. 일반 tree는 children 0개, binary tree는 left/right 둘 다 null, 배열 heap은 child index가 size 범위 밖인지로 판단한다. [출처: 01-24-leaf.md §4]
F3. leaf는 트리 재귀의 base case가 되며, null child와 실제 leaf node를 혼동하면 node 수와 leaf 수가 달라진다. [출처: 01-24-leaf.md §4]
F4. edge 수 기준으로 leaf height는 0이다. [출처: 01-24-leaf.md §5, §14 질문 지도]
F5. 모든 leaf가 같은 depth라는 것은 일반 tree의 성질이 아니고 포화 이진 트리·B-Tree 계열의 추가 불변식이다. 완전 이진 트리의 leaf는 두 level에 걸칠 수 있다. [출처: 01-24-leaf.md §5, §14 Q4]
F6. node가 root 하나뿐인 tree에서 root는 leaf이기도 하다. [출처: 01-24-leaf.md §14 Q1]
