### 🎫 핵심 원리 (Core Principles & Concepts)

FIFO는 First In First Out의 약자로, 먼저 들어온 항목이 먼저 나가는 순서 정책이야. 비유는 줄 서기 — 먼저 온 사람이 먼저 처리되는 거지. 🤓 [출처: 01-17-fifo.md §1 정의]

왜 필요할까? 요청이나 작업을 들어온 순서대로 공정하게 처리해야 할 때 필요해. FIFO는 도착 순서에 관한 공정성을 제공하는 정책이야 — 먼저 기다린 항목을 앞서 처리해서 뒤 항목이 계속 추월하지 않게 하는 거지. [출처: 01-17-fifo.md §2 탄생 배경과 필요성, §1 어려운 말 먼저 풀기]

보장 범위를 정확히 긋자. FIFO는 먼저 enqueue된 element가 먼저 dequeue된다는 **상대 순서**를 보장해. 처리 시간, 스레드 공정성, 장애 시 유실 방지까지 자동으로 보장하는 개념은 아니야. 이 경계선이 이 문서에서 제일 중요한 문장이야. [출처: 01-17-fifo.md §3 FIFO가 보장하는 것과 보장하지 않는 것]

```text
enqueue A -> enqueue B -> enqueue C
dequeue 결과: A -> B -> C
```

공식 근거는 Oracle Java 17 Queue 문서야. Java API 동작은 Oracle Java 17 문서가 명시한 범위에서만 보장으로 말하고, OpenJDK 내부 세부는 소스가 연결된 경우에만 구현 세부로 구분해. [출처: Oracle Java 17 Queue, https://docs.oracle.com/en/java/javase/17/docs/api/java.base/java/util/Queue.html (원본 01-17-fifo.md §0 재인용)] [출처: 01-17-fifo.md §11 공식 보장과 구현 세부]

#### 📖 어려운 말 먼저 풀기 (면접 대비 핵심 — 원형 보존)

| 용어 | 쉬운 뜻 |
|---|---|
| FIFO | 먼저 들어온 항목을 먼저 내보내는 순서 규칙이다. |
| 순서 정책 | 저장 구조와 별개로 어떤 항목부터 처리할지를 정한 규칙이다. |
| enqueue | Queue 뒤에 새 항목을 넣는 동작이다. |
| dequeue | Queue 앞의 가장 오래된 항목을 꺼내는 동작이다. |
| 공정성 | 먼저 기다린 항목을 앞서 처리해 뒤 항목이 계속 추월하지 않게 하는 성질이다. FIFO는 도착 순서에 관한 공정성을 제공한다. |

[출처: 01-17-fifo.md §1 어려운 말 먼저 풀기]

### ⏱️ FIFO를 사건 순서로 해부하기

#### 📬 기본 동작과 정확한 사건 분해

**정의.** 기본 동작은 단순해. enqueue한 순서 그대로 dequeue되는 거야. [출처: 01-17-fifo.md §4 예]

```text
enqueue A
enqueue B
enqueue C
dequeue -> A
dequeue -> B
dequeue -> C
```

**메커니즘.** 그런데 시스템에서는 다음 시점들이 서로 달라. FIFO Queue가 직접 보장하는 것은 보통 enqueue 순서에 따른 dequeue 순서야. 네트워크 도착 순서, 여러 consumer의 처리 완료 순서, 장애 후 재처리 순서까지 자동으로 같아지는 것은 아니야. [출처: 01-17-fifo.md §4 FIFO를 정확한 사건 순서로 나누기]

```text
요청 생성 -> Queue 도착 -> dequeue -> 처리 시작 -> 처리 완료 -> 응답
```

**예시.** FIFO의 상태 추적은 이 사건 분해를 기준으로 해. 핵심 연산의 예시(위 enqueue/dequeue 흐름)를 입력부터 결과까지 따라가며 값·index·Queue 중 실제로 변하는 상태를 확인하면 돼. FIFO 자체가 독립 ADT 연산을 정의하지 않는 경우에는 상위 자료구조(Queue)의 삽입·조회·삭제 과정에서 이 개념이 사용되는 지점을 기준으로 설명해. `ADT`는 내부 코드를 정하는 말이 아니라 사용할 수 있는 연산과 그 연산이 지켜야 할 규칙을 정한 사용 계약이니까, 이 용어 자체에 독립 연산이 없다면 실제 자료구조의 어느 동작에 참여하는지를 따라가며 이해하는 거야. [출처: 01-17-fifo.md §5 핵심 연산, §6 상태 추적과 구현]

#### 👥 단일 consumer vs 여러 consumer

**정의.** dequeue 순서와 완료 순서가 같아지려면 조건이 필요해. 단일 consumer가 처리를 직렬로 완료한 뒤 다음 작업을 꺼내면 완료 순서도 FIFO가 돼. [출처: 01-17-fifo.md §4 단일 consumer]

```text
dequeue A -> A 완료 -> dequeue B -> B 완료
```

**메커니즘.** 여러 consumer가 붙는 순간 이야기가 달라져. dequeue는 A → B 순서였는데 완료는 B → A가 되는 상황이 생겨. [출처: 01-17-fifo.md §4 여러 consumer]

```text
consumer 1: A 시작 ---------------- A 완료
consumer 2:   B 시작 ---- B 완료

dequeue: A -> B
완료:    B -> A
```

**예시와 함의.** 이게 바로 FIFO와 처리량을 위한 병렬성 사이의 순서 trade-off야. 순서를 지키려면 직렬화가 필요하고, 처리량을 올리려면 병렬화가 필요한데 그러면 완료 순서 보장이 약해져. 메시지 큐 실무 질문의 단골 구도지. 🤔 [출처: 01-17-fifo.md §4 여러 consumer]

#### 🚧 head-of-line blocking

**정의.** 앞 작업이 오래 걸리면 뒤 작업이 준비되어 있어도 기다리는 현상이야. FIFO의 구조적 약점이지. [출처: 01-17-fifo.md §4 head-of-line blocking]

```text
Queue: [A 30초] [B 10ms] [C 10ms]
              ^ B와 C는 A 뒤에서 대기
```

**메커니즘 — 대응책.** 작업을 독립 partition으로 나누거나, deadline/priority 기준을 도입하거나, 여러 worker로 병렬화할 수 있어. 하지만 각 방식은 전역 FIFO 보장을 약하게 만들 수 있어. 공짜 해법은 없다는 거야. [출처: 01-17-fifo.md §4]

**예시 — 적용과 한계 정리.** 작업 도착 순서가 중요할 때 FIFO는 자연스럽지만, 긴 작업 A가 앞에 있으면 짧은 B와 C도 기다리는 head-of-line blocking이 생길 수 있어. 우선순위나 마감 시간이 중요하면 Priority Queue, 공정한 멀티스레드 처리가 필요하면 별도의 스케줄링 정책이 필요해. [출처: 01-17-fifo.md §8 적용과 한계]

> 😒 **Bailey**: 흥. "우리 시스템 메시지 큐는 FIFO라서 순서 문제 없어요"라고 말하는 개발자, 내가 제일 못 믿어. consumer 몇 개야? 재시도 정책은? 그중 하나라도 있으면 완료 순서는 이미 FIFO가 아닐 수 있어. §4의 사건 분해 그림 다시 봐. 😠 [출처: 01-17-fifo.md §4, §14 Q2·Q4]

### ⚠️ 불변식·실패·경계 — 보장의 경계선

`불변식`은 삽입·조회·삭제 전후에도 계속 참이어야 하는 규칙이야. 자료구조가 망가지지 않았다고 판단하는 필수 조건이고, 연산이 끝난 뒤에도 다시 맞아야 해. FIFO에서 유지해야 할 구체 조건은 위 내부 구조와 연산 설명(enqueue 상대 순서 = dequeue 상대 순서)을 기준으로 확인하고, 상위 자료구조의 불변식과 충돌하는 변경은 허용하지 않아. [출처: 01-17-fifo.md §7 불변식]

실패·경계에서 기억할 것들을 모아 보면:

- FIFO가 보장하는 것: enqueue → dequeue의 상대 순서. [출처: 01-17-fifo.md §3]
- FIFO가 보장하지 않는 것: 처리 시간, 스레드 공정성, 장애 시 유실 방지, 완료 순서. [출처: 01-17-fifo.md §3, §4]
- 구조적 약점: head-of-line blocking — 앞의 긴 작업이 뒤의 짧은 작업 시작을 막는다. [출처: 01-17-fifo.md §4, §8]
- 재시도가 끼어들면: A 처리 실패 후 뒤로 재삽입하고 B가 성공하면 완료 순서는 B, A가 된다. [출처: 01-17-fifo.md §14 Q4]

### 🧭 성능·비용과 대안 선택

FIFO 자체가 독립 연산을 모두 정의하는 것은 아니야. 접근 비용은 배열·Node·Tree 등 실제 저장 표현과 상위 자료구조의 연산 계약을 기준으로 판단해. Queue 문서에서 본 것처럼 같은 FIFO라도 원형 배열이냐 연결 구조냐에 따라 비용 전제가 달라지지. [출처: 01-17-fifo.md §9 성능과 비용]

대안 선택 기준. 이 개념만 떼어 자료구조를 선택하지 않아. 필요한 조회·삽입·삭제·정렬·범위·순회 연산을 기준으로 인접 개념과 상위 자료구조를 비교하고, 실무에서는 개념 이름보다 사용하는 구현체의 API 계약, 입력 크기, 데이터 분포, 변경 빈도와 동시 접근 조건을 함께 확인해. [출처: 01-17-fifo.md §10 대안 비교와 선택 기준, §12 실무 판단 기준]

정리하면 선택 지도는 이래. [출처: 01-17-fifo.md §8, §14 Q6]

- 도착 순서가 중요 → FIFO (Queue)
- 긴급도·deadline·작업 비용이 더 중요 → Priority Queue 또는 별도 scheduler
- 여러 대상에 처리 기회를 순환 배분 → round-robin (FIFO와 다른 스케줄링 정책)

#### Q1. FIFO가 항상 공정성을 보장하나?

<details><summary>답 보기</summary>

순서 정책 자체는 먼저 들어온 항목을 먼저 처리하지만, 실제 시스템 공정성은 우선순위, 타임아웃, 재시도, 스케줄러 정책에 따라 달라질 수 있다. [출처: 01-17-fifo.md §14 Q1]

</details>

#### Q2. 메시지 큐를 FIFO라고 하면 처리 완료 순서도 같은가?

<details><summary>답 보기</summary>

반드시 그렇지 않다. 여러 consumer가 병렬 처리하면 전달 시작 순서는 FIFO여도 작업 시간 차이로 완료 순서는 바뀔 수 있다. 제품의 partition, consumer, 재시도 계약을 별도로 확인해야 한다. [출처: 01-17-fifo.md §14 Q2]

</details>

#### Q3. FIFO는 데이터 저장 위치도 순서대로라는 뜻인가?

<details><summary>답 보기</summary>

아니다. FIFO는 관찰 가능한 제거 순서다. 원형 배열은 물리적으로 뒤 index 다음에 0번 index를 사용해도 dequeue 결과는 FIFO일 수 있다. [출처: 01-17-fifo.md §14 Q3]

</details>

#### Q4. 재시도가 있으면 FIFO가 깨질 수 있는가?

<details><summary>답 보기</summary>

가능하다. A 처리 실패 후 뒤로 재삽입하고 B가 성공하면 완료 순서는 B, A가 된다. 정확한 보장은 사용하는 Queue나 메시지 시스템의 재시도 계약을 확인해야 한다. [출처: 01-17-fifo.md §14 Q4]

</details>

#### Q5. FIFO와 round-robin은 같은가?

<details><summary>답 보기</summary>

아니다. FIFO는 하나의 대기열에서 도착 순서를 따르는 정책이다. round-robin은 여러 대상에 처리 기회를 순환 배분하는 스케줄링 정책이다. [출처: 01-17-fifo.md §14 Q5]

</details>

#### Q6. FIFO가 적합하지 않은 요구사항은?

<details><summary>답 보기</summary>

긴급도, deadline, 작업 비용이 도착 순서보다 중요한 경우다. Priority Queue나 별도 scheduler가 더 직접적일 수 있다. [출처: 01-17-fifo.md §14 Q6]

</details>

### 📌 요약 (Summary)

- FIFO는 먼저 들어온 항목이 먼저 나가는 **순서 정책**이며, 도착 순서에 관한 공정성을 제공한다. Queue가 대표 구현이다. [출처: 01-17-fifo.md §1, §2]
- FIFO가 보장하는 것은 enqueue → dequeue의 상대 순서까지다. 처리 시간·스레드 공정성·장애 시 유실 방지·완료 순서는 자동 보장이 아니다. [출처: 01-17-fifo.md §3, §4]
- 사건을 분해하라: 요청 생성 → Queue 도착 → dequeue → 처리 시작 → 처리 완료 → 응답. 각 시점의 순서는 서로 다를 수 있다. [출처: 01-17-fifo.md §4]
- 단일 consumer 직렬 처리면 완료 순서도 FIFO. 여러 consumer 병렬 처리면 dequeue A→B, 완료 B→A가 나올 수 있다 — 순서 vs 처리량 trade-off. [출처: 01-17-fifo.md §4]
- head-of-line blocking: 앞의 긴 작업이 뒤의 짧은 작업을 막는다. partition 분리·deadline/priority·병렬화로 대응하되 전역 FIFO 보장은 약해질 수 있다. [출처: 01-17-fifo.md §4, §8]
- FIFO는 저장 위치의 순서가 아니라 관찰 가능한 제거 순서다 (원형 배열의 wrap-around 참고). [출처: 01-17-fifo.md §14 Q3]
- round-robin은 FIFO와 다른, 순환 배분 스케줄링 정책이다. [출처: 01-17-fifo.md §14 Q5]

## 사실 (F)

F1. FIFO는 First In First Out의 약자로, 먼저 들어온 항목이 먼저 나가는 순서 정책이다. [출처: 01-17-fifo.md §1; Oracle Java 17 Queue, https://docs.oracle.com/en/java/javase/17/docs/api/java.base/java/util/Queue.html (원본 §0 재인용)]

F2. FIFO는 먼저 enqueue된 element가 먼저 dequeue된다는 상대 순서를 보장하며, 처리 시간·스레드 공정성·장애 시 유실 방지는 자동 보장이 아니다. [출처: 01-17-fifo.md §3]

F3. 시스템 사건은 요청 생성 → Queue 도착 → dequeue → 처리 시작 → 처리 완료 → 응답으로 나뉘고, FIFO Queue가 직접 보장하는 것은 보통 enqueue 순서에 따른 dequeue 순서다. [출처: 01-17-fifo.md §4]

F4. 단일 consumer가 직렬로 완료 후 다음을 꺼내면 완료 순서도 FIFO가 되고, 여러 consumer 병렬 처리에서는 dequeue A→B에 완료 B→A가 나올 수 있다. [출처: 01-17-fifo.md §4]

F5. head-of-line blocking은 앞의 긴 작업이 뒤의 준비된 짧은 작업을 대기시키는 현상이며, partition 분리·deadline/priority 도입·worker 병렬화로 대응하되 각 방식은 전역 FIFO 보장을 약하게 만들 수 있다. [출처: 01-17-fifo.md §4, §8]

F6. FIFO는 관찰 가능한 제거 순서이지 물리 저장 위치의 순서가 아니다 — 원형 배열이 wrap-around해도 dequeue 결과는 FIFO일 수 있다. [출처: 01-17-fifo.md §14 Q3]

F7. round-robin은 여러 대상에 처리 기회를 순환 배분하는 스케줄링 정책으로, 하나의 대기열에서 도착 순서를 따르는 FIFO와 다르다. [출처: 01-17-fifo.md §14 Q5]

## 모름 (U)

U1. 실제 메시지 큐 제품(Kafka, SQS 등)별 partition·consumer·재시도 계약 — 원본이 "제품의 계약을 별도로 확인해야 한다"고만 명시했다.

U2. head-of-line blocking의 정량적 지연 영향 — 원본은 30초/10ms 예시 시나리오만 제시했고 실측 근거는 없다.

U3. 스레드 공정성(fair lock 등)과 FIFO 정책의 관계 상세 — 원본이 "자동 보장 아님"이라고 경계만 그었다.

U4. Oracle Java 17 Queue 문서가 FIFO 외 순서(우선순위 등)에 대해 명시하는 전체 조항 — 이 문서는 원본의 재인용 범위만 사용했고 원문 전체를 직접 확인하지 않았다.
