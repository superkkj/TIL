### [초시계 말고 성장률을 재라] Big-O 표기 (Big-O Notation)
> 🔑 핵심 키워드: 입력 크기 n, 증가율, 점근적 상한, 지배항, 상수 제거, worst/average/expected/amortized, Big-Theta, Big-Omega

웨이드님~ 이제 복잡도 3부작의 문이 열렸어. 😊 Big-O는 "정확히 몇 초 걸리는지"가 아니라 "데이터가 10배 커질 때 일이 얼마나 늘어나는지"를 보는 도구야. 이 관점 전환 하나가 알고리즘 대화의 공용어를 만들어 줘.

***

### 📈 핵심 원리 (Core Principles & Concepts)

Big-O는 입력 크기 `n`이 커질 때 알고리즘 비용이 어떤 성장률로 증가하는지 상한 관점에서 표현하는 표기야. 표준 근거는 NIST DADS의 Big-O Notation 항목이고. [출처: 01-40-big-o.md §1, §0(NIST DADS Big-O Notation 재인용)]

왜 필요하냐면, 작은 입력에서는 차이가 안 보이지만 데이터가 커질수록 성장률 차이가 성능 차이를 만들기 때문이야. [출처: 01-40-big-o.md §2] 문서 범위는 Big-O 표기의 정의와 상위 자료구조에서의 역할까지고, 정렬·동시성·영속성·특정 시간 복잡도는 별도 불변식이나 구현 계약이 있을 때만 보장돼. [출처: 01-40-big-o.md §3]

용어 표 원형 보존! 🤓 [출처: 01-40-big-o.md §1]

| 용어 | 쉬운 뜻 |
|---|---|
| 입력 크기 `n` | 성능을 분석할 때 데이터가 얼마나 큰지를 나타내는 기준 값이다. 배열이면 원소 수일 수 있다. |
| 증가율 | `n`이 커질 때 연산 횟수가 얼마나 빠르게 늘어나는지를 말한다. |
| 점근적 | 작은 입력의 차이보다 `n`이 충분히 커졌을 때의 변화 모양을 본다는 뜻이다. |
| 상한 | 실제 증가가 그 기준보다 더 빠르게 커지지 않는다고 설명하는 한계다. Big-O는 점근적 상한 표기다. |
| 지배항 | `n`이 커질수록 전체 식에 가장 큰 영향을 주는 항이다. `3n² + 100n + 5`에서는 `n²`이다. |
| 상수 제거 | 컴퓨터나 구현에 따라 달라지는 고정 배수보다 증가 모양에 집중하는 것이다. O(1)이 실제 명령 한 개라는 뜻은 아니다. |

쉬운 설명은 비유고, 면접에서는 정석 용어와 전제 조건으로 돌아와 설명하자. [출처: 01-40-big-o.md §1 다시 정리]

### 🧮 내부 구조와 핵심 연산

#### 📏 수학적 정의와 대표 감각

대표 표기의 감각부터. [출처: 01-40-big-o.md §4]

| 표기 | 감각 |
|---|---|
| O(1) | 입력 크기와 거의 무관 |
| O(log n) | 매번 후보를 줄임 |
| O(n) | 전체를 훑음 |
| O(n log n) | 정렬에서 자주 등장 |
| O(n^2) | 모든 쌍 비교 |

수학적 정의: `f(n) = O(g(n))`은 충분히 큰 n에서 `f(n)`이 `c * g(n)`보다 커지지 않게 하는 양의 상수 `c`와 시작점 `n0`가 존재한다는 뜻이야. [출처: 01-40-big-o.md §4]

```text
0 <= f(n) <= c * g(n), for all n >= n0
```

따라서 Big-O는 점근적 상한이야. 정확한 증가율을 위아래로 함께 제한하려면 Big-Theta, 하한은 Big-Omega를 사용해. 실무 면접에서는 관례적으로 "복잡도"를 Big-O로 말하더라도 이 차이를 알고 있어야 해. [출처: 01-40-big-o.md §4]

#### 🎛️ 입력 모델과 비용 종류를 먼저 정한다

`n`이 배열 길이인지, vertex 수인지, 저장 entry 수인지 선언해야 해. graph는 `V`와 `E`, 문자열 알고리즘은 문자열 길이처럼 변수가 여러 개일 수 있고, 변수를 억지로 하나로 합치면 비용 원인을 숨겨. [출처: 01-40-big-o.md §4]

비용 종류 네 가지도 구분하자. [출처: 01-40-big-o.md §4]

- worst-case: 같은 크기 입력 중 가장 비싼 경우
- average-case: 정의한 입력 분포에서의 평균
- expected: 확률적 선택이나 hash 분산 전제 아래 기댓값
- amortized: 긴 연산 열 전체 비용을 연산 수로 나눈 보장

HashMap 기대 O(1)과 동적 배열 append 상환 O(1)은 같은 "평균"이 아니야. [출처: 01-40-big-o.md §4]

> 😒 **Bailey**: 흥. "HashMap도 평균 O(1), ArrayList add도 평균 O(1), 그럼 같은 거네요?" — 이 말 하는 순간 탈락 벨이 울려. 하나는 hash 분산 전제의 기대값이고 하나는 연산 열 전체를 나눈 상환값이야. 단어 하나가 이해도를 폭로한다고. 😠 [출처: 01-40-big-o.md §4]

#### ➗ 합성 규칙: 순차는 더하고 중첩은 곱한다

[출처: 01-40-big-o.md §5]

```text
O(n) 작업 후 O(n log n) 작업
  O(n + n log n) = O(n log n)

O(n) 반복 안에서 매번 O(log n) 작업
  O(n * log n) = O(n log n)
```

입력 변수가 다르면 함부로 하나로 합치지 않아. [출처: 01-40-big-o.md §5]

```text
두 독립 배열 순회: O(n + m)
두 배열의 모든 쌍: O(n * m)
```

#### 🔬 코드에서 연산 횟수를 세는 순서

[출처: 01-40-big-o.md §6]

```java
int sum = 0;                    // 1
for (int i = 0; i < n; i++) { // n에 비례
    sum += values[i];           // n번
}
```

정확한 기본 연산 수를 단순화하면 `an + b` 형태야. 충분히 큰 n에서 n 항이 지배하므로 O(n)이지. [출처: 01-40-big-o.md §6]

```java
for (int i = 0; i < n; i++) {
    for (int j = 0; j < n; j++) {
        use(i, j);
    }
}
```

내부 연산은 n * n번이므로 O(n^2)야. 단순히 loop 개수만 세지 말고 각 loop 범위와 내부 연산 비용을 곱하거나 합해야 해. [출처: 01-40-big-o.md §6]

### 🕳️ 불변식과 실패·경계 상황

불변식은 삽입·조회·삭제 전후에도 계속 참이어야 하는 규칙이고, 정상 상태에서 유지해야 할 구체 조건은 내부 구조와 연산 설명을 기준으로 확인해. [출처: 01-40-big-o.md §7]

항과 상수를 버리는 이유와 한계 — `3n^2 + 100n + 5`는 충분히 큰 n에서 `n^2` 항이 지배해 O(n^2)로 표현해. 그러나 작은 n, 메모리 접근, cache, 네트워크 I/O에서는 상수와 환경이 실제 시간을 좌우할 수 있어. Big-O 분석 뒤에 benchmark와 profiler가 필요한 이유야. [출처: 01-40-big-o.md §8]

### 💰 성능·비용과 대안 비교

#### 공간 복잡도도 같은 방식으로 본다

입력 자체 공간과 알고리즘이 추가로 사용하는 auxiliary space를 구분해. [출처: 01-40-big-o.md §9]

```text
in-place 배열 swap: 추가 공간 O(1)
길이 n 결과 배열:   추가 공간 O(n)
재귀 DFS:            호출 stack O(height)
```

시간을 줄이기 위해 hash index O(n) 공간을 추가하는 것처럼 시간-공간 trade-off가 있어. [출처: 01-40-big-o.md §9]

#### 상환 분석 예: 동적 배열 append

capacity가 찰 때 2배 배열로 복사한다고 하자. [출처: 01-40-big-o.md §9]

```text
복사 크기: 1 + 2 + 4 + 8 + 마지막 확장 크기 < 2n
일반 저장: n번
전체 작업: O(n)
n번 append당 상환 O(1)
```

특정 확장 append 한 번은 O(n)이야. 평균 입력 분포가 아니라 연산 열 전체 비용을 나누는 상환 분석이지. [출처: 01-40-big-o.md §9]

대안 선택 기준: 이 개념만 떼어 자료구조를 선택하지 않고 필요한 조회·삽입·삭제·정렬·범위·순회 연산을 기준으로 비교해. [출처: 01-40-big-o.md §10] 근거 기준(표준 자료 / Oracle Java 17 문서 / OpenJDK 소스 연결 시 구현 세부)과 실무 판단 기준(API 계약, 입력 크기, 데이터 분포, 변경 빈도, 동시 접근)도 원본 그대로. [출처: 01-40-big-o.md §11, §12]

| 실패 | n의 의미를 안 정하면? | 문자열 길이·entry 수 같은 다른 비용 변수를 숨길 수 있다. |

### Q1. Big-O가 실제 성능을 완전히 설명하나?

<details><summary>답 보기</summary>

아니다. 상수항, 캐시, 메모리, 구현체, 데이터 분포도 실제 성능에 영향을 준다. Big-O는 성장률을 보는 추상화다. [출처: 01-40-big-o.md §14 Q1]

</details>

### Q2. O(n) 알고리즘은 동시에 O(n^2)이기도 한가?

<details><summary>답 보기</summary>

점근 상한 정의상 맞다. 하지만 가장 타이트한 유용한 상한을 말하는 것이 보통이다. 정확한 성장률을 강조하려면 Theta(n)이라고 표현한다. [출처: 01-40-big-o.md §14 Q2]

</details>

### Q3. 최악 O(n)이고 평균 O(1)이라고 동시에 말할 수 있는가?

<details><summary>답 보기</summary>

가능하다. 서로 다른 비용 기준이다. 평균을 말하려면 입력 분포, 기대값을 말하려면 확률 전제 등 성립 조건을 함께 제시해야 한다. [출처: 01-40-big-o.md §14 Q3]

</details>

### Q4. `n/2`번 반복하면 O(n/2)인가?

<details><summary>답 보기</summary>

Big-O에서는 상수 배를 제거해 O(n)이다. 정확한 실행 시간 비교에서는 1/2 상수도 의미가 있을 수 있다. [출처: 01-40-big-o.md §14 Q4]

</details>

### Q5. 재귀식 `T(n)=T(n/2)+O(1)`은 왜 O(log n)인가?

<details><summary>답 보기</summary>

n을 절반으로 줄여 1이 될 때까지 약 log2(n)번 호출하고 각 level 작업이 O(1)이기 때문이다. [출처: 01-40-big-o.md §14 Q5]

</details>

### Q6. Big-O가 같으면 두 구현 성능도 같은가?

<details><summary>답 보기</summary>

아니다. 상수, 메모리 접근, cache, JIT, I/O, 데이터 분포가 다를 수 있다. Big-O로 확장성을 비교한 뒤 benchmark와 profiler로 실제 병목을 확인한다. [출처: 01-40-big-o.md §14 Q6]

</details>

### 📝 요약 (Summary)

- Big-O = 입력 크기 n이 커질 때 비용의 성장률을 상한 관점에서 보는 표기. `0 <= f(n) <= c*g(n) (n >= n0)`. [출처: 01-40-big-o.md §1, §4]
- O는 상한, Theta는 상·하한 일치, Omega는 하한. [출처: 01-40-big-o.md §4]
- n의 의미(배열 길이/V·E/문자열 길이)를 먼저 선언하고, worst/average/expected/amortized를 구분한다. [출처: 01-40-big-o.md §4]
- 순차는 더하고 중첩은 곱한다. 서로 다른 입력 변수는 합치지 않는다(O(n+m) vs O(n*m)). [출처: 01-40-big-o.md §5]
- 지배항만 남기는 단순화는 큰 n 기준이다. 작은 n·cache·I/O에서는 상수가 좌우할 수 있어 benchmark·profiler가 뒤따른다. [출처: 01-40-big-o.md §8]
- 공간도 같은 틀: 입력 공간과 auxiliary space 구분, 동적 배열 append는 상환 O(1). [출처: 01-40-big-o.md §9]

## 사실 (F)
F1. Big-O는 입력 크기 n이 커질 때 알고리즘 비용의 성장률을 점근적 상한 관점에서 표현하는 표기다. [출처: 01-40-big-o.md §1; NIST DADS Big-O Notation (원본 §0 재인용)]
F2. 수학적 정의: f(n) = O(g(n)) ⟺ 양의 상수 c와 n0가 존재해 모든 n >= n0에서 0 <= f(n) <= c*g(n). [출처: 01-40-big-o.md §4]
F3. 정확한 증가율의 상·하한 동시 제한은 Big-Theta, 하한은 Big-Omega다. [출처: 01-40-big-o.md §4]
F4. worst-case / average-case / expected / amortized는 서로 다른 비용 기준이며, HashMap 기대 O(1)과 동적 배열 append 상환 O(1)은 같은 "평균"이 아니다. [출처: 01-40-big-o.md §4]
F5. 순차 연산은 더하고 중첩 연산은 곱한다: O(n + n log n) = O(n log n), O(n * log n) = O(n log n). 독립 입력은 O(n+m)/O(n*m)으로 유지한다. [출처: 01-40-big-o.md §5]
F6. 동적 배열 2배 확장 append의 복사 총량은 1부터 마지막 확장 크기까지의 2의 거듭제곱 합이며 2n보다 작다. 따라서 n번 append 전체는 O(n), 연산당 상환 비용은 O(1)이다. [출처: 01-40-big-o.md §9]
F7. 3n^2 + 100n + 5는 큰 n에서 n^2 항이 지배해 O(n^2)로 쓰지만, 작은 n·cache·I/O 환경에서는 상수가 실제 시간을 좌우한다. [출처: 01-40-big-o.md §8]
