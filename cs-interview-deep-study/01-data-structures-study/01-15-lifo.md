> 🔑 핵심 키워드: `순서 정책` · `top` · `Stack` · `중첩 구조` · `undo/redo` · `starvation` · `상환 O(1)`

웨이드님~ Stack 문서에서 계속 나오던 그 단어, LIFO를 오늘 정면으로 다뤄보자! 😊 포인트는 딱 하나 — LIFO는 자료구조가 아니라 "순서 정책"이라는 거야. 이 구분만 잡아도 꼬리질문 절반은 막을 수 있어.

### 🍽️ 핵심 원리 (Core Principles & Concepts)

LIFO는 Last In First Out의 약자로, 마지막에 들어온 항목이 먼저 나가는 순서 정책이야. 비유하면 접시 더미에서 맨 위 접시를 먼저 꺼내는 방식 — 맨 위 접시는 가장 마지막에 올린 접시지. 🤓 [출처: 01-15-lifo.md §1 정의]

왜 필요할까? 최근 상태를 먼저 되돌리거나, 깊이 방향으로 탐색하거나, 함수 호출처럼 마지막에 시작한 작업을 먼저 끝내야 할 때 적합해. 다시 말해 "가장 최근의 미완료 작업으로 돌아가야 하는 문제"와 맞아. 함수 복귀, DFS의 다음 경로, undo 기록, 파서의 중첩 구조가 그 예야. 반대로 도착 순서의 공정성이 필요하면 FIFO Queue가 더 적합해. [출처: 01-15-lifo.md §2 탄생 배경과 필요성]

그리고 LIFO가 맞지 않는 경우도 분명히 알아야 해. 요청 A가 먼저 도착했는데 새 요청을 계속 먼저 처리하면, LIFO 작업 대기열에서는 오래된 A가 계속 밀리는 starvation이 생긴다는 게 원본이 지적하는 한계야. 도착 순서 공정성이 중요하면 FIFO, 긴급도가 중요하면 Priority Queue처럼 요구사항에 맞는 정책을 사용해. [출처: 01-15-lifo.md §3 LIFO가 맞지 않는 경우]

```text
요청 A가 먼저 도착
요청 B가 나중에 도착
새 요청을 계속 먼저 처리
```

공식 근거는 Oracle Java 17 Stack 문서야. Java API 동작은 Oracle Java 17 문서가 명시한 범위에서만 보장으로 말하고, OpenJDK 내부 세부는 소스가 연결된 경우에만 구현 세부로 구분해. [출처: Oracle Java 17 Stack, https://docs.oracle.com/en/java/javase/17/docs/api/java.base/java/util/Stack.html (원본 01-15-lifo.md §0 재인용)] [출처: 01-15-lifo.md §11 공식 보장과 구현 세부]

### 🧠 정책으로서의 LIFO — 구조·연산·설계

#### 📜 LIFO는 자료구조 이름이 아니라 제거 순서 정책이다

**정의.** Stack은 LIFO를 제공하는 대표 ADT이고, LIFO 자체는 "가장 최근에 들어온 항목이 가장 먼저 나온다"는 관찰 가능한 순서야. 내부가 배열인지 연결 리스트인지는 LIFO 정의에 포함되지 않아. 정책과 구현을 분리하는 게 이 문서 전체의 뼈대야. [출처: 01-15-lifo.md §4 LIFO는 자료구조 이름이 아니라 제거 순서 정책이다]

**메커니즘.** LIFO를 연산 계약으로 표현하면 이렇게 돼. push를 n번 하고 중간 삭제가 없으면, pop 결과는 역순이야. 배열과 연결 리스트 모두 이 결과를 만들 수 있어. 사용자가 관찰할 수 있는 제거 순서가 같으면 둘 다 LIFO 구현이지만, 확장 비용과 메모리 구조는 다를 수 있어. [출처: 01-15-lifo.md §5 LIFO를 연산 계약으로 표현하기]

```text
push(x1)부터 push(xn)까지 차례로 실행
중간 삭제가 없으면 pop 결과는
xn부터 x1까지 역순
```

**예시.** 기본 동작과 상태 추적. [출처: 01-15-lifo.md §4 예, §6 상태 추적]

```text
push A
push B
push C
pop -> C
pop -> B
pop -> A
```

```text
push A: [A]
push B: [A, B]
push C: [A, B, C]
pop   : C
pop   : B
```

여기서 계약의 경계 하나 — 중간의 A를 먼저 제거할 수 있게 API를 열면 순수 Stack 계약이 아니야. [출처: 01-15-lifo.md §6]

#### 🪆 중첩 문제와 LIFO — 왜 이 정책이 자연스러운가

**정의.** LIFO가 필요한 근본 이유는 "가장 최근에 시작했지만 아직 끝나지 않은 작업"을 먼저 끝내야 하는 문제와 잘 맞기 때문이야. 이건 원본이 명시한 인과 관계야. [출처: 01-15-lifo.md §4 중첩 문제와 LIFO]

**메커니즘.** 함수 호출의 중첩을 보면 구조가 그대로 보여. 안쪽에서 시작한 것이 먼저 끝나야 바깥이 이어질 수 있어. [출처: 01-15-lifo.md §4]

```text
함수 A 시작
  함수 B 시작
    함수 C 시작
    함수 C 종료
  함수 B 종료
함수 A 종료
```

**예시.** 열린 괄호도 같아. 가장 마지막에 연 괄호를 가장 먼저 닫는 거지. [출처: 01-15-lifo.md §4]

```text
입력: ([{}])
가장 마지막에 연 괄호 {를 가장 먼저 }로 닫는다.
```

#### ↩️ 제한된 undo 기록 설계 — 정책과 저장 구조는 별개

**정의.** 실무 undo는 무한 Stack이 아니야. 기록 개수에 상한을 두고, 새 작업이 들어오면 redo 기록을 비우는 정책이 같이 붙어. [출처: 01-15-lifo.md §4 제한된 undo 기록 설계]

**메커니즘.** 원본이 제시한 흐름은 이래. [출처: 01-15-lifo.md §4]

```text
push(newCommand)
  1. undoStack에 기록
  2. redoStack 비우기
  3. 최대 개수를 넘으면 가장 오래된 기록 제거
```

**예시와 함의.** 순수 Stack API만으로 가장 오래된 바닥 원소를 효율적으로 제거하기 어렵다면 Deque나 원형 buffer 등 다른 구현을 선택할 수 있어. 정책은 LIFO여도 저장 구조 선택은 별개라는 것 — 이게 이 서브섹션의 결론이야. [출처: 01-15-lifo.md §4]

> 😒 **Bailey**: 흥. "undo는 Stack으로 만들면 끝이죠?"라고 하면 바로 반격 들어간다? 최대 개수 초과 시 **바닥** 원소를 지워야 하는데, 순수 Stack 계약에는 바닥을 건드리는 연산이 없어. 그래서 원본도 Deque나 원형 buffer를 언급한 거야. 정책과 저장 구조를 섞어 말하는 순간 감점이야. 😠 [출처: 01-15-lifo.md §4]

### ⚠️ 불변식과 실패·예외·경계

`불변식`은 삽입·조회·삭제 전후에도 계속 참이어야 하는 규칙이야. 자료구조가 망가지지 않았다고 판단하는 필수 조건이고, 연산이 끝난 뒤에도 다시 맞아야 해. LIFO에서 유지해야 할 구체 조건은 위 내부 구조와 연산 설명(push 순서의 역순으로 pop, top 일관성)을 기준으로 확인하고, 상위 자료구조의 불변식과 충돌하는 변경은 허용하지 않아. [출처: 01-15-lifo.md §7 불변식]

실패·경계는 이렇게 나눠 봐. `경계값`은 첫 index, 마지막 index, 0개처럼 조건이 바뀌는 가장자리 값이야. 실패는 예외가 발생해 바로 드러나는 경우와, 예외 없이 틀린 값이나 깨진 연결이 남는 경우로 나눠서 확인해. 경계값과 빈 구조를 먼저 확인하고, 상위 자료구조의 불변식이 깨졌을 때 발생하는 예외와 예외 없이 잘못되는 결과를 구분하는 거야. [출처: 01-15-lifo.md §8 실패·예외·경계 상황]

운영 관점의 대표 실패는 앞에서 본 starvation이야. 새 작업이 계속 들어오면 오래된 작업이 처리되지 않으니까, 최대 대기 시간, aging, FIFO 전환 같은 별도 스케줄링 정책이 필요할 수 있어. [출처: 01-15-lifo.md §3, §14 Q6]

### 🧮 성능·비용과 대안 선택

LIFO 자체가 독립 연산을 모두 정의하는 것은 아니야. 접근 비용은 배열·Node·Tree 등 실제 저장 표현과 상위 자료구조의 연산 계약을 기준으로 판단해. "LIFO니까 O(1)"이라는 문장은 성립하지 않아 — 동적 배열 push는 평균이 아니라 상환 O(1)이고 확장 순간에는 O(n) 복사가 발생한다는 게 대표 반례야. [출처: 01-15-lifo.md §9 성능과 비용, §14 Q2]

대안 선택은 이 개념만 떼어 하지 않아. 필요한 조회·삽입·삭제·정렬·범위·순회 연산을 기준으로 인접 개념과 상위 자료구조를 비교하고, 실무에서는 개념 이름보다 사용하는 구현체의 API 계약, 입력 크기, 데이터 분포, 변경 빈도와 동시 접근 조건을 함께 확인해. 요약하면 이 그림이야. [출처: 01-15-lifo.md §10 대안 비교와 선택 기준, §12 실무 판단 기준]

- 최근 것 먼저 (되돌리기·중첩·깊이 탐색) → LIFO (Stack)
- 도착 순서 공정성 → FIFO (Queue)
- 긴급도·우선순위 → Priority Queue [출처: 01-15-lifo.md §2, §3]

#### Q1. LIFO와 FIFO 차이는?

<details><summary>답 보기</summary>

LIFO는 마지막에 들어온 것이 먼저 나가고, FIFO는 먼저 들어온 것이 먼저 나간다. Stack은 LIFO, Queue는 FIFO가 대표적이다. [출처: 01-15-lifo.md §14 Q1]

</details>

#### Q2. LIFO이면 항상 O(1)인가?

<details><summary>답 보기</summary>

아니다. LIFO는 순서 정책이고 시간 복잡도는 구현에 달렸다. 동적 배열 push는 평균이 아니라 상환 O(1)이며 확장 순간에는 O(n) 복사가 발생한다. [출처: 01-15-lifo.md §14 Q2]

</details>

#### Q3. LIFO와 Stack은 정확히 무엇이 다른가?

<details><summary>답 보기</summary>

LIFO는 제거 순서 정책이고 Stack은 그 정책을 제공하는 ADT다. Stack은 push, pop, peek 같은 연산 계약까지 포함하지만 LIFO라는 말 자체는 순서만 가리킨다. [출처: 01-15-lifo.md §14 Q3]

</details>

#### Q4. 재귀가 LIFO인 이유는?

<details><summary>답 보기</summary>

가장 최근에 호출된 함수가 먼저 끝나야 그 함수를 호출한 이전 함수 위치로 돌아갈 수 있기 때문이다. 호출 frame과 복귀 순서가 LIFO를 따른다. [출처: 01-15-lifo.md §14 Q4]

</details>

#### Q5. LIFO가 FIFO보다 항상 빠른가?

<details><summary>답 보기</summary>

아니다. 두 용어는 제거 순서 정책이다. 성능은 배열, 연결 구조, 동시성 제어 등 실제 구현과 연산에 따라 달라진다. [출처: 01-15-lifo.md §14 Q5]

</details>

#### Q6. LIFO 작업 처리의 대표 실패는?

<details><summary>답 보기</summary>

새 작업이 계속 들어오면 오래된 작업이 처리되지 않는 starvation이다. 최대 대기 시간, aging, FIFO 전환 같은 별도 스케줄링 정책이 필요하다. [출처: 01-15-lifo.md §14 Q6]

</details>

### 📌 요약 (Summary)

- LIFO는 마지막에 들어온 항목이 먼저 나가는 **순서 정책**이며, 저장 코드나 자료구조 이름이 아니다. Stack은 이 정책을 push/pop/peek 연산으로 제공하는 대표 ADT다. [출처: 01-15-lifo.md §1, §4, §14 Q3]
- 연산 계약: push(x1..xn) 후 중간 삭제가 없으면 pop 결과는 xn..x1. 이 관찰 가능한 순서가 같으면 배열이든 연결 리스트든 LIFO 구현이다. [출처: 01-15-lifo.md §5]
- 중첩 문제(함수 호출, 괄호)가 LIFO의 본질적 적용처다 — 가장 최근에 시작한 미완료 작업을 먼저 끝낸다. [출처: 01-15-lifo.md §2, §4]
- 실무 undo는 무한 Stack이 아니다 — redo 비우기, 최대 개수 초과 시 바닥 제거가 붙고, 그래서 Deque·원형 buffer 같은 다른 저장 구조를 선택하기도 한다. [출처: 01-15-lifo.md §4]
- LIFO 작업 대기열의 대표 실패는 starvation이고, 공정성이 필요하면 FIFO, 긴급도가 필요하면 Priority Queue를 검토한다. [출처: 01-15-lifo.md §3, §14 Q6]
- "LIFO = O(1)"은 성립하지 않는다. 성능은 실제 저장 표현과 연산 계약이 정한다. [출처: 01-15-lifo.md §9, §14 Q2]

## 사실 (F)

F1. LIFO는 Last In First Out의 약자로, 마지막에 들어온 항목이 먼저 나가는 순서 정책이다. [출처: 01-15-lifo.md §1; Oracle Java 17 Stack, https://docs.oracle.com/en/java/javase/17/docs/api/java.base/java/util/Stack.html (원본 §0 재인용)]

F2. LIFO는 관찰 가능한 제거 순서만 정하며 내부 저장 구조(배열/연결 리스트)는 정의에 포함되지 않는다. Stack은 이 정책을 연산으로 제공하는 ADT다. [출처: 01-15-lifo.md §4, §14 Q3]

F3. 연산 계약: `push(x1)`부터 `push(xn)`까지 실행한 뒤 중간 삭제가 없으면 pop 결과는 `xn`부터 `x1`까지 역순이다. [출처: 01-15-lifo.md §5]

F4. 함수 호출·괄호 같은 중첩 문제는 "가장 최근에 시작한 미완료 작업을 먼저 끝낸다"는 LIFO 성질과 맞는다. [출처: 01-15-lifo.md §2, §4]

F5. LIFO 작업 대기열에서는 새 요청 우선 처리로 오래된 요청이 밀리는 starvation이 생기며, 대응책으로 최대 대기 시간·aging·FIFO 전환 같은 정책이 있다. [출처: 01-15-lifo.md §3, §14 Q6]

F6. 동적 배열 push는 상환 O(1)이고 확장 순간 O(n) 복사가 발생한다 — "LIFO이면 항상 O(1)"은 성립하지 않는다. [출처: 01-15-lifo.md §14 Q2]

F7. 제한된 undo 설계는 undoStack 기록 → redoStack 비우기 → 최대 개수 초과 시 가장 오래된 기록 제거 순서를 따르며, 바닥 제거가 필요하면 Deque·원형 buffer 등 다른 저장 구조를 선택한다. [출처: 01-15-lifo.md §4]

## 모름 (U)

U1. 중간 원소 제거를 허용하는 변형 구조들의 이름과 계약 — 원본은 "중간 A를 먼저 제거하게 API를 열면 순수 Stack 계약이 아니다"까지만 말했다.

U2. aging·FIFO 전환 같은 스케줄링 정책의 구체 알고리즘 — 원본이 이름만 언급했고 상세는 이 문서 범위 밖이다.

U3. 실제 편집기 제품들의 undo 최대 기록 개수 기본값 — 원본에 근거가 없어 본문에 넣지 않았다.

U4. Oracle Java 17 Stack 문서가 LIFO 정책 자체에 대해 추가로 명시하는 세부 조항 — 이 문서는 원본의 재인용 범위만 사용했고 원문 전체를 직접 확인하지 않았다.
